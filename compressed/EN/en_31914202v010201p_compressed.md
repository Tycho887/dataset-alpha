# ETSI EN 319 142-2: Electronic Signatures and Trust Infrastructures (ESI); PAdES digital signatures; Part 2: Additional PAdES signatures profiles
**Source**: ETSI Technical Committee ESI | **Version**: V1.2.1 | **Date**: 2025-07 | **Type**: Normative
**Original**: PDF available from ETSI Search & Browse Standards

## Scope (Summary)
This document defines multiple profiles for PAdES digital signatures embedded within PDF. It contains: a profile for CMS-based PDF signatures (clause 4) for interoperability beyond ISO 32000-2; extended PAdES signature profiles (clause 5) that extend ETSI EN 319 142-1 baseline signatures with higher optionality (BES, EPES, LTV levels); and profiles for XAdES signatures on XML content embedded in PDF (clause 6) including XAdES signatures for XML documents and XFA forms.

## 4 Profile for CMS digital signatures in PDF
### 4.1 Features
- CMS-encoded (RFC 5652 [13]), support serial signatures, optionally include time-stamps, revocation info, reasons, location, contact info.
- Optionally legal content attestation for PDF capabilities (e.g. JavaScript).

### 4.2 Requirements of Profile for CMS Signatures in PDF
#### 4.2.1 Requirements on PDF signatures
- a) PDF Signatures **shall** be as specified in ISO 32000-2 [1], clause 12.8.
- b) Signature information **shall** be embedded; ByteRange **shall** cover entire file except signature itself.
- c) PDF Signature (DER-encoded CMS binary) **shall** be placed into Contents entry.
- d) CMS object **shall** conform to IETF RFC 5652 [13]; at minimum, **shall** include signer's X.509 signing certificate.
  - NOTE 1: PKCS#1 not supported by this profile.
- e) Timestamping and revocation info **should** be included.
- f) Revocation info, if present, **shall** be a signed attribute.
- g) RFC 5755 attribute certificates **should not** be used.
- h) Single signer only (one SignerInfo).

#### 4.2.2 Requirements on PDF signature handlers
- a) A PDF reader **may** substitute a different signature handler if it supports SubFilter.
- b) Only SubFilter values: `adbe.pkcs7.detached` and `ETSI.CAdES.detached` **shall** be used.
- Use of SHA-1 is phased out; other hashing algorithms **shall** be used.

#### 4.2.3 Requirements on signature validation
- a) Validate document digest matches signature as per ISO 32000-2 [1], clause 12.8.1.
- b) Validate certificate path per RFC 5280 [3]; validity checks at time indicated by electronic time-stamp or other trusted signing time; revocation checked as per clause 4.2.5.
- c) Signing certificate reference attribute **should** be ignored if present.

#### 4.2.4 Requirements on Time Stamping
##### 4.2.4.1 Creation
- a) Electronic time-stamp from trusted TSA **should** be applied as soon as possible after signing.
- b) If embedding, **shall** be as per ISO 32000-2 [1], clause 12.8.3.3.1.

##### 4.2.4.2 Validation
- a) Encode signature field, compute digest using algorithm indicated in time-stamp.
- b) Check digest matches time-stamp.

#### 4.2.5 Requirements on revocation checking
- A signature handler **may** ignore embedded revocation info in favour of alternative storage per its own policies.

#### 4.2.6 Requirements on Seed Values
- Seed values requiring violation of this profile **shall not** be used.
  - Example: PKCS#1 not permitted.

#### 4.2.7 Requirements on encryption
- Requirements from ETSI EN 319 142-1 [4], clause 5.5 **shall** apply.

## 5 Extended PAdES signature profiles
### 5.1 Features
- Based on building blocks from ETSI EN 319 142-1 [4].
- Three levels: PAdES-E-BES (basic), PAdES-E-EPES (with signature policies), PAdES-E-LTV (long-term).

### 5.2 General Requirements
#### 5.2.1 Requirements from ETSI EN 319 142-1
- Requirements in clauses 4.1 and 6.2.1 of ETSI EN 319 142-1 [4] **shall** apply to all profiles in clause 5.

#### 5.2.2 Notation of Requirements
- Tables with columns: Attribute/Field/Service, Presence (shall be present, shall not be present, may be present, conditioned presence), Cardinality (0, 1, 0 or 1, ≥0, ≥1), Additional notes and requirements, Reference.

### 5.3 PAdES-E-BES Level
- Requirements in Table 1 (see original document for full table). Key points:
  - `content-type` **shall be present** (value id-data).
  - `message-digest` **shall be present**.
  - `signing-certificate reference` **shall be present** (use ESS signing-certificate-v2 in preference to SHA-1 based).
  - `signer-attributes-v2` **may be present**.
  - `content-time-stamp` **may be present**.
  - `signature-time-stamp` **may be present**.
  - `commitment-type-indication` **conditioned presence**: only if Reason entry not used.
  - `signing-time` **may be present**.
  - Entries in Signature Dictionary: M, Location **may be present**; Reason **conditioned**; Filter, ByteRange, SubFilter, Contents **shall be present**; Name, ContactInfo **may be present**; Cert **shall not be present**.
- Additional requirements:
  - a) content-type value id-data.
  - b-c) ESS signing-certificate or signing-certificate-v2 used appropriately.
  - d) Migration to signing-certificate-v2 recommended.
  - e) commitment-type-indication only if Reason not used.
  - f) M entry should contain UTC time; if signing-time present, M shall not indicate time.
  - g) Reason not used if commitment-type-indication present.
  - h) Verifier may substitute Filter handler if supporting SubFilter.
  - i) ByteRange covers entire file except signature.
  - j) SubFilter value `ETSI.CAdES.detached`.
  - k) PKCS#1 and PKCS#7 not used.

### 5.4 PAdES-E-EPES Level
- Additions to Table 1 (replacement): `signature-policy-identifier` **shall be present**; entry with key Reason **shall not be present**.

### 5.5 PAdES-E-LTV Level
- Signature handlers **shall** support documents with:
  - a) Document security store (DSS) as per ETSI EN 319 142-1 [4] clause 5.4.2.
  - b) Document time-stamps as per clause 5.4.3.
- Signed PDF documents **should** contain DSS followed by document time-stamp.
- Validation data **shall** be carried by values within DSS (no external references).
- Systems **shall** support creation/validation with one or more DSS entries and document time-stamps.
- Object IDs **should not** be reused (to avoid hiding validation data).

## 6 Profiles for XAdES Signatures signing XML content in PDF
### 6.1 Features
- Two main sub-profiles: (1) XAdES signatures of signed XML documents embedded in PDF containers, (2) XAdES signatures on XFA forms.

### 6.2 Profiles for XAdES signatures of signed XML documents embedded in PDF containers
#### 6.2.1 Overview
- Defines basic profile (clause 6.2.2) for XAdES-E-BES, EPES, T levels, and long-term profile (clause 6.2.3) for C to A levels.
- Scenario: XML document created and signed outside PDF, then embedded; augmentation done by extracting, augmenting XAdES, and re-embedding.

#### 6.2.2 Profile for Basic XAdES signatures of XML documents embedded in PDF containers
##### 6.2.2.1 Features
- Signed XML created independent of PDF; XAdES levels BES, EPES, T; serial via countersignatures; parallel signatures supported.

##### 6.2.2.2 General syntax and requirements
- Signatures **shall** be XAdES-E-BES, EPES, or T per ETSI EN 319 132-2 [7] with restrictions in this document.
- Unsigned properties not found may be present and ignored unless used with other profiles.

##### 6.2.2.3 Requirements for applications generating signed XML document to be embedded
- Signed XML **shall** be created independent of PDF.
- **Shall** contain at least one XAdES signature and one or more signed data objects.
- Signed data objects and signatures satisfy one of:
  1) All data objects embedded within the signed XML document; or
  2) If detached, ds:Reference **shall** reference it per W3C XML-Signature [9].

##### 6.2.2.4 Mandatory operations
###### 6.2.2.4.1 Protecting the signing certificate
- Certificate **shall** be referenced in SigningCertificateV2 or included in ds:KeyInfo and covered by signature. SigningCertificateV2 **should** be used.
- NOTE: Signing whole ds:KeyInfo locks it; XPath transforms may be used.

##### 6.2.2.5 Requirements on XAdES optional properties
- Table 3 lists properties that may be present: SigningTime, SignaturePolicyIdentifier, SignatureProductionPlaceV2, SignerRoleV2, DataObjectFormat, CommitmentTypeIndication, AllDataObjectsTimeStamp, IndividualDataObjectsTimeStamps, SignatureTimeStamp, SignaturePolicyStore.
- If present, syntax, semantics, usage **shall** be as per ETSI EN 319 132-1 [6].

##### 6.2.2.6 Serial Signatures
- Supported via mechanisms in ETSI EN 319 132-1 [6] clause 5.2.7. CounterSignature optional; detached countersignature may be used.

##### 6.2.2.7 Parallel Signatures
- May include several XAdES signatures signing the same data objects.

##### 6.2.2.8 PAdES Signatures
- CMS signatures (clause 4), PAdES-E-BES, PAdES-E-EPES, optionally augmented with PAdES-E-LTV, can be applied to document with XAdES signature.
- NOTE: XAdES signature cannot be augmented once PDF signature applied.

#### 6.2.3 Profile for long-term XAdES signatures of signed XML documents embedded in PDF containers
##### 6.2.3.1 Features
- Long-term signatures; levels C, X, XL, A (ETSI EN 319 132-2 [7]).

##### 6.2.3.2 Augmentation mechanism
- Conforming readers **shall** detach the signed XML document, add unsigned properties, then re-embed.

##### 6.2.3.3 Optional properties
- No additional requirements; if present, syntax etc. per ETSI EN 319 132-1 [6].

##### 6.2.3.4 Validation Process
- Document Secure Store **should not** be accessed during validation.

### 6.3 Profiles for XAdES signatures on XFA Forms
#### 6.3.1 Overview
- Two profiles: basic (clause 6.3.2) and long-term (clause 6.3.3) using DSS and VRI from ETSI EN 319 142-1 [4] for XFA forms.

#### 6.3.2 Profile for Basic XAdES signatures on XFA forms
##### 6.3.2.1 Features
- XAdES signs XFA data or any XML content allowed by XFA spec; levels BES, EPES, T; serial and parallel signatures supported.

##### 6.3.2.2 General syntax and requirements
- Signatures **shall** be XAdES per ETSI EN 319 132-1 [6] with restrictions.
- **Shall** respect XFA requirements except those conflicting with XAdES.
- Unsigned properties not found may be present and ignored.
- Time-stamp **should** be applied immediately after signing.
- Signature handlers **shall** sign SignedProperties and ds:SignatureProperties without listing them in manifest.

##### 6.3.2.3 Mandatory operations
###### 6.3.2.3.1 Protecting the signing certificate
- Recommends using SigningCertificateV2; alt technique using ds:KeyInfo permitted. At least one **shall** be present.

##### 6.3.2.4 Requirements on XAdES optional properties
- Table 4: SigningTime **shall not be present**; SignaturePolicyIdentifier, SignatureProductionPlaceV2, SignerRoleV2, DataObjectFormat **may be present**; CommitmentTypeIndication **conditioned**; AllDataObjectsTimeStamp, IndividualDataObjectsTimeStamps, SignatureTimeStamp, SignaturePolicyStore **may be present**.
- Signed XMLDSig elements generated by XFA processors include additional elements in ds:SignatureProperties.
- SignatureTimeStamp **should** be present.
- If SignaturePolicyIdentifier absent, CommitmentTypeIndication **shall not be present**.
- If SignaturePolicyIdentifier present, CommitmentTypeIndication **may be present**; if signature is EPES, ds:SignatureProperties **shall not** include Reason via Dublin Core description.

##### 6.3.2.5 Serial Signatures
- Supported via same mechanisms as clause 6.2.2.6.

##### 6.3.2.6 Parallel Signatures
- May include several XAdES signatures signing the XML content.

#### 6.3.3 Profile for long-term validation XAdES signatures on XFA forms
##### 6.3.3.1 Overview
- Profiles basic signatures from clause 6.3.2 to support long-term validation equivalent to XAdES-E-XL and XAdES-E-A by augmenting with LTV mechanisms from ETSI EN 319 142-1 [4] clause 5.4.

##### 6.3.3.2 Features
- Same features as clause 6.3.2 plus long-term validation via DSS and VRI.

##### 6.3.3.3 General Requirements
- Signature handlers **shall** sign/validate signed XFA forms; **shall** support documents with DSS and document time-stamps per ETSI EN 319 142-1 [4] clauses 5.4.2 and 5.4.3.
- Validation data carried by value within DSS.
- Handlers **shall** support generation/validation with one or more DSS entries and document time-stamps.

### 6.3.4 Extensions Dictionary
- The extensions dictionary (ISO 32000-2 [1], clause 7.12) **should** include entry: `<</ESIX <</BaseVersion /1.7 /ExtensionLevel 1>> >>` to identify extensions as per clause 5.4 of ETSI EN 319 142-1 [4].
