# ETSI EN 319 122-1 V1.3.1: Electronic Signatures and Infrastructures (ESI); CAdES digital signatures; Part 1: Building blocks and CAdES baseline signatures
**Source**: ETSI | **Version**: V1.3.1 | **Date**: 2023-06 | **Type**: Normative
**Original**: https://www.etsi.org/deliver/etsi_en/319100_319199/31912201/01.03.01_60/en_31912201v010301p.pdf

## Scope (Summary)
Specifies CAdES digital signatures built on CMS signatures (IETF RFC 5652 [7]) by incorporating signed and unsigned attributes. Defines ASN.1 for attributes and four levels of baseline signatures (B‑B, B‑T, B‑LT, B‑LTA) for long‑term validity. Creation/augmentation/validation procedures are out of scope (see ETSI EN 319 102‑1 [i.5], ETSI TR 119 100 [i.4]).

## Definitions and Abbreviations
- **CAdES signature**: digital signature satisfying ETSI EN 319 122 part 1 or part 2 [i.6]
- **Certificate Revocation List (CRL)**: signed list of revoked certificates
- **digital signature**: data appended to or cryptographic transformation of a data unit for source/integrity proof and forgery protection
- **digital signature value**: result of the cryptographic transformation
- **electronic time‑stamp**: data binding other electronic data to a particular time (e.g., RFC 3161 timeStampToken)
- **Legacy CAdES 101 733 signature**: according to ETSI TS 101 733 [1]
- **Legacy CAdES baseline signature**: according to ETSI TS 103 173 [i.1]
- **Legacy CAdES signature**: legacy CAdES 101 733 or baseline
- **signature augmentation/creation/validation policy**: sets of rules for augmentation, creation, validation
- **signature policy**: any combination of above
- **validation data**: data used to validate a digital signature
- **Abbreviations**: ATSv2, ATSv3, MIME (as defined in clauses 3.3 and body)

## General Syntax (Clause 4)
### 4.1 General requirements
- CAdES signatures **shall** build on CMS (IETF RFC 5652 [7]) by incorporation of signed and unsigned attributes as per clause 5.1.
- CAdES signatures **shall** comply with clauses 2, 3, 4 and 5 of IETF RFC 5652 [7].

### 4.2 The data content type
- **shall** be as defined in CMS (IETF RFC 5652 [7], clause 4). Identified by OID id‑data.

### 4.3 The signed‑data content type
- **shall** be as defined in CMS (IETF RFC 5652 [7], clause 5).

### 4.4 The SignedData type
- **shall** be as defined in CMS (IETF RFC 5652 [7], clause 5.1). CMSVersion set per clause 5.1 of [7].

### 4.5 The EncapsulatedContentInfo type
- **shall** be as defined in CMS (IETF RFC 5652 [7], clause 5.2). For long‑term validation, either eContent **should** be present, or the signed data archived preserving encoding.

### 4.6 The SignerInfo type
- **shall** be as defined in CMS (IETF RFC 5652 [7], clause 5.3). Degenerate case (no signers) **shall not** be used.

### 4.7 ASN.1 Encoding
- **4.7.1 DER**: **shall** be as defined in ITU‑T X.690 [17].
- **4.7.2 BER**: if used, **shall** be as defined in ITU‑T X.690 [17].

### 4.8 Other standard data structures
- **TimeStampToken**: as per IETF RFC 3161 [4] updated by RFC 5816 [9].
- Additional types: VisibleString, BMPString, IA5String, GeneralizedTime, UTCTime per X.680; DirectoryString per X.520; AttributeCertificate per RFC 5755 [8]; ResponderID, OCSPResponse, BasicOCSPResponse per RFC 6960 [14]; Name, Certificate, AlgorithmIdentifier per RFC 5280 [6]; Attribute per RFC 5280; CertificateList per RFC 5280; RevocationInfoChoices per RFC 5652 [7].

### 4.9 Attributes
- Clause 5 defines attributes (CMS, ESS, new). Distinction: signed attributes (covered by signature) and unsigned attributes (added after signing).

## Attribute Semantics and Syntax (Clause 5)
### 5.1 CMS defined basic signed attributes
#### 5.1.1 content‑type
- Signed attribute. Indicates type of signed content. **shall** be as defined in CMS (IETF RFC 5652 [7], clause 11.1).

#### 5.1.2 message‑digest
- Signed attribute. Specifies message digest of content. **shall** be as defined in CMS (IETF RFC 5652 [7], clause 11.2). Calculation per clause 5.4 of [7].

### 5.2 Basic attributes for CAdES signatures
#### 5.2.1 signing‑time
- Signed attribute. **shall** specify time signer claims to have performed signing. Syntax per CMS [7] clause 11.3.

#### 5.2.2 Signing certificate reference attributes
- **5.2.2.1 General**: attributes **shall** contain one reference to the signing certificate; **may** include references to certificates in the path. Each reference **shall** contain a digest value.
- **5.2.2.2 ESS signing‑certificate**: signed attribute using SHA‑1. **shall** be as defined in IETF RFC 2634 [3] clause 5.4. `policies` field **shall not** be used.
- **5.2.2.3 ESS signing‑certificate‑v2**: signed attribute using hash algorithm other than SHA‑1. **shall** be as defined in IETF RFC 5035 [5] clause 4. `policies` field **shall not** be used.

#### 5.2.3 commitment‑type‑indication
- Signed attribute. **shall** qualify the signed data object with a commitment. Value **shall** be `CommitmentTypeIndication` identified by OID id‑aa‑ets‑commitmentType. Exactly one `AttributeValue`.

#### 5.2.4 Attributes for identifying the signed data type
- **5.2.4.1 content‑hints**: signed attribute. **shall not** be used in countersignature. Provides info on innermost content. Syntax per ESS [3] clauses 1.3.4 and 2.9. When MIME types used, contentType **shall** be id‑data.
- **5.2.4.2 mime‑type**: signed attribute. **shall not** be used in countersignature. Indicates MIME type of signed data. Value **shall** be `MimeType` (UTF8String). Identified by OID id‑aa‑ets‑mimeType.

#### 5.2.5 signer‑location
- Signed attribute. Specifies address associated with signer. Value **shall** be `SignerLocation`. At least one of countryName, localityName, postalAddress **shall** be present.

#### 5.2.6 Incorporating attributes of the signer
- **5.2.6.1 signer‑attributes‑v2**: signed attribute. Encapsulates claimed, certified, or signed assertion attributes. Value **shall** be `SignerAttributeV2`. Empty attribute **shall not** be created.
- **5.2.6.2 claimed‑SAML‑assertion**: claimed assertion including SAML assertion. Used in `claimedAttributes` field. Value **shall** be `ClaimedSAMLAssertion` (OCTET STRING).
- **5.2.6.3 signed‑SAML‑assertion**: identified by OID id‑ets‑signedSAML. Value **shall** be `SignedSAMLAssertion` (OCTET STRING).

#### 5.2.7 countersignature
- Unsigned attribute. **shall** include a counter signature on the CAdES signature. Syntax per CMS [7] clause 11.4.

#### 5.2.8 content‑time‑stamp
- Signed attribute. Encapsulates one time‑stamp token of the signed data content before signing. Value **shall** be `ContentTimestamp` (TimeStampToken). `messageImprint` **shall** be hash of eContent (attached) or external data (detached), without ASN.1 tag and length.

#### 5.2.9 signature‑policy‑identifier and SigPolicyQualifierInfo
- **5.2.9.1 signature‑policy‑identifier**: signed attribute. Contains explicit identifier of signature policy. Value **shall** be `SignaturePolicyIdentifier`. `signaturePolicyImplied` choice **shall not** be used. Zero‑hash values allowed for unknown policy hash.
- **5.2.9.2 SigPolicyQualifierInfo**: may further qualify the policy. Defines qualifiers: SPuri (URL), SPUserNotice (display text), SPDocSpecification (technical specification OID/URI).

#### 5.2.10 signature‑policy‑store
- Unsigned attribute. Contains the signature policy document or URI to local store. Value **shall** be `SignaturePolicyStore`. `spDocument` **shall** contain encoded document or local URI.

#### 5.2.11 content‑reference
- Signed attribute. Links one `SignedData` to another. Syntax per ESS [3] clauses 1.3.4 and 2.11.

#### 5.2.12 content‑identifier
- Signed attribute. Provides identifier for signed content. Value per ESS [3] clauses 1.3.4 and 2.7.

#### 5.2.13 cms‑algorithm‑protection
- Signed attribute. Contains and protects digest and signature algorithms. Syntax per IETF RFC 6211 [19].

### 5.3 signature‑time‑stamp
- Unsigned attribute. Encapsulates time‑stamp token computed on digital signature value for a specific signer. Value **shall** be `SignatureTimeStampToken`. `messageImprint` **shall** be hash of the `signature` field (without ASN.1 tag and length) within `SignerInfo`.

### 5.4 Attributes for validation data values
- **5.4.1 Introduction**: specifies places to incorporate missing validation data.
- **5.4.2 OCSP responses**: **shall** be incorporated using `OCSPResponse` or `BasicOCSPResponse` (prefer `OCSPResponse`). **should** be added using encoding of `OCSPResponse` as per IETF RFC 5940 [13].
- **5.4.3 CRLs**: **shall** be as defined in IETF RFC 5280 [6].

### 5.5 Attributes for long term availability and integrity of validation material
#### 5.5.2 ats‑hash‑index‑v3
- Unsigned attribute. Provides unambiguous imprint of essential components for archive time‑stamp. Value **shall** be `ATSHashIndexV3`. **shall** be DER encoded. `hashIndAlgorithm` **shall** be same hash algorithm as used for archive time‑stamp. `certificatesHashIndex` **shall** contain hash of every `CertificateChoices` instance; `crlsHashIndex` **shall** contain hash of every `RevocationInfoChoice` instance; `unsignedAttrValuesHashIndex` **shall** contain hash of concatenated `attrType` and each `AttributeValue`. No other hash values **shall** be included.

#### 5.5.3 archive‑time‑stamp‑v3
- Unsigned attribute. Time‑stamp token of signed document and signature, protected by `ats‑hash‑index‑v3`. Value **shall** be `ArchiveTimeStampToken`. Message imprint computation concatenates: (1) `eContentType`, (2) hash of signed data (same algorithm as archive time‑stamp), (3) fields `version`, `sid`, `digestAlgorithm`, `signedAttrs`, `signatureAlgorithm`, `signature` from the relevant `SignerInfo`, (4) single `ATSHashIndexV3` instance. The `ats‑hash‑index‑v3` attribute **shall** be included as unsigned attribute. Before adding new ATSv3, validation data **shall** be extended. If ATSv2 or earlier exists, root `SignedData.certificates` and `crls` **shall not** be modified; new validation data provided inside the latest archive time‑stamp token.

## CAdES Baseline Signatures (Clause 6)
### 6.1 Signature levels
Defines four levels: B‑B (creation), B‑T (time‑stamp), B‑LT (validation material), B‑LTA (long‑term availability/integrity). Each level includes all lower levels.

### 6.2 General requirements
#### 6.2.1 Algorithm requirements
- Algorithms and key lengths **should** be as per ETSI TS 119 312 [i.8]. MD5 **shall not** be used as digest algorithm.

#### 6.2.2 Notation for requirements
Table 1 defines presence/cardinality for each attribute/field/service across levels. Terms: shall be present, shall not be present, may be present, shall be provided, conditioned presence, "*" (should not be present).

### 6.3 Requirements on components and services (Table 1)
Key requirements extracted:
- **SignedData.certificates**: **shall be present** at all levels; generator **shall** include signing certificate (a); **should** include all certificates for path building; **shall** include full set including trust anchors (d).
- **content‑type**: **shall be present**; value id‑data (f).
- **message‑digest**: **shall be present**.
- **Service: protection of signing certificate**: **shall be provided** at all levels via ESS signing‑certificate or signing‑certificate‑v2 (conditioned presence).
- **signing‑time**: **shall be present** at all levels.
- **commitment‑type‑indication**: **may be present**.
- **Service: identifying the signed data type**: **should be present** at all levels via content‑hints or mime‑type (conditioned).
- **signer‑location**, **signer‑attributes‑v2**, **countersignature**, **content‑time‑stamp**, **signature‑policy‑identifier**, **content‑reference**, **content‑identifier**, **cms‑algorithm‑protection**: vary by level (see table).
- **signature‑time‑stamp**: **shall be present** at B‑T, B‑LT, B‑LTA; not at B‑B (=*).
- **Service: revocation values in long‑term validation**: **shall be provided** at B‑LT, B‑LTA via `SignedData.crls.crl` or `SignedData.crls.other` (conditioned).
- **archive‑time‑stamp‑v3**: **shall be provided** only at B‑LTA (cardinality ≥1).
- Additional requirements (a‑t) for certificate inclusion, encoding, etc. (listed after table).

### 6.4 Legacy CAdES baseline signatures
- When new unsigned attributes are added to legacy signatures, they **shall** comply with present document.

## Annex A (normative): Additional Attributes Specification
### A.1 Attributes for validation data
#### A.1.1 Certificates validation data
- **A.1.1.1 complete‑certificate‑references**: unsigned attribute. **Shall** contain references to trust anchor and CA certificates (excluding signing certificate). **Shall not** contain references to CA certificates exclusively used for attribute certificates/signed assertions. Value **shall** be `CompleteCertificateRefs`.
- **A.1.1.2 certificate‑values**: unsigned attribute. **Shall** contain values of certificates referenced in certificates references not already in `SignedData.certificates`. No other certificates **shall** be included. Value **shall** be `CertificateValues`.

#### A.1.2 Revocation validation data
- **A.1.2.1 complete‑revocation‑references**: unsigned attribute. **Shall** contain references to revocation values for signing certificate and CA certificates (except trust anchor). **Shall not** contain references for attribute certificate paths. **Should** be used in preference to `OtherRevocationInfoFormat`. Value **shall** be `CompleteRevocationRefs`.
- **A.1.2.2 revocation‑values**: unsigned attribute. **Shall** contain actual revocation data (CRLs, OCSP responses) corresponding to references not already in `SignedData.crls`. No other elements **shall** be included. Value **shall** be `RevocationValues`.

#### A.1.3 attribute‑certificate‑references
- Unsigned attribute. **Shall** contain references to certificates for signing certificates of attribute certificates/signed assertions not already in `complete‑certificate‑references`.

#### A.1.4 attribute‑revocation‑references
- Unsigned attribute. **Shall** contain revocation references for attribute certificate paths not already in `complete‑revocation‑references`.

#### A.1.5 Time‑stamps on references to validation data
- **A.1.5.1 time‑stamped‑certs‑crls‑references**: unsigned attribute. Encapsulates time‑stamp token of concatenated `complete‑certificate‑references` and `complete‑revocation‑references`. Value **shall** be `TimestampedCertsCRLs`.
- **A.1.5.2 CAdES‑C‑timestamp**: unsigned attribute. Encapsulates time‑stamp token covering signature field, signature‑time‑stamp, complete‑certificate‑references, complete‑revocation‑references. Value **shall** be `ESCTimeStampToken`.

### A.2 Deprecated attributes
- **A.2.1 Usage**: deprecated attributes **shall not** be added to new signatures (exception: long‑term‑validation may be added to signatures already containing it).
- **A.2.2 other‑signing‑certificate**: deprecated; use signing‑certificate‑v2.
- **A.2.3 signer‑attributes**: deprecated; use signer‑attributes‑v2.
- **A.2.4 archive‑time‑stamp (ATSv2)**: deprecated; new ATSv2 **shall not** be created; extend with ATSv3.
- **A.2.5 long‑term‑validation**: deprecated; new instances **shall not** be created; extend with ATSv3.
- **A.2.6 ats‑hash‑index**: deprecated; use ats‑hash‑index‑v3.

## Annex B (normative): Alternative mechanisms for long term availability and integrity of validation data
If an unsigned attribute is used for an alternative mechanism, the following **shall** be specified: (1) semantics/syntax/OID; (2) strategy to protect all necessary parts; (3) handling of signatures with present document attributes; (4) handling of legacy signatures. Such mechanisms do not represent CAdES‑B‑LTA or extended levels.
