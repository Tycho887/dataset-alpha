# ETSI EN 319 122-2 V1.1.1: Electronic Signatures and Infrastructures (ESI); CAdES digital signatures; Part 2: Extended CAdES signatures

## Scope (Summary)
Specifies extended CAdES digital signatures built on CMS signatures (RFC 5652) by incorporating signed/unsigned attributes for long-term validity. Defines additional signature levels (CAdES-E-BES, -EPES, -T, -A, -C, -X, -X-Long, -X-L) that incrementally address requirements. Creation, augmentation, and validation procedures are out of scope (see ETSI EN 319 102-1). Supports regulatory frameworks including eIDAS Regulation (EU) 910/2014.

## 4 Additional CAdES levels without references to validation data

### 4.1 Overview
- Specifies additional CAdES levels built on attributes from ETSI EN 319 122-1 [1].
- Note: Clause 4.3 defines CAdES-E-BES, -EPES, -T, -A built on -T. Annex A defines levels with validation data references.
- Note: Tables use format from Clause 6.2.2 of [1].

### 4.2 General requirements
- **R-GEN-1**: The general CMS syntax **shall** be as specified in ETSI EN 319 122-1 [1], clause 4.
- **R-GEN-2**: The signature **shall** contain a CMS SignedData (per [1] clause 4.4) and at least one SignerInfo (per [1] clause 4.6).
- **R-GEN-3**: Algorithms and key lengths **should** be as specified in ETSI TS 119 312 [i.7]. (Note: National recommendations may supersede.)

### 4.3 CAdES-E-BES, CAdES-E-EPES, CAdES-E-T and CAdES-E-A built on CAdES-E-T signatures
- **R-LEVELS**: CAdES-E-BES, -EPES, -T, -A built on -T **shall** satisfy the requirements in this clause.
- **R-BUILD-EPES**: Built on -BES by adding one `signature-policy-identifier` attribute.
- **R-BUILD-T**: Built on -BES or -EPES by adding one or more `signature-time-stamp` attributes.
- **R-BUILD-A**: Built on -T, -C, -X (Type 1/2), -X-Long, -X-L (Type 1/2).

Additional requirements (letters refer to notes in original Table 1):
- (a) `ESS signing-certificate` **shall** be used if SHA-1 is used.
- (b) `ESS signing-certificate-v2` **shall** be used when hash algorithm other than SHA-1 is used.
- (c) `signature-policy-store` **may** be present only if `signature-policy-identifier` is present and contains `sigPolicyHash` of the policy document; otherwise **shall not** be present.
- (d) `attribute-certificate-references` and `attribute-revocation-references` **may** be present only if at least one certified signer attribute (`certifiedAttributesV2`) or signed assertion (`signedAssertions`) is present; otherwise **shall not**.
- (e) The `issuerSerial` field **should not** be included in `ESSCertID`, `ESSCertIDv2`, or `OtherCertID` types.
- (f) Time-stamp tokens in `signature-time-stamp` **shall** be created before the signing certificate has been revoked or expired.
- (g) Service: full set of certificates (including trust anchors if available) **shall** be included for validation of signing certificate, attribute certificates, revocation info, and TSA certificates.
- (h) Duplication of certificate values **should** be avoided.
- (i) If certificate values are not yet included elsewhere, they **shall** be included in `SignedData.certificates` per [1] clause 5.5.3.
- (j) Full set of revocation data (CRL or OCSP) **shall** be included for validation of signing certificate, attribute certificates, revocation info, and TSA certificates.
- (k) Duplication of revocation values **should** be avoided.
- (l) CRL values **shall** be included in `SignedData.crls.crl` if not already included otherwise.
- (m) OCSP responses **shall** be included in `SignedData.crls.other` if not already included otherwise.
- (n) Before generating `archive-time-stamp-v3`, all required validation material not already in the signature **shall** be included, including material for previous archive time stamps.
- Note 1: Multiple instances of `content-time-stamp`, `signature-time-stamp`, `CAdES-C-timestamp`, `time-stamped-certs-crls-references` may occur from different TSUs.
- Note 2: Signature policy can set specific requirements for other attributes.
- Note 3: Further info on signature policies in ETSI TS 119 172-1 [i.6].
- Note 4: Trusted time indications provide initial steps for long-term validity.

## 5 Legacy signatures
- **R-LEGACY**: When new attributes are incorporated to legacy CAdES TS 101 733 signatures, those attributes **shall** comply with ETSI EN 319 122-1 [1].

## Annex A (normative): CAdES levels with references to validation data

### A.1 CAdES-E-C, CAdES-E-X, CAdES-E-X-Long and CAdES-E-X-L signatures
- **R-C**: CAdES-E-C built on -T by adding references to certificates and status data.
- **R-X**: CAdES-E-X built on -C by adding time-stamp container properties.
  - Type 1: Add one or more `CAdES-C-Timestamp` attributes.
  - Type 2: Add one or more `time-stamped-certs-crls-references` attributes.
- **R-X-Long**: Built on -C by adding certificate and revocation values.
- **R-X-L**: 
  - Type 1: Built on -X Type 1 by adding certificate and revocation values.
  - Type 2: Built on -X Type 2 by adding certificate and revocation values.

Additional requirements for Table A.1:
- (a) In case of direct trust (signing certificate contains trust anchor), CAdES-E-C and any built on it **shall not** be generated.

- (d) Revocation data **shall** be added as per [1] clause A.1.2.2.
- Note 1: In direct trust, references would contain no references.
- Note 2: If signer provided CAdES-E-BES or -EPES, it can later be extended to CAdES-E-C while signature is valid.
- Note 3: Time-stamp tokens may include unsigned attributes for their own validation.
