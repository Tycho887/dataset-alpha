# ETSI EN 319 132-1 V1.3.1: Electronic Signatures and Trust Infrastructures (ESI); XAdES digital signatures; Part 1: Building blocks and XAdES baseline signatures
**Source**: ETSI/ESI | **Version**: V1.3.1 | **Date**: 2024-07 | **Type**: Normative
**Original**: ETSI EN 319 132-1 V1.3.1 (2024-07)

## Scope (Summary)
This European Standard specifies XAdES digital signatures, building on XMLDSIG by incorporating signed and unsigned qualifying properties. It defines XML Schema definitions for qualifying properties and mechanisms for their inclusion. It also specifies four levels of XAdES baseline signatures (B-B, B-T, B-LT, B-LTA) addressing incremental requirements for long‑term validity. Procedures for creation, augmentation, and validation are out of scope (refer to ETSI EN 319 102-1 and ETSI TR 119 100).

## Definitions and Abbreviations
- **attribute certificate**: data structure, digitally signed by an attribute authority, that binds some attribute values with identification information about its holder.
- **certificate revocation list (CRL)**: signed list indicating a set of certificates that are no longer considered valid by the certificate issuer.
- **data object**: actual binary/octet data being operated on (transformed, digested, or signed) by an application.
- **digital signature**: data appended to, or a cryptographic transformation of a data unit that allows a recipient to prove the source and integrity of the data unit and protect against forgery.
- **digital signature value**: result of cryptographic transformation of a data unit that allows a recipient to prove the source and integrity of the data unit and protect against forgery.
- **electronic time-stamp**: data in electronic form which binds other electronic data to a particular time establishing evidence that these data existed at that time. In the case of IETF RFC 3161 updated by IETF RFC 5816, the electronic time-stamp refers to the timeStampToken field within TimeStampResp.
- **legacy XAdES 101 903 signature**: digital signature generated according to ETSI TS 101 903 (V1.4.2).
- **legacy XAdES baseline signature**: digital signature generated according to ETSI TS 103 171 (V2.1.1).
- **legacy XAdES signature**: legacy XAdES 101 903 signature or legacy XAdES baseline signature.
- **message imprint**: digest value of the data that is going to be time-stamped.
- **signature augmentation policy**: set of rules defining technical and procedural requirements for augmentation of one or more digital signatures.
- **signature creation policy**: set of rules defining technical and procedural requirements for creation of one or more digital signatures.
- **signature policy**: signature creation policy, signature augmentation policy, signature validation policy or any combination thereof.
- **signature validation policy**: set of rules defining technical and procedural requirements for validation of one or more digital signatures.
- **trust anchor**: entity that is trusted by a relying party and used for validating certificates in certification paths.
- **validation data**: data that is used to validate a digital signature.
- **XAdES signature**: digital signature that satisfies the requirements specified within the present document or ETSI EN 319 132-2 or ETSI TS 119 132-3.

**Abbreviations**: ASN.1, BER, CA, CD, CER, CRL, DER, ERS, HTTP, MD5, MIME, OCSP, OID, PER, PI, PKI, SAML, SIM, SPO, TSA, TSP, TSU, URI, URL, URN, UTC, XER, XML, XMLDSIG, XSLT.

## 4 General Syntax
### 4.1 General requirements
- XAdES signatures **shall** build on XMLDSIG as specified in [1] by incorporation of XML [5] signed and unsigned qualifying properties.
- Qualifying properties **shall** be instances of XML types using XML Schema syntax [2][3].

### 4.2 XML Namespaces
Namespaces used:
- `http://uri.etsi.org/01903/v1.3.2#`
- `http://uri.etsi.org/01903/v1.4.1#`
- `http://www.w3.org/2000/09/xmldsig#`
- `http://www.w3.org/2001/XMLSchema`

Two XML Schema files accompany the standard: "1913201-XAdES01903v132.xsd" and "1913201-XAdES01903v141.xsd". In case of discrepancies between schema excerpts and the XSD files, the XSD files **shall** take precedence.

### 4.3 The QualifyingProperties container
- **QualifyingProperties** element **shall** act as container for all qualifying information added to an XML signature.
- Qualifying properties are split into signed (cryptographically bound) and unsigned (not bound).
- Syntax (from XSD):
  - `QualifyingPropertiesType` contains `SignedProperties` (optional) and `UnsignedProperties` (optional).
  - `Target` attribute **shall** refer to the `Id` of the corresponding `ds:Signature`.
  - `Id` attribute is optional.
- A XAdES signature **shall not** incorporate empty `QualifyingProperties` elements.

### 4.3.2 SignedProperties
- Contains signed qualifying properties collectively signed by the XML signature.
- A `ds:Reference` in `ds:SignedInfo` **shall** be generated so that `SignedProperties` contributes to the signature value.
- May contain `SignedSignatureProperties` and `SignedDataObjectProperties`.
- Must not be empty.

### 4.3.3 UnsignedProperties
- Contains unsigned qualifying properties (not signed by the XML signature).
- May contain `UnsignedSignatureProperties` and `UnsignedDataObjectProperties`.
- Must not be empty.

### 4.3.4 SignedSignatureProperties
- Contains signed qualifying properties that qualify the XML signature itself or the signer.
- Allowed child elements: `SigningTime`, `SigningCertificate`, `SigningCertificateV2`, `SignaturePolicyIdentifier`, `SignatureProductionPlace`, `SignatureProductionPlaceV2`, `SignerRole`, `SignerRoleV2`, and `xsd:any` from other namespaces.
- `SigningCertificate`, `SignatureProductionPlace`, and `SignerRole` are obsoleted by their V2 counterparts and **shall not** be incorporated.
- Must not be empty.

### 4.3.5 SignedDataObjectProperties
- Contains signed qualifying properties that qualify signed data objects.
- Allowed child elements: `DataObjectFormat`, `CommitmentTypeIndication`, `AllDataObjectsTimeStamp`, `IndividualDataObjectsTimeStamp`, and `xsd:any`.
- Must not be empty.

### 4.3.6 UnsignedSignatureProperties
- Contains unsigned qualifying properties that qualify the XML signature. The signature **shall not** cover the content.
- Allowed child elements: `CounterSignature`, `SignatureTimeStamp`, `CompleteCertificateRefs`, `CompleteRevocationRefs`, `AttributeCertificateRefs`, `AttributeRevocationRefs`, `SigAndRefsTimeStamp`, `RefsOnlyTimeStamp`, `CertificateValues`, `RevocationValues`, `AttrAuthoritiesCertValues`, `AttributeRevocationValues`, `ArchiveTimeStamp`, and `xsd:any`.
- `CompleteCertificateRefs`, `AttributeCertificateRefs`, `SigAndRefsTimeStamp`, `RefsOnlyTimeStamp` (v1.3.2 namespace) are obsoleted by V2 versions (v1.4.1 namespace). `ArchiveTimeStamp` v1.3.2 is obsoleted by v1.4.1.
- Obsoleted properties **shall not** be incorporated.
- Must not be empty.

### 4.3.7 UnsignedDataObjectProperties
- Contains unsigned qualifying properties that qualify signed data objects. Not covered by signature.
- Contains one or more `UnsignedDataObjectProperty` of type `AnyType`.
- May be empty (only used for completeness).

### 4.4 Incorporating qualifying properties into XAdES signatures
#### 4.4.1 General requirements
- Use `ds:Object` auxiliary element for incorporation.
- Two means: direct (child `QualifyingProperties` in `ds:Object`) or indirect (one or more `QualifyingPropertiesReference` in `ds:Object` referencing an external `QualifyingProperties`).
- Restrictions:
  - All `QualifyingProperties` and `QualifyingPropertiesReference` **shall** occur within a single `ds:Object`.
  - At most one instance of `QualifyingProperties` within that `ds:Object`.
  - All signed qualifying properties **shall** occur within a single `QualifyingProperties`.
- XAdES signatures may contain other `ds:Object` elements.

#### 4.4.2 Signing properties
- All signed qualifying properties **shall** be children of `SignedProperties`.
- A `ds:Reference` **shall** be added to the XML signature with `Type` attribute set to `http://uri.etsi.org/01903#SignedProperties`.

#### 4.4.3 QualifyingPropertiesReference
- References an external `QualifyingProperties` (not descendant of `ds:Signature`).
- Contains `URI` attribute (required) and `Id` (optional).
- URI **shall** be a bare-name XPointer fragment.

### 4.5 Managing canonicalization of XML nodesets
- When generating new XAdES signatures, all qualifying properties that provide optional canonicalization algorithm identifier **shall** include that identifier.
- When augmenting legacy signatures, the qualifying property **shall** include the canonicalization algorithm identifier.

## 5 Qualifying properties semantics and syntax
### 5.1 Auxiliary syntax
#### 5.1.1 AnyType
- Allows arbitrary XML elements, text, and attributes. Used for open content.

#### 5.1.2 ObjectIdentifierType
- Contains a permanent identifier (URI or OID encoded as URN or URI).
- Optional `Description` and `DocumentationReferences`.

#### 5.1.3 EncapsulatedPKIDataType
- For incorporating non-XML PKI objects (e.g., certificates, CRLs, OCSP responses, time-stamps).
- Content is base64-encoded PKI object.
- `Encoding` attribute identifies original encoding (DER, BER, CER, PER, XER). Default is DER.

#### 5.1.4 Types for electronic time-stamps management
- **GenericTimeStampType** (abstract base): Two concrete types:
  - **XAdESTimeStampType**: for time-stamps on XAdES components and/or detached signed data objects. Uses `Include` elements for explicit identification or implicit mechanism.
  - **OtherTimeStampType**: for time-stamps on external data not in the signature. Uses `ReferenceInfo` elements containing digests.
- Include mechanism: `Include` elements reference data objects; order defines concatenation for message imprint.

### 5.2 Basic qualifying properties for XAdES signatures
#### 5.2.1 SigningTime
- **Signed** qualifying property.
- Value specifies the time at which the signer claims to have performed the signing process.
- Type: `xsd:dateTime`.

#### 5.2.2 SigningCertificateV2
- **Signed** qualifying property.
- Contains one reference to the signing certificate (digest value + algorithm identifier).
- May contain references to other certificates in the signing certificate path.
- `CertDigest` contains digest of DER-encoded certificate.
- `IssuerSerialV2` optional (base64 of DER-encoded IssuerSerial from RFC 5035).
- URI attribute optional (hint for location).

#### 5.2.3 CommitmentTypeIndication
- **Signed** qualifying property.
- Indicates one commitment made by the signer for a subset or all signed data objects.
- Commitment type expressed as URI.
- May contain qualifiers.

#### 5.2.4 DataObjectFormat
- **Signed** qualifying property for one specific signed data object.
- Contains description, object identifier, MIME type (RFC 2045), encoding.
- At least one of Description, ObjectIdentifier, MimeType **shall** be present.
- ObjectReference attribute required (references `ds:Reference`).

#### 5.2.5 SignatureProductionPlaceV2
- **Signed** qualifying property.
- Specifies address associated with signer (City, StreetAddress, StateOrProvince, PostalCode, CountryName).
- Must not be empty.

#### 5.2.6 SignerRoleV2
- **Signed** qualifying property.
- Encapsulates signer attributes: claimed roles, certified roles (X.509 attribute certificates or other), signed assertions.
- Must not be empty.

#### 5.2.7 Countersignatures
- **5.2.7.1 Countersignature identifier**: URI `http://uri.etsi.org/01903#CountersignedSignature` for `Type` attribute of `ds:Reference`.
- **5.2.7.2 Enveloped countersignatures**: `CounterSignature` unsigned qualifying property contains one countersignature (XMLDSIG or XAdES) that signs the `ds:SignatureValue` of the embedding XAdES signature.
- Countersignature's `ds:DigestValue` is base64 of canonicalized `ds:SignatureValue`.

#### 5.2.8 Time-stamps on signed data objects
- **5.2.8.1 AllDataObjectsTimeStamp**: Signed, encapsulates time-stamp(s) before signature production. Input is concatenation of all `ds:Reference` in `ds:SignedInfo` except the one referencing `SignedProperties`.
- **5.2.8.2 IndividualDataObjectsTimeStamp**: Signed, time-stamps selected `ds:Reference` elements. Uses explicit `Include` with `referencedData="true"`.

#### 5.2.9 SignaturePolicyIdentifier
- **Signed** qualifying property.
- Either an explicit `SignaturePolicyId` (with identifier, optional transforms, hash) or `SignaturePolicyImplied`.
- Signature policy qualifiers: `SPURI`, `SPUserNotice`, `SPDocSpecification`.

#### 5.2.10 SignaturePolicyStore
- **Unsigned** qualifying property.
- Contains the signature policy document (base64) or a local URI.
- Requires `SPDocSpecification`.

### 5.3 SignatureTimeStamp
- **Unsigned** qualifying property.
- Encapsulates time-stamp(s) on `ds:SignatureValue` (canonicalized).

### 5.4 Qualifying Properties for validation data values
#### 5.4.1 Introduction
- XAdES signature may contain certificates and/or revocation data in any of the qualifying properties specified in this clause.

#### 5.4.2 CertificateValues
- **Unsigned**.
- Contains certificates: trust anchor (if exists and not in ds:KeyInfo), CA certificates in signing path (not in ds:KeyInfo), signing certificate (if not in ds:KeyInfo), certificates used to sign revocation status information, and optionally certificates for countersignature validation.
- Do not include certificates already present elsewhere in the signature (recommended).

#### 5.4.3 RevocationValues
- **Unsigned**.
- Contains CRLs, OCSP responses, other revocation data.
- Mandatory for signing certificate and CA certificates (if not in ds:KeyInfo); optional for revocation signers and countersignatures.

#### 5.4.4 AttrAuthoritiesCertValues
- **Unsigned**.
- Contains certificates for attribute certificate/signed assertion signers and their paths.

#### 5.4.5 AttributeRevocationValues
- **Unsigned**.
- Contains revocation data for attribute certificate/signed assertion signers and their paths.

#### 5.4.6 AnyValidationData
- **Unsigned**.
- Contains certificate values and/or revocation values for validating any digital signature present within the XAdES signature (including countersignatures, time-stamps, etc.).
- URI attribute not allowed.

### 5.5 Qualifying properties for long term availability and integrity of validation material
#### 5.5.1 TimeStampValidationData
- **Unsigned**.
- Contains validation data (certificates and revocation values) for full verification of electronic time-stamps.
- URI attribute references the time-stamp container; used when not placed immediately after the container.
- Specific rules for placement when used with `SignatureTimeStamp`, `SigAndRefsTimeStampV2`, `RefsOnlyTimeStampV2`, `ArchiveTimeStamp`.

#### 5.5.2 ArchiveTimeStamp (namespace v1.4.1)
- **Unsigned**.
- Encapsulates time-stamp(s) on all data objects incorporated at the time of generation.
- Purpose: long-term availability and integrity of validation material.
- Generation steps:
  1. Add any missing certificates/revocation data.
  2. Compute message imprint as per clauses 5.5.2.3 (not distributed) or 5.5.2.4 (distributed).
  3. Request time-stamp(s) from TSA.
  4. Build `ArchiveTimeStamp` encapsulating time-stamp(s).
  5. Incorporate as new unsigned property.
  6. In distributed case, add `Include` elements for each time-stamped unsigned property.
- Message imprint computation (not distributed): concatenate canonicalized `ds:Reference` outputs, `ds:SignedInfo`, `ds:SignatureValue`, `ds:KeyInfo`, all unsigned signature properties before this `ArchiveTimeStamp`, and other `ds:Object` elements (excluding the one containing `QualifyingProperties`).

#### 5.5.3 RenewedDigestsV2
- **Unsigned**.
- Used when signed `ds:Manifest` references detached data objects and digest algorithms become weak.
- Contains recomputed digest values with a stronger algorithm.
- Deprecates `RenewedDigests` from V1.1.1.

## 6 XAdES baseline signatures
### 6.1 Signature levels
- **B-B**: basic signed and unsigned properties at signature creation.
- **B-T**: adds a trusted token (time-stamp) proving existence at a certain time.
- **B-LT**: incorporates all validation material (certificates and revocation data) for long‑term availability.
- **B-LTA**: adds archive time-stamp for long‑term integrity and availability.

### 6.2 General requirements
#### 6.2.1 Algorithm requirements
- Algorithms and key lengths should follow ETSI TS 119 312.
- MD5 **shall not** be used as digest algorithm.

#### 6.2.2 Notation for requirements
- Table 2 defines presence and cardinality for each level.
- Columns: Presence in B-B, B-T, B-LT, B-LTA.
- Values: "shall be present", "shall not be present", "may be present", "conditioned presence", "*" (should not be present).
- For services, "shall be provided" and SPO rows.

### 6.3 Requirements on XAdES signature's elements, qualifying properties and services
Key requirements (abridged from Table 2 and notes):
- `ds:KeyInfo/X509Data`: **shall be present** (signing certificate required).
- `ds:SignedInfo/ds:CanonicalizationMethod`: **shall be present** (one of specified URIs).
- `ds:Reference`: **shall be present**.
- `SigningTime`: **shall be present** (claimed UTC time).
- `SigningCertificateV2`: **shall be present** (URI attribute not generated).
- `DataObjectFormat`: conditioned presence; if present, `MimeType` **shall be present**.
- `SignatureProductionPlaceV2`: may be present (not empty).
- `SignerRoleV2`: may be present.
- `CommitmentTypeIndication`: may be present.
- `CounterSignature`: may be present.
- `SignatureTimeStamp`: B-B: 0; B-T and above: **shall be present** (one per element).
- `CertificateValues`: B-B, B-T: *; B-LT, B-LTA: conditioned presence.
- `RevocationValues`: B-B, B-T: *; B-LT, B-LTA: conditioned presence.
- `AnyValidationData`: B-B, B-T: *; B-LT, B-LTA: conditioned presence.
- `CompleteCertificateRefsV2`: B-B, B-T: may be present (0 or 1); B-LT, B-LTA: **shall not be present**.
- `CompleteRevocationRefs`: B-B, B-T: may be present; B-LT, B-LTA: **shall not be present**.
- `SigAndRefsTimeStampV2`: B-B, B-T: may be present; B-LT, B-LTA: **shall not be present**.
- `RefsOnlyTimeStampV2`: same as above.
- Service "Incorporation of validation data for electronic time-stamps": B-LT and B-LTA: **shall be provided**.
- `ArchiveTimeStamp` (v1.4.1): B-LTA: **shall be present** (1 instance).
- `RenewedDigestsV2`: B-LTA: conditioned presence.

Additional requirements (a through cc):
- a-c: `ds:KeyInfo/X509Data` – include signing certificate, all certificates for path-building.
- d-e: canonicalization algorithm values.
- f-g: transforms (base64, XPath, enveloped-signature, XSLT, XPath Filter 2.0, Relationships transform).
- h: `SigningTime` = claimed UTC time.
- i: `SigningCertificateV2` – do not generate URI attribute.
- j: `IssuerSerialV2` should not be included.
- k: `DataObjectFormat` – one per signed data object except SignedProperties; countersignature special case.
- l: cardinalities for DataObjectFormat children.
- m: `SignaturePolicyStore` only if `SignaturePolicyIdentifier` with `SigPolicyHash`.
- n: `SignatureTimeStamp` – one time-stamp per element.
- o: time-stamps created before signing certificate revocation/expiry.
- p-u: certificate and revocation value inclusion rules, avoiding duplication.
- v: revocation values should be in `RevocationValues` or `AnyValidationData` rather than `ds:KeyInfo`.
- x-y: validation data for electronic time-stamps should be in `TimeStampValidationData` or `AnyValidationData`.
- z-aa: `ArchiveTimeStamp` may contain multiple time-stamps; before generation, include all required validation material for signed data.
- bb: `RenewedDigestsV2` should be used when digest algorithms become weak.
- cc: `AnyValidationData` may provide missing certificates/revocation values.

### 6.4 Legacy XAdES baseline signatures
- New unsigned qualifying properties incorporated into legacy signatures **shall** comply with the present document.
