# ETSI TR 119 476-1: Electronic Signatures and Trust Infrastructures (ESI); Selective disclosure and zero-knowledge proofs applied to Electronic Attestation of Attributes; Part 1: Feasibility study
**Source**: ETSI | **Version**: V1.3.1 | **Date**: 2025-08 | **Type**: Informative (Technical Report)
**Original**: ETSI TR 119 476-1 V1.3.1 (2025-08)

## Scope (Summary)
This report provides a comprehensive analysis of cryptographic schemes (signature schemes, formats, protocols) for selective disclosure, unlinkability, and predicate proofs in the context of (Qualified) Electronic Attestation of Attributes ((Q)EAAs) and Person Identification Data (PID). It evaluates applicability to eIDAS2 and the EUDI Wallet, serving as input to ETSI standards on issuance policies, profiles, and wallet interfaces.

## Definitions and Abbreviations
### Terms
- **selective disclosure**: Capability of the EUDI Wallet that enables a user to present a subset of attributes from a PID and/or (Q)EAA.
- **unlinkability**: Lack of information required to connect a user's selectively disclosed attributes beyond what is disclosed. Types: verifier unlinkable, issuer unlinkable, fully unlinkable.
- **predicate proof**: Verifiable computation on attributes or cryptographic meta-data in a (Q)EAA, where only the result (e.g., boolean, range) is shared, not the underlying value.
- **atomic (Q)EAA**: (Q)EAA with a single attribute claim.
- **salted attribute hashes**: Combining each attribute with a random salt, hashing the pair, and including the digest in a signed list.
- **multi-message signature scheme**: Signature scheme that can sign multiple messages and allows selective disclosure of any subset via ZKP (e.g., BBS+, CL, PS-MS).
- **programmable ZKP**: ZKP for arbitrary statements (arithmetic circuits), e.g., zk-SNARKs, zk-STARKs.

### Abbreviations
- (Q)EAA – (Qualified) Electronic Attestation of Attributes
- PID – Person Identification Data
- MSO – Mobile Security Object
- SD-JWT – Selective Disclosure JSON Web Token
- ZKP – Zero-Knowledge Proof
- CL – Camenisch-Lysyanskaya
- BBS+ – Boneh-Boyen-Shacham+ (multi-message signature)
- ARF – Architecture and Reference Framework
- QTSP – Qualified Trust Service Provider
- PIDP – Person Identification Data Provider
- IACA – Issuing Authority Certification Authority
- VICAL – Verified Issuer Certificate Authority List
- SOG-IS – Senior Officials Group Information Systems Security
- QSC – Quantum-Safe Cryptography

## Selective Disclosure Signature Schemes (Clause 4)
### Atomic (Q)EAA schemes (4.2)
- (Q)EAA with a single attribute claim; issued on demand or batchwise. Achieves verifier unlinkability if batch issued; fully unlinkable not possible.
- Supports SOG-IS approved algorithms and QSC.

### Salted attribute hashes (4.3)
- **General**: Each attribute combined with a random salt, hashed, and included in a signed list. User presents selected attributes with corresponding salts. Verifier rehashes and compares. Provides selective disclosure; verifier unlinkable only if unique salts and keys per presentation.
- **Issuance (4.3.2)**: Issuer parses attributes, salts, hashes, orders in signed list.
- **Presentation/verification (4.3.3)**: Wallet sends disclosed attributes and signed hash list; verifier checks signature, rehashes, and compares.
- **Unlinkability (4.3.4)**: Verifier unlinkable requires unique salts and keys (e.g., via ARKG). Not full unlinkability.
- **HashWires (4.3.7)**: Hash-chain based range proof; plausible quantum-safe.
- **ACDC (4.3.8)**: Directed Acyclic Graph structure for selective disclosure; verifier unlinkable.
- **Gordian Envelopes (4.3.9)**: Hierarchical structure with salting; verifier unlinkable.

### Multi-message signature schemes (4.4)
- **CL signatures (4.4.1)**: Based on RSA or bilinear pairings; supports Pedersen commitments and ZKP for predicates. Fully unlinkable. Not SOG-IS approved, not quantum-safe.
- **BBS+ (4.4.2)**: Pairing-based; constant-size proofs; supports ZKP for selective disclosure and unlinkability. IRTF CFRG draft [i.177]. Not SOG-IS approved, not quantum-safe. Device binding possible via split signatures or P-256 binding.
- **BBS# (4.4.3)**: Pairing-free variant using ECDSA/ECSDSA holder binding; achieves full unlinkability. Uses MAC_BBS for issuer side. Compatible with SOG-IS curves.
- **Mercurial signatures (4.4.4)**: Allow signature transformation to equivalent keys; fully unlinkable when blinded. Not quantum-safe.
- **PS-MS (4.4.5)**: Pairing-based; efficient for distributed ABC. Fully unlinkable. Not SOG-IS approved.
- **ISO standardization (4.4.6)**: ISO/IEC 20008 (anonymous signatures), ISO/IEC 24843 (privacy-preserving ABC, including BBS+/PS), ISO/IEC CD 27565 (ZKP guidelines).

### Proofs for arithmetic circuits (programmable ZKPs) (4.5)
- **zk-SNARKs (4.5.2)**: Succinct, non-interactive, zero-knowledge. Trusted or transparent setup. Cryptographic analysis: quantum-safety depends on underlying primitives (see table A.4). Fully unlinkable.
- **zk-STARKs (4.5.3)**: Transparent, post-quantum secure (hash-based). Proofs larger than zk-SNARKs.
- **Bulletproofs (4.5.4)**: Inner product arguments; range proofs; no trusted setup. Relevant for inequality tests.

## (Q)EAA Formats with Selective Disclosure (Clause 5)
### Atomic (Q)EAA formats (5.2)
- **PKIX X.509 attribute certificate (5.2.2)**: Single attribute, ASN.1/DER; SOG-IS and QSC compatible. Mature.
- **W3C VC with atomic attribute (5.2.3)**: JWT/JSON-LD; SOG-IS and QSC compatible. Mature.

### Salted attribute hashes formats (5.3)
- **IETF SD-JWT (5.3.2)**: JWS with salted attribute hashes; supports selective disclosure and key binding (SD-JWT+KB). Draft but widely adopted per ARF. SOG-IS/QSC.
- **ISO/IEC 18013-5 MSO (5.3.3)**: CBOR with signed value digests; device binding key. SOG-IS/QSC. Mature.

### Multi-message signature formats (5.4)
- **W3C VC Data Model with ZKP (5.4.1)**: Supports BBS+, CL, Idemix, etc. JSON-LD.
- **W3C Data Integrity BBS Cryptosuite (5.4.2)**: W3C draft; uses BBS+; not SOG-IS, not QSC.
- **Hyperledger AnonCreds (5.4.4)**: CLRSA signatures; JSON; fully unlinkable. Not SOG-IS/QSC.

### JSON container formats (5.5)
- **IETF JSON WebProof (JWP) (5.5.1)**: Generic container for ZKP/signature proofs; supports multiple payloads.
- **W3C JSON Web Proofs for Binary Merkle Trees (5.5.2)**: Merkle audit path encoding; plausible QSC.
- **JSON Web Zero Knowledge (JWZ) (5.5.3)**: Iden3 standard; Groth16 proofs; payload and ZKP in one token.

## Selective Disclosure Systems and Protocols (Clause 6)
### Atomic attribute protocols (6.2)
- **PKIX X.509 AC (6.2.1)**: Short-lived attribute certificates for authorization. SOG-IS/QSC.
- **VC-FIDO (6.2.2)**: W3C VC with FIDO2 for enrolment; prototype.

### Salted attribute hashes protocols (6.3)
- **OpenAttestation (6.3.1)**: Ethereum-based; salted hash chains. Plausible QSC.

### Multi-message signature protocols (6.4)
- **Hyperledger AnonCreds (6.4.1)**: CLRSA-based; fully unlinkable. Widely deployed.
- **DAA with TPMs (6.4.2)**: ISO/IEC 20008; BBS or PS-MS; used in TPM 2.0. Not QSC.

### Programmable ZKP solutions (6.5)
- **Anonymous credentials from existing identities + zk-SNARKs (6.5.1)**: General model; reuses legacy formats; supports full unlinkability and arbitrary predicates. Immature.
- **Cinderella (6.5.2)**: X.509 + Pinocchio; offline proof generation (~minutes). Prototype.
- **zk-creds (6.5.3)**: ICAO passports + zk-SNARKs; prototype.
- **Anonymous credentials from ECDSA (6.5.4)**: mdoc + optimised ZKP; ~1.2s on mobile; standardization in ISO/IEC SC17 WG10 and IETF CFRG.
- **Crescent (6.5.5)**: JWTs/mdoc + Circom/Groth16; 20ms per session after preprocessing.
- **Analysis (6.5.6)**: Benefits: legacy reuse, predicates, post-quantum options. Challenges: proving time, complexity, standardization.

### Anonymous attribute-based credentials systems (6.6)
- **Idemix (6.6.1)**: CL-based; full unlinkability. Not QSC.
- **U-Prove (6.6.2)**: Blind signatures based on DLP; single-show unlinkability. Not QSC.
- **ISO/IEC 18370 (6.6.3)**: Blind signatures (U-Prove mechanism 4). Not recommended due to ROS vulnerability and lack of multi-show.
- **KVAC (6.6.4)**: Algebraic MACs; requires issuer-verifier same entity. BBS# overcomes this: publicly verifiable, pairing-free.
- **FIDO-AC (6.6.5)**: FIDO2 + zk-SNARKs for ICAO eMRTD; proving time ~3s.

### ISO mDL flows (6.7)
- **Device retrieval (6.7.2)**: NFC/BLE; MSO + DeviceSigned. Mature. SOG-IS/QSC.
- **Server retrieval (6.7.3)**: WebAPI or OIDC. Mature. SOG-IS/QSC.
- **Unattended flow (ISO/IEC 18013-7) (6.7.4)**: REST or OID4VP. Draft.
- **ISO/IEC 23220-4 (6.7.5)**: Operational protocols for wallet; supports mdoc, SD-JWT VC, OID4VP. Draft.

### OpenID for Verifiable Credentials (OpenID4VC) (6.8)
- **OID4VCI (6.8.1)**: Issuance protocol; supports batch issuance, key attestations, ARKG.
- **OID4VP (6.8.2)**: Presentation protocol; DCQL for queries; format-agnostic.
- **HAIP (6.8.3)**: High Assurance profile for OID4VCI/OID4VP; mandates X.509, response encryption, specific algorithms.

### Iden3 protocol (6.9)
- Uses Groth16 on BN254, BabyJubJub EdDSA, Poseidon hash, Sparse Merkle Trees. Supports selective disclosure, predicates, revocation. Compatible with blockchain.

## Implications for (Q)EAA/PID Standards (Clause 7)
### General (7.1)
- Focus on mdoc (ISO/IEC 18013-5) and SD-JWT as per ARF; also BBS+/BBS# and programmable ZKPs for future.

### mdoc with selective disclosure (7.2)
- **QTSP/PIDP issuing**: Use ISO mDL IACA PKI; certificate profiles in line with SOG-IS (e.g., Brainpool curves). EU TL can bridge to VICAL.
- **Trusted Lists**: Recommend ETSI TS 119 612 add CDDL/CMS profile or ISO add XML profile.
- **Issuance**: Batch MSOs with unique salts for verifier unlinkability; predetermined predicates (e.g., age_over_NN). MSO signed with COSE, SOG-IS/QSC.
- **Authentication key**: mdoc authentication key used for device binding; can use HDK for unlinkability.
- **Flows**: Device retrieval (NFC/BLE) and unattended (REST/OID4VP) specified.

### SD-JWT selective disclosure (7.3)
- SD-JWT VC as QEAA format; verifier unlinkability via batch issuance with unique keys and salts.
- Issuer = QTSP/PIDP; verifier uses EU TL.
- Signed with JOSE; supports SOG-IS/QSC.

### Feasibility of BBS+ and BBS# (7.4)
- **BBS+**: Not fully standardized (ISO 24843 in progress); pairing-based, not SOG-IS approved. Provides full unlinkability. Post-quantum soundness not assured.
- **BBS#**: Pairing-free, uses SOG-IS curves (ECDSDA/ECSDSA); full unlinkability. Standardization ongoing. Post-quantum soundness not assured but anonymity unconditional.

### Feasibility of programmable ZKPs (7.5)
- Solutions like Cinderella, zk-creds, Crescent, and Frigo-Shelat can be applied to eIDAS2 credentials.
- Benefits: full unlinkability, arbitrary predicates, compatibility with legacy formats (X.509, mdoc, JWT).
- Challenges: proving time, memory, complexity, standardization.

### Secure storage of keys (7.6)
- EUDI Wallet must store keys in secure hardware (SE/TEE). Key splitting (for BBS#) allows use of conventional ECDSA.

### Proportionality of privacy goals (7.7)
- Weigh selective disclosure, unlinkability, predicates against costs. Batch issuance provides verifier unlinkability at issuer cost. Full unlinkability requires more advanced crypto (BBS#, zk-SNARKs).

## Privacy Aspects of Revocation and Validity Checks (Clause 8)
- **OCSP Must-Staple** (8.2): Privacy-preserving if used with stapling.
- **Revocation lists** (8.3): Can use Private Information Retrieval (PIR) or Private Set Intersection (PSI) to protect privacy.
- **Validity status lists** (8.4): Similar to CRLs; can be combined with ZKPs.
- **Cryptographic accumulators** (8.5): Efficient and privacy-preserving but complex.
- **Programmable ZKPs** (8.6): Combine with existing credentials; wallet performs validity checks and reveals only result.
- **Conclusions** (8.7): Recommend OCSP Must-Staple, privacy-enhanced lists, and eventually accumulators.

## Post-Quantum Considerations (Clause 9)
- **Threats**: Quantum computers break DLP-based algorithms (ECDSA, RSA, pairing-based). Salted hash based schemes can use QSC signatures.
- **Solutions**: Atomic (Q)EAAs and salted hash formats (mdoc, SD-JWT) can adopt NIST standard QSC (CRYSTALS-Dilithium, SPHINCS+). Multi-message signature schemes (BBS+, CL) are not quantum-safe for soundness but preserve data confidentiality of undisclosed attributes.
- **Lattice-based anonymous credentials**: Research underway (e.g., [i.29], [i.10], [i.197]); plausible post-quantum.
