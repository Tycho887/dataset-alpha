# ETSI TR 119 479-2: Electronic Signatures and Trust Infrastructures (ESI); Technological Solutions for the EU Digital Identity Framework; Part 2: EAA Extended Validation Services Framework and Application
**Source**: ETSI | **Version**: V1.1.1 | **Date**: 2025-07 | **Type**: Technical Report (Informative)
**Original**: [ETSI Search & Browse Standards](https://www.etsi.org/standards) | **Keywords**: attribute attestation, digital identity, EAA, encryption, EUDI Wallet, policies, trust services

## Scope (Summary)
This document proposes an advanced attestation validation schema for the EUDI Wallet, introducing a “pricing_policy” that preserves user privacy and unlinkability while enabling direct privacy‑preserving communication between Attestation Provider and Relying Party. It covers attestation refreshing, cyphered attestation presentation, and extensions to attestation metadata for policy parameters.

## 4 Extended Validation Service(s)
### 4.1 Overview
The proposal involves two steps that do not require changes to existing attestation issuance and are format-agnostic:
1) **Attestation Refreshing**: attestations are refreshed before presentation to guarantee validity without the RP needing additional controls, except integrity.
2) **Cyphered Attestation Presentation**: attestations are encrypted by the User's Wallet before sharing with RP; RP can only access through direct communication with User and Attestation Provider, allowing the Attestation Provider to count verifications per RP without revealing User information.

### 4.2 Attestation Refreshing
- **Linked credentials**: Attestation Provider provides a Status Assertion [i.5] linked to an attestation; User presents both to RP as proof of validity status.
- **Credential reissuance**: based on multiple access to RP endpoint (clause 13.5 in OID4VCI [i.10]); Wallet can retrieve updated attestation using valid Access Token or Refresh Token without User interaction; if no valid tokens, Attestation Provider should reissue from beginning (requires User interaction). Re-issuance means replacement of a PID or attestation already in a Wallet Unit by one with same document type (see ARF Topic B [i.4]).
- **Batch Issuance**: instead of issuing a single PID or attestation, issuer issues a batch; re-issuance also in batch (see ARF Topic B [i.4]).

Refreshing may occur at Wallet startup, on demand, or just before presentation. These approaches should apply only to attestations that require refreshing. RP can implement own policies (e.g., accept attestation refreshed within last N hours).

### 4.3 Cyphered Attestation Presentation
Encrypts attestation at presentation time using Elliptic Curve Cryptography and Hierarchical Deterministic (HD) Key Derivation. Attestation is encrypted as JWE [i.7]; encryption key derived from Attestation Provider's public key. RP must contact Attestation Provider to obtain decryption key.

**Flow**:
- **Refreshing**:
  1) Wallet sends Attestation Provider request to refresh attestation.
  2) Attestation Provider sends updated attestation (or notifies it is no longer valid).
- **Cyphered Attestation Presentation**:
  1) Wallet generates transaction-specific symmetric key K, encrypts attestation A → K(A).
  2) Wallet generates JWE containing: X (timestamp + random nonce), KID (to identify Attestation Provider), CP(K) (K encrypted using derived public key CP), and body containing K(A).
  3) RP receives JWE, sends CP(K) and X to Attestation Provider, requesting decryption.
  4) Attestation Provider uses derived private key corresponding to X to decrypt CP(K) → obtains K, sends K to RP.
  5) RP uses K to decrypt K(A) → retrieves A.

## 5 Central Rulebook for Attributes
### 5.1 Overview
Introduces a **pricing_policy** to complement existing policies (user consent, disclosure). Leverages Attestation Rulebooks catalogue [i.4] to standardize structure and pricing. Cyphered Attestation Presentation is format-agnostic and supports pricing options such as:
- Free-to-use attributes
- Issuance-based fee
- Verification-based fee

### 5.2 Attestation Metadata Extension
Attestation Providers could use the Attestation Rulebooks catalogue to publish attributes. A new "pricing_policy" parameter could be introduced in attestation metadata. Pricing policy may originate from embedded disclosure policy.

**Price models**:
1) **Static price**: fixed cost embedded in attribute schema, unit price per verification.
2) **Dynamic price**: cost fluctuates; stakeholders check a specific URL for updated prices.

OID4VP extension (Figure 3 in original):
1) **Discovery Phase**: RP queries central registry for attributes, providers, costs; specifies willingness to pay.
2) **Attestations Exchange**: uses encrypted attestation exchange (clause 4.3).
3) **Verification Cost Exchange**: Attestation Provider verifies payment conditions; secure payment protocol may be used.

## 6 Security Considerations
1) Threat of CRQCs breaking current cryptography (RSA, ECC). "Harvest now, decrypt later" motivates transition to quantum-resistant crypto.
2) Only approved curves and parameters should be used (e.g., NIST SP 800-56A Rev. 3 [i.9], ETSI TS 119 312 [i.2], SOG-IS [i.13]).
3) Security guaranteed (excluding CRQCs) by Discrete Logarithm Problem and non-reversibility of hash functions.
4) HD key mechanism not formally standardized by a formal SDO (described in clause 4.4.4.2 of ETSI TR 119 476 [i.3]).
5) Mechanism does not suffer from private key leakage (BIP 32 [i.1]) because Attestation Provider's private key is never exposed publicly.

## 7 Privacy Considerations
1) Collusion between Attestation Provider and RP could allow decryption of JWE if RP shares entire JWE with Attestation Provider.
2) If Attestation Provider generates a new key pair for each request, tracking of Wallet User activity may occur; therefore controlling the number of public keys for Attestation Provider is advisable.
3) Privacy preserved because:
   - Cryptographic material (e.g., CP) is derived by Wallet from Attestation Provider's public key without Provider’s awareness; multiple child public keys can be generated from same parent.
   - Attestation Provider never gets attestation in plain unless RP explicitly shares it without User's consent.
4) In worst case where only one attestation is issued, it could be linked to the Wallet User.

## Informative Annexes (Condensed)
- **Annex A: Hierarchical Deterministic (HD) Key Derivation**: Describes the mathematical mechanism used in clause 4.3 for deriving child public keys from a master public key using HMAC-SHA512, based on BIP-0032 [i.1]. It explains how Alice (Attestation Provider) and Bob (Wallet User) generate corresponding private and public keys such that the Attestation Provider can derive child private keys while the Wallet derives child public keys, enabling encryption without exposing the master private key.
