# ETSI TR 119 530 V1.1.1: Registered Electronic Mail (REM); Feasibility study: Interoperability profile between ETSI EN 319 532-based REM systems and PReM-based systems
**Source**: ETSI Technical Committee ESI | **Version**: V1.1.1 | **Date**: 2019-02 | **Type**: Informative Technical Report
**Original**: [ETSI TR 119 530](https://www.etsi.org/deliver/etsi_tr/119500_119599/119530/01.01.01_60/)

## Scope (Summary)
This feasibility study examines the gaps between the latest REM (ETSI EN 319 532) and PReM (UPU S52-2) specifications, and between their historical predecessors (ETSI TS 102 640 and UPU S52-1). It provides recommendations for updating UPU S52-2 and for producing a new interoperability profile to enable REM-PReM gateway operation.

## 4 Goals and Approaches
This feasibility study includes:
- Gap analysis between ETSI EN 319 522/532 and the historical ETSI TS 102 640 regarding aspects covered within UPU S52-2 [i.25].
- Gap analysis between UPU S52-2 and UPU S52-1 regarding interoperability with REM-based systems.
- Recommendations for updating UPU S52-2 / CEN/TS 16326.
- Recommendations for producing a new interoperability profile between updated UPU S52-2 and ETSI EN 319 532.

## 5 Gap Analysis

### 5.1 REM – ETSI TS 102 640 vs ETSI EN 319 532

#### 5.1.1 Introduction
ETSI EN 319 522/532 include modifications to evidence schemas, new evidence types, changes in message contents, etc., compared to ETSI TS 102 640. Clauses 5.1.2–5.1.6 outline these differences. Tables summarize semantics and syntactical differences; only definitions relevant to the gap analysis appear.

#### 5.1.2 Changes on Terms and Boundary Roles
- **Consignment**: (EN) Act of making user content available to recipient within ERDS boundaries. (TS) Not defined.
- **ERDS evidence**: (EN) Data generated by ERDS to prove event occurred. (TS) Same, but called REM-MD evidence.
- **Handover**: (EN) Act of user content crossing recipient's ERDS border. (TS) Not defined.
- **Original message**: (EN) Data including user content and submission metadata. (TS) Similar.
- **REMS/REM-MD**: (EN) Entity providing REM service. (TS) Same, but term REM-MD used.
- **REM dispatch/REM envelope**: (EN) Signed structure containing user content, relay metadata, and evidence. (TS) REM-MD Envelope.
- **REM message**: (EN) Data composed of user content, metadata, zero or more evidence. (TS) REM-MD Message.
- **REM payload**: (EN) REM message containing user content and relay metadata. (TS) Not defined.
- **REM interoperability domain/REM Policy Domain**: (EN) Homogeneous operational space of REMSPs able to interoperate. (TS) Similar.
- **REMS notification**: (EN) Data with relay metadata and evidence referencing user content. (TS) Not defined.
- **REMS receipt**: (EN) Data with relay metadata and evidence. (TS) Not defined.
- **User content**: (EN) Original data produced by sender. (TS) Not defined.
- **REM Object**: (TS) Message object handled by REM-MD. (EN) Not used.

*Boundary roles/components*:
- **REMS message delivery agent**: (EN) Equivalent to ERDS Message delivery system; ensures user content is made available to recipient. (TS) REM-MD Message Transfer Agent.
- **REMS evidence provider**: (EN) Produces ERDS evidence. (TS) REM-MD Evidence Provider.
- **REMS evidence repository**: (EN) Storage of evidence; optional third party. (TS) REM-MD repository.
- **REMS message store**: (EN) Mailboxes for senders/recipients. (TS) REM Message Store.
- **REMS user directory**: (EN) Translates recipient identification to delivery endpoint. (TS) Part of REM-MD repository.
- **REMS evidence verifier**: (EN) Task performed by REMS. (TS) Role supporting verification.
- **Registered Electronic Mail Service Provider (REMSP)**: (EN) Entity providing REM service, may also act as gateway. (TS) REM-MD.
- **ERD User Agent/Application (ERD-UA)**: (EN) Software/hardware by which senders/recipients participate. (TS) REM User Agent.
- **Recipient**: (EN) Natural or legal person to which user content is addressed. (TS) Physical or legal entity responsible for mailbox.
- **Sender**: (EN) Natural or legal person that submitted user content. (TS) Physical or legal entity responsible for mailbox.
- **REM Third Party**: (EN) Party authorized to access REM objects. (TS) Not defined.

#### 5.1.3 Changes on Event and Evidence Types
Events defined in EN 319 522-1 vs TS 102 640-1 mapped to flows in TS 102 640-6-1 tables (4–7):
- **RelayAcceptance** (EN B.1) vs **R-REM-MD Acceptance** (TS): Receiving REMS accepted relayed message.
- **RelayRejection** (EN B.2) vs **R-REM-MD Rejection**: Receiving REMS rejected; shall inform sending REMS.
- **RelayFailure** (EN B.3) vs **Expiration of time to deliver to R-REM-MD**: Unable to relay within time period.
- **ContentConsignment** (EN D.1) vs **Message Delivery** (TS): User content made available to recipient.
- **ContentConsignmentFailure** (EN D.2) vs **Expiration of time to deliver message**: Could not make available within time.
- **ContentHandover** (EN E.1) vs **Retrieval** (TS): User content passed through REM MRI to recipient client.
- **ContentHandoverFailure** (EN E.2) vs **Expiration of time for Retrieval**: Did not pass within given time.

*Evidence mapping* (EN 319 522-1 vs TS 102 640-2):
- **RelayToREMMDAcceptanceRejection** (EN) combines acceptance and rejection; (TS) has separate Acceptance/Rejection evidence.
- **RelayToREMMDFailure** (EN) attests impossibility to relay; (TS) similar.
- **DeliveryNonDeliveryToRecipient** (EN) attests consignment success/failure; (TS) similar but with different reason codes.
- **RetrievalNonRetrievalByRecipient** (EN) attests handover success/failure; (TS) similar.

##### 5.1.4.1 Introduction
Summarizes semantics and syntactical differences between REM message components in EN 319 532-3 vs TS 102 640-2. Tables follow same format.

##### 5.1.4.2 Metadata as Optional Extension Headers
| # | Field (TS) | In TS 102 640-2 | In EN 319 532-3 |
|---|---|---|---|
| 1 | REM-Message-Type (X-REM-Msg-Type) | Optional Extension header | **Required** URI value identifying REM message type as per cl. 4.3.5 [i.6] and 6.1 [i.3] |
| 2 | REM-MessageDigest (X-REM-hashValue) | Optional Extension header for digest | New REM-MessageDigest header for same purpose |
| 3 | REM-DigestAlgorithm (X-REM-hashAlgorithm) | Optional Extension for algorithm | New REM-DigestAlgorithm header |
| 4 | REM-UAMessageIdentifier (X-REM-UAMessageIdentifier) | Optional Extension for UA identifier | New REM-UAMessageIdentifier header |
| 5 | Other X-REM-<component> | Allowed for additional info | New headers: REM-MetadataVersion, REM-RelayDate, REM-ExpirationDate, REM-RecipientAssuranceLevel, REM-ApplicablePolicy, REM-ModeOfConsignment, REM-ScheduledDelivery, REM-EventIdentifier |
| Note: X-REM-EvidenceType and X-REM-EventCode (TS 102 640-5) are replaced by REM-EventIdentifier. |

##### 5.1.4.3 Outermost MIME Section Header
- **Content-Type**: Both `multipart/signed` with `protocol=application/pkcs7-signature`. (EN) recommends `micalg` conformant to ETSI TS 119 312; (TS) to ETSI TS 102 176-1.
- **To**: (TS) always matches original message 'To'. (EN) if carrying evidence for sender, can match 'From'; for dispatch/payload, equals original 'To'.
- **Reply-To**: (TS) always matches original 'From'. (EN) for evidence to sender, recommends absence; if present, recommends REM service address.
- **Cc**: (TS) unspecified. (EN) requirements in cl. 6.2.1 [i.3].
- **Subject**: (TS) "REM Dispatch: <subject>" or "REM Delivery Receipt: <subject>". (EN) "REM <event identifier>: <original subject>".
- **Return-Path, Received, In-Reply-To**: (TS) not specified. (EN) requirements in cl. 6.2.1 [i.3].

##### 5.1.4.4 Signed Data MIME Section Header
- **Content-Type**: Both `multipart/mixed`. (EN) recommends boundary per RFC 2046.

##### 5.1.4.5 Introduction MIME Section Header
- **Content-Type**: Both `multipart/alternative`.
- **REM-Section-Type**: Both use `rem_message/introduction`.
- **FREE-TEXT**: (EN) Content-Type `text/plain`, charset UTF-8, Content-Disposition inline, Content-Transfer-Encoding 7bit/8bit/quoted-printable. (TS) similar, but Content-Transfer-Encoding "7bit" only.
- **HTML**: (EN) Content-Type `text/html`, charset UTF-8, Content-Transfer-Encoding 7bit or quoted-printable. (TS) "7bit or quoted-printable".

##### 5.1.4.6 Original Message MIME Section Header
| Field | TS | EN |
|---|---|---|
| Content-Type | `message/rfc822`, name=AttachedMimeMessage | Same, name value as parameter |
| Content-Disposition | attachment, filename=AttachedMimeMessage | attachment, filename same as name parameter |
| Content-Transfer-Encoding | 7bit | binary |
| Content-Description | unspecified | brief text description |
| REM-Section-Type | `rem_message/original` | Same |

##### 5.1.4.7 Extensions MIME Section Header
| Field | TS | EN |
|---|---|---|
| Content-Type | `application/xml`, name=REMExtension.xml | `application/xml` or `application/octet-stream`, name variable, charset UTF-8 recommended |
| Content-Disposition | attachment, filename same as name | Same |
| Content-Transfer-Encoding | quoted-printable | quoted-printable, base64, or binary |
| Content-Description | unspecified | brief text description |
| REM-Section-Type | `rem_message/extension` | Same |
| REM-Extension-Namespace-URI | not specified | namespace URI |
| REM-Extension-Code (X-REM-Extension-Code) | Unique code identifying type | Same |

##### 5.1.4.8 Evidence MIME Section Header
- **XML Evidence**: (TS) Content-Type `application/xml`, name `<REM_EVIDENCE_NAME>.xml`, charset URf-8 (typo in original), Content-Transfer-Encoding quoted-printable. (EN) same, but charset UTF-8; Content-Transfer-Encoding can be base64 or binary; Content-Description required; REM-Section-Type `rem_message/xml_evidence`.
- **PDF Evidence**: (TS) Content-Type `application/pdf', name `<REM_EVIDENCE_NAME>.pdf`, Content-Transfer-Encoding base64. (EN) same but binary allowed; Content-Description required; REM-Section-Type `rem_message/pdf_evidence`.
- **Note**: New REM standard no longer specifies ASN.1 evidence; ERDS evidence set is XML. PDF is allowed but format out of scope.

##### 5.1.4.9 Signature MIME Section Header
No differences between TS and EN: Content-Type `application/pkcs7-signature`, name `smime.p7s`, Content-Disposition attachment, filename `smime.p7s`, Content-Transfer-Encoding base64, Content-Description "S/MIME Cryptographic Signature".

#### 5.1.5 Changes on Evidence Structure and Semantic
Key differences in ERDS Evidence (EN 319 522-3) vs REM Evidence (TS 102 640-2):
- **Target namespace**: TS: `http://uri.etsi.org/02640/v1#` ; EN: `http://uri.etsi.org/19522/v1#`
- **Version attribute**: TS: "1.2.1" ; EN: "EN319522v1.1.1"
- **Root element**: TS has multiple elements per evidence type (e.g., `SubmissionAcceptanceRejection`); EN has single `Evidence` element of type `EvidenceType`, with `ERDSEventId` identifying the event.
- **EventReasons**: EN changes some reason codes and adds new ones (cl. 5.2.2.7 of [i.6]).
- **EvidenceIssuerDetails**: TS uses `rem:EntityDetailsType` with `NamesPostalAddresses`, `ElectronicAddress`/`AttributedElectronicAddress`, `CertificateDetails` (xades:CertIDType). EN uses `EntityDetailsType` with `Identity` (SAML Attribute), `CertificateDetails` (CertIDTypeV2), and drops `AttributedElectronicAddress`.
- **SenderDetails**: TS uses `EntityDetailsType`. EN uses `UserDetailsType` with mandatory `Identity`, `Identifier` (string with scheme attribute), `AssuranceLevelsDetails` (identification/authentication assurance levels).
- **RecipientDetails**: TS sequence of `EntityDetailsType`. EN one or more `UserDetailsType` instances.
- **RecipientsDelegateDetails**: TS `RecipientsDelegateType` with `DelegateDetails` (EntityDetailsType) and optional `DelegatingRecipients` list. EN uses `RecipientsDelegatesDetailsType` extending `UserDetailsType` with `DelegatingRecipients` list.
- **SenderDelegateDetails**: Not in TS. EN includes optional `SenderDelegateDetails` (instance of `DelegateDetailsType`, same as UserDetailsType, no delegating list).
- **MessageIdentifier**: TS present as `MessageIdentiferByREMMD` in `SenderMessageDetails`. EN optional `MessageIdentifier` child of Evidence.
- **UserContentInfo**: TS has `DigestValue` and `DigestMethod` in `SenderMessageDetails`. EN has `UserContentInfo` with `AppLayerIdentifier`, `ComposingParts`, `PartsInfo` (identifier, contentType, digest for each part).
- **ExternalERDSDetails**: Not in TS. EN optional, instance of `EntityDetailsType`.
- **ExternalSystem (ForwardedToExternalSystem)**: TS only for forwarding to non-REMS. EN also for receiving from non-ERDS.
- **ds:Signature**: TS XAdES per TS 103 171. EN XAdES per EN 319 132-1.
- **Authentication details**: TS `SenderAuthenticationDetails` (choice between time+method or SAML Assertion). EN `AssuranceLevelsDetails/AuthenticationAssuranceLevel` mandatory, includes `AssuranceLevel` URI, policy references, and authentication details (choice including OAuth2 token). Also available for recipients and delegates.
- **RecipientAuthenticationDetails**: TS similar. EN same as for sender.
- **ReplyToAddress**: TS optional in evidence. EN dropped from evidence, kept only in metadata.

#### 5.1.6 Changes on Trusting
- **Trust establishment**: TS defines REM Policy Domain with Trust-service Status Lists (TSL) using ETSI TS 102 231; specifies three scenarios (Closed, Interoperable without Root TSL, Interoperable with Root TSL). EN references ETSI EN 319 522-2 which allows various methods (local configuration, Trusted Lists per TS 119 612, European Trusted Lists, domain PKI, metadata). EN recommends using TL; note that EU TL can be used. EN does not detail interconnection of different trust domains with pointers.
- **REMS information in Trusted Lists**:
  - Service type identifier: TS `http://uri.etsi.org/TrstSvc/Svctype/REM`. EN `http://uri.etsi.org/TrstSvc/Svctype/EDS/REM` or `.../EDS/REM/Q` for qualified.
  - Service digital identity: TS TSP X.509 certificate and optional X509SKI. EN X.509 certificate (REMS signing certificate or CA certificate used to issue REMS signing certificates), optional X509SKI.
  - Service supply point: TS provides access info for MD-RI (SMTP or web service). EN provides URI for REM RI (SMTP URI recommended if SMTP).
  - TSP service definition URI: TS points to published general info. EN may point to such info.
  - Service information extensions: Both discourage using critical extensions.

### 5.2 PReM – UPU S.52 2008 vs UPU S.52 CEN/TS 16326 (2013)

#### 5.2.1 Introduction
Gap analysis between PReM S52-1 (used for original interoperability profile) and UPU S52-2 / CEN/TS 16326, considering only parts critical for interoperability with REM.

#### 5.2.2 Changes on Flows
- **Operational verbs**: S52-1: CheckIntegrity, LogEvent, Postmark, RetrieveResults, Sign, Verify. S52-2: CheckIntegrity, Postmark, Sign, Verify (LogEvent and RetrieveResults removed; replaced by internal steps).
- **Additional server-side verbs**: S52-1: SendMessageToDestination, SubscribeNotification, UnsubscribeNotification, RejectMessage, ReceiveNotification. S52-2: SendMessageToDestination, RetrieveMessage (new), RejectMessage, ReceiveNotification.
- **Interaction**: S52-1: Web-based client interface. S52-2: Extension to existing email client SW.
- **Workflow changes**: S52-2 introduces new state diagram and operation workflows (Figure 3, 4). Key mapping differences (Table 20): Invoke SendMessageToDestination to forward PReM Dispatch; Check Parameters negative case; VerifyRequesterSignature; Store EFW-DSP-DOO, EFF-UNR-DOO; Save forwarded evidence and send notification.
- **New workflow steps** (Table 21): Conversion to PReMObject in MIME format; Sign RequesterSignature; Use SendMessageToDestination to accept Mailer's PReM Object; Mailer address format must be IETF RFC 2822; Generating evidence for acceptance/rejection, notification success/failure; Prepare PReM Message with Introduction, PReM Object, Postmark, signature; Prepare PReM Dispatch with Introduction, PReM Message, Evidence Group, Postmark, signature; Generate Evidence of Sign-PReM Dispatch; Decompose PReM Dispatch; Extract PReM Message from PReM Dispatch.
- **Interoperation with non-PReM**: Clause 5.2.4 of S52-2 must be considered.

#### 5.2.3 Changes on Messages
- Format specifications more detailed in S52-2, promoting interoperability (point #1, #5 of Table 21).
- Deep structure of PReM objects (Introduction, PReM Object, Postmark, signature) differs from REM dispatch structure; specification does not facilitate interoperability (point #2, #8, #9).
- Message identifier requirement (point 2.6 of table in 5.2.2) can be addressed at gateway with double id.
- Enveloping depth and structure of PReM message vs REM dispatch are different; recommendation to align at least one structure level (e.g., PReM Message aligned with REM dispatch, Postmark as extension) to simplify gateway.

#### 5.2.4 Changes on Evidence Structure and Semantic
- Evidence types and formats have changed; S52-2 lists all evidence (Table 1, 8.3) and defines format (Figure 8).
- Key questions for interoperability: 1) Format of evidence, 2) Format of enveloping, 3) Type of evidence, 4) Flow of evidence. Recommendation: at least evidence format should be fully aligned; other aspects can be handled at gateway.

#### 5.2.5 Changes on Signature
- S52-2 defines signature format (Figures 9, 10) applied to PReM message and dispatch.
- Key questions: 1) Format, 2) Type, 3) Objects to sign. Recommendation: format and type should be aligned; object mapping can be handled at gateway.

#### 5.2.6 Changes on Trusting
- S52-2 introduces check if DOD is in Designated Operator Trust List (point 2.7).
- Central Yellow Page repository implemented for DO service point location.
- Key questions: 1) Mapping between REM trusting/location and PReM Yellow Page; 2) Type of information in repository. Recommendation: location mechanism and information should be mutually compatible; e.g., each side can lookup on own environment using keys from other side.

## 6 Recommendations for Follow-up Activities

### 6.1 Overview
The historical interoperability profile (TS 102 640-6-1) relied on PReM referencing REM formats. Both standards have drifted. Recommendations aim to re-align and define a new REM/PReM Gateway approach.

### 6.2 UPU-side Activities

#### 6.2.1 Update of References
- **TS 102 640** → **EN 319 532**
- **TS 101 862** (certificate profile) → **EN 319 412**
- **CWA 14169** (secure signature creation device) → **CEN EN 419211**
- **Directive 1999/93/EC** → **Regulation (EU) 910/2014 (eIDAS)**
- **RFC 2822** → **RFC 5322**
- **RFC 3851** → **RFC 5751**
- **TS 101 903** (XAdES) → **EN 319 132**

#### 6.2.2 Update of Security Techniques
- Replace SSL with TLS (current v1.3, RFC 8446).
- Replace MD5/SHA1 (as recommended by S43) with SHA2/SHA3 families; refer to ETSI TS 119 312 for recommended cryptographic suites.

#### 6.2.3 Adaptation of Flows
PReM should map to ERDS concepts: user content, submission, relay, consignment, handover, notification for acceptance, notification of consignment. Not all features required (e.g., consignment or handover, optional relay/notification). Adaptation needed to identify these concepts in PReM workflow.

#### 6.2.4 Adaptation of Message Formats
- Alternative 1 (recommended): Adapt PReM message structure to align with latest REM message structure (based on clause 5.1.4) so implementations handle both alike.
- Alternative 2: Use gateway with re-enveloping – less recommended due to complexity.

#### 6.2.5 Adaptation of Evidence Format
- Strongly recommended to align PReM evidence format with ERDS evidence format (EN 319 522-3) to avoid format transformation breaking digital signatures. Clause 5.1.5 provides support for this update.

#### 6.2.6 Update of Policy Considerations
- Update references from Directive 1999/93/EC to eIDAS Regulation.
- Consider ETSI EN 319 521/531 for policy/security requirements.
- Add eID (electronic identification means per Chapter II of eIDAS) as option for identification/authentication.
- Continue to use Advanced/Qualified Electronic Signatures as redefined by eIDAS.
- Consider remote (server-based) signature and validation services.

### 6.3 ETSI-side Activities
The new interoperability profile should define a REM/PReM Gateway (as in TS 102 640-6-1) but updated. Required tasks:
- Map PReM events/evidence to ERD event types.
- Map PReM operations (verbs) to REM message sequences (SubscribeNotification, LogEvent, etc. removed; RetrieveMessage new).
- Recreate functional gap analysis considering changes in PReM workflows and REM logical model.
- Define transformation rules between message structures (depends on format alignment).
- Map protocol elements for SendMessageToDestination, ReceiveNotification.
- Redefine mutual recognition system considering changes in trust establishment (clauses 5.1.6, 5.2.6).

## 7 Conclusions
The historical REM-PReM interoperability profile relied on PReM referencing REM formats. Both standards have evolved; formats have drifted. Clause 5 analyzes gaps; Clause 6 provides recommendations for follow-up. The recommended approach is a REM/PReM Gateway that acts as REM service in one network and Designated Operator in the other, performing relay and translation. To facilitate this, UPU S52-2 should be updated to align with ETSI EN 319 532 in:
1. Format of evidence set.
2. Format of messages.
3. Format of digital signatures.
4. Deep structure of messages/enveloping.
5. Compatibility of other mechanisms (e.g., PostMark) using extensions.

If aligned, interoperability will be simpler. Otherwise, complex re-enveloping and translation will be required.
