# ETSI TS 119 102-1 V1.2.1: Electronic Signatures and Infrastructures (ESI); Procedures for Creation and Validation of AdES Digital Signatures; Part 1: Creation and Validation
**Source**: ETSI | **Version**: V1.2.1 | **Date**: 2018-08 | **Type**: Technical Specification (Normative)
**Original**: https://www.etsi.org/deliver/etsi_ts/119100_119199/11910201/01.02.01_60/ts_11910201v010201p.pdf

## Scope (Summary)
This document specifies procedures for creating and technically validating AdES digital signatures (as per ETSI EN 319 122-1, 319 132-1, 319 142-1) based on public key cryptography and certificates. It defines signature classes (Basic, With Time, With Long-Term Validation Material, Providing Long Term Availability and Integrity of Validation Material) and provides creation and validation processes. Legal interpretation, key generation, and format/encoding are out of scope.

## Definitions and Abbreviations
- **Signer's Document (SD)**: document upon which signature is generated and to which it is associated.
- **Data to be signed (DTBS)**: information objects covered by the signature (SD/SDR + signed attributes).
- **Data to be signed representation (DTBSR)**: hash of DTBSF (formatted data).
- **Signed Data Object (SDO)**: data structure containing signature value, signed attributes, optionally SD/SDR and unsigned attributes.
- **Signature class**: set of signatures achieving a given functionality (Basic, With Time, With Long-Term Validation Material, Providing Long Term Availability and Integrity of Validation Material).
- **Validation data**: PKCs, revocation status, time-assertions included for validation.
- **Best-signature-time**: earliest time proven that signature existed.
- **Driving Application (DA)**: application using creation or validation services.
- **Signature Validation Application (SVA)**: validates signature against a policy.

## 4 Signature Creation
### 4.1 Signature Creation Model
- **SCS** receives SD, composes DTBS, formats to DTBSF, produces signature over DTBSR, formats into SDO.
- **SCDev** shall hold signing certificates and corresponding creation data, authenticate signer, create signature value using signer's signature creation data.

### 4.2 Signature Creation Information Model
#### 4.2.2 Signature Creation Constraints
- Process controlled by constraints: formal policy, system config, or implicit.

#### 4.2.3 Signer's Document (SD)
- **SD** selected by signer/DA.

#### 4.2.4 Signer's Document Representation (SDR)
- Provided by DA or calculated by SCA. It shall be infeasible to find another SD with same SDR.

#### 4.2.5 Signature Attributes
- **General**: shall be signed or unsigned; set defined by policy.
- **Signing certificate identifier** (signed): shall contain reference to signing certificate, may include path references, shall include digest and algorithm.
- **Signature policy identifier** (signed): may identify creation policy, shall contain digest of policy document.
- **Signature policy store** (unsigned): shall hold policy document or URI.
- **Data content type** (signed): indicates type of SD.
- **Commitment type indication** (signed): OID or URI, selected from policy if present.
- **Counter signatures** (unsigned): contains countersignature.
- **Claimed signing time** (signed): time claimed by signer.
- **Claimed signer location** (signed): geographical location.
- **Signer's attributes** (signed): claimed attributes, attribute certificates, or signed assertions.

#### 4.2.6-4.2.9 DTBS / DTBSF / DTBSR / Signature
- **DTBS**: constructed from SD/SDR + signed attributes.
- **DTBSF**: formatted sequence.
- **DTBSR**: hash of DTBSF.
- **Signature**: SCDev applies signature algorithm to DTBSR.

#### 4.2.10 SDO
- **SDO** shall contain signature value and signed attributes; may contain SD/SDR and unsigned attributes.

#### 4.2.11 Validation Data
- **Validation data** (result of augmentation) shall include PKCs, revocation status, time-assertions.

### 4.3 Signature Classes and Creation Processes
#### 4.3.1 Introduction
- Four classes: Basic, With Time, With Long-Term Validation Material, Providing Long Term Availability and Integrity of Validation Material.

#### 4.3.2 Creation of Basic Signatures
- **Inputs**: SD/SDR (mandatory), signing certificate (mandatory), other attributes, creation policy (optional).
- **Outputs**: SDO containing signature value, reference/copy of signing certificate as signed attribute, optional attributes.
- **Process**:
  - Selection of documents to sign (DA, may involve signer).
  - Attribute/parameter selection: signing certificate identifier shall be included in DTBS when required; process guided by parameters (format, class).
  - Pre-signature presentation: DA shall present document when required; presented document shall be equal in content to signed document.
  - Signature invocation: if user consent required, DA shall follow presentation, inform signer, get consent, then invoke.
  - Signing: SCDev performs signing; SCS should check certificate validity before invocation; SCA should verify after.
  - Signer authentication: may be required; mechanism may prevent impersonation.
  - SDO composition: compose SDO according to format; return status (OK/FAILED).

#### 4.3.3 Creation of a Signature with Time
- **Process**: request time-stamp token(s) covering signature value, encapsulate as unsigned attribute.

#### 4.3.4 Creation of Signatures with Long-Term Validation Material
- **Process**: validate current signature (clause 5.5), add all validation material used that is not already present, return augmented signature with status.

#### 4.3.5 Creation of Signatures providing Long Term Availability and Integrity of Validation Material
- **Process**: optionally validate current signature, add missing validation material, request time-assertion(s) covering signer's document and all data objects in signature, encapsulate as unsigned attribute(s). Repeat as needed.

## 5 Signature Validation
### 5.1 Signature Validation Model
#### 5.1.1 General Requirements
- **SVA** validates signature against a validation policy (set of constraints), outputs status indication and report.
- **Validation always starts with the process for Signatures providing Long Term Availability and Integrity of Validation Material**, which calls lower-level processes.
- **Status indication** per building block: PASSED, FAILED, INDETERMINATE.
- **Overall status**: TOTAL-PASSED, TOTAL-FAILED, INDETERMINATE.
- **DA** should accept TOTAL-PASSED as technically valid; should not accept TOTAL-FAILED; for INDETERMINATE may retry.

#### 5.1.2 Selecting Validation Processes
- SVA shall support appropriate processes based on signature classes.
- When validating, SVA proceeds through steps: if DA does not require specific process, go step 2; if no support for long term, go to next step; etc.

#### 5.1.3 Status Indication and Report
- Outputs: status indication, policy/constraints indication, date/time of determination, validation process used.
- **Sub-indications** defined in Table 6 (e.g., FORMAT_FAILURE, HASH_FAILURE, REVOKED, EXPIRED, etc.).
- When TOTAL-PASSED or TOTAL-FAILED, same inputs shall return same result; additional POEs may change result.

#### 5.1.4 Validation Constraints
- Supports: X.509 validation constraints, cryptographic constraints, signature elements constraints.
- Constraints may come from policy, system config, or DA parameters.
- SVA shall check all constraints; policy cannot force skipping a constraint that would lead to TOTAL-FAILED.

### 5.2 Basic Building Blocks
#### 5.2.1 Description
- Building blocks: Format Checking, Identification of Signing Certificate, Validation Context Initialization, Revocation Freshness Checker, X.509 Certificate Validation, Cryptographic Verification, Signature Acceptance Validation.

#### 5.2.2 Format Checking
- Checks conformance to base format to allow cryptographic processing.
- Output: PASSED or FAILED.

#### 5.2.3 Identification of the Signing Certificate
- Identifies signing certificate using references in signed attributes.
- Output: signing certificate or INDETERMINATE/NO_SIGNING_CERTIFICATE_FOUND.

#### 5.2.4 Validation Context Initialization
- Initializes constraints and parameters from policy, signature policy reference, etc.
- Output: PASSED with constraints, or INDETERMINATE with sub-indication.

#### 5.2.5 Revocation Freshness Checker
- Checks freshness of revocation status info at given validation time.
- Output: PASSED or FAILED.

#### 5.2.6 X.509 Certificate Validation
- Validates signing certificate at validation time using chain building, PKIX path validation, revocation checking.
- **Shall** support shell model or chain model as per X.509 validation constraint.
- Output: PASSED with chain, or INDETERMINATE with sub-indication (e.g., NO_CERTIFICATE_CHAIN_FOUND, REVOKED_NO_POE, etc.).

#### 5.2.7 Cryptographic Verification
- Verifies signature integrity: check signed data items, hash matching, cryptographic signature.
- Output: PASSED, FAILED (HASH_FAILURE, SIG_CRYPTO_FAILURE), INDETERMINATE (SIGNED_DATA_NOT_FOUND).

#### 5.2.8 Signature Acceptance Validation (SAV)
- Checks additional constraints on signature attributes (e.g., signing certificate reference, claimed signing time, etc.).
- Output: PASSED or INDETERMINATE (SIG_CONSTRAINTS_FAILURE, CRYPTO_CONSTRAINTS_FAILURE_NO_POE).

#### 5.2.9 Signature Validation Presentation
- Optional block: presents data, signer info, validation time, policy, status, report.

### 5.3 Validation Process for Basic Signatures
- Steps: format checking, identification, context initialization, X.509 certificate validation, cryptographic verification, SAV.
- Handles REVOKED_NO_POE, OUT_OF_BOUNDS_NO_POE etc. by checking content-time-stamps if present.

### 5.4 Time-stamp Validation Building Block
- Validates time-stamp token as a Basic Signature; extracts generation time and message imprint.

### 5.5 Validation Process for Signatures with Time and Long-Term Validation Material
- Initializes best-signature-time from input or current time.
- Performs Basic Signature validation; if PASSED/INDETERMINATE with certain sub-indications, processes signature time-stamps.
- Compares times: if revocation after best-signature-time, may become PASSED.
- Returns PASSED, FAILED, or INDETERMINATE.

### 5.6 Validation Process for Signatures providing Long Term Availability and Integrity of Validation Material
#### 5.6.1 Introduction
- Processes additional attributes (archive time-stamps, evidence records) and POEs.

#### 5.6.2 Additional Building Blocks
- **Past certificate validation**: validates certificate at a past date using POEs and validation time sliding.
- **Validation time sliding**: slides control-time to the past when encountering revocation or cryptographic constraints failures.
- **POE extraction**: derives POEs from time-stamps (direct and indirect).
- **Past signature validation**: uses past certificate validation and POEs to resolve INDETERMINATE status.

#### 5.6.3 Validation Process for Signatures providing Long Term Availability and Integrity of Validation Material
- Processes evidence records if present.
- Performs validation for Signatures with Time (clause 5.5).
- If returned INDETERMINATE with certain sub-indications, processes time-stamps (newest first) to extract POEs.
- Runs past signature validation.
- Determines earliest POE time, runs SAV, returns final status.

## Informative Annexes (Condensed)
- **Annex A (Validation examples)**: Provides three examples illustrating the validation algorithm: revoked certificate (basic validation yields INDETERMINATE/REVOKED_NO_POE, signature with time yields TOTAL-PASSED), revoked CA certificate (basic and signature with time yield INDETERMINATE/REVOKED_CA_NO_POE, long-term validation with archive time-stamp yields TOTAL-PASSED). Uses timeline diagrams and step-by-step rationale.
- **Annex B (Signature Classes and AdES Signatures)**: Maps signature classes from this document to signature levels in CAdES, XAdES, PAdES standards (baseline and extended).
- **Annex C (Applicability rules checking and format conformance check)**: Clarifies that signature validation and applicability/conformance checking are independent. Conformance checking is not part of validation status; if required by policy, it must be implemented separately (by DA or as extension).
- **Annex D (Change History)**: Lists changes from V1.1.1 to V1.2.1, including integration of numerous change requests.
