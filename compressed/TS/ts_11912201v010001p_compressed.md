# ETSI TS 119 122-1 V1.0.1: CAdES digital signatures; Part 1: Building blocks and CAdES baseline signatures
**Source**: ETSI | **Version**: V1.0.1 | **Date**: 2015-07 | **Type**: Normative Technical Specification
**Original**: http://www.etsi.org/standards-search

## Scope (Summary)
This document specifies CAdES digital signatures built on CMS signatures (IETF RFC 5652 [7]). It defines ASN.1 definitions for signed/unsigned attributes and four levels of baseline signatures (B-B, B-T, B-LT, B-LTA) that provide incremental long-term validity. Supports electronic signatures under Regulation (EU) No 910/2014 [i.13].

## Definitions and Abbreviations
See ETSI TR 119 001 [i.3] and the following:
- **CAdES signature**: digital signature satisfying requirements in this document or TS 119 122-2 [i.6]
- **Certificate Revocation List (CRL)**: signed list indicating certificates no longer valid
- **digital signature**: data appended to, or cryptographic transformation of, a data unit to prove source and integrity
- **electronic time-stamp**: data binding other electronic data to a particular time
- **Legacy CAdES 101 733 signature**: digital signature according to ETSI TS 101 733 [1]
- **Legacy CAdES baseline signature**: digital signature according to ETSI TS 103 173 [i.1]
- **Abbreviations**: ATSv2, ATSv3 (see clauses A.2.4 and 5.5.3)

## 4 General Syntax
### 4.1 General Requirements
- CAdES signatures **shall** build on CMS (IETF RFC 5652 [7]) by incorporating signed and unsigned attributes as defined in clause 5.
- CAdES signatures **shall** comply with clauses 2, 3, 4 and 5 of IETF RFC 5652 [7].

### 4.2 The data content type
- As defined in CMS (clause 4). Identified by id-data OID.

### 4.3 The signed-data content type
- As defined in CMS (clause 5). Represents content to sign and signature values.

### 4.4 The SignedData type
- As defined in CMS (clause 5.1). CMSVersion **shall** be 1 or 3 as per RFC 5652.

### 4.5 The EncapsulatedContentInfo type
- As defined in CMS (clause 5.2). For long-term validation, either eContent **should** be present or data **should** be archived preserving encoding.

### 4.6 The SignerInfo type
- As defined in CMS (clause 5.3). Degenerate case (no signers) **shall not** be used.

### 4.7 ASN.1 Encoding
- 4.7.1 DER: As defined in ITU-T X.690 [17].
- 4.7.2 BER: As defined in ITU-T X.690 [17].

### 4.8 Other standard data structures
- **TimeStampToken**: As defined in IETF RFC 3161 [4] updated by IETF RFC 5816 [9].
- **Additional types**: VisibleString, BMPString, IA5String, GeneralizedTime, UTCTime (X.680 [16]); DirectoryString (X.520 [15]); AttributeCertificate (RFC 5755 [8]); ResponderID, OCSPResponse, BasicOCSPResponse (RFC 6960 [14]); Name, Certificate, AlgorithmIdentifier (RFC 5280 [6]); Attribute, CertificateList (RFC 5280 [6]); RevocationInfoChoices (RFC 5652 [7]).

### 4.9 Attributes
- Distinguishes signed attributes (covered by signature) and unsigned attributes (added later, may be secured by time-stamps). Stored in signedAttrs and unsignedAttrs fields of SignerInfo.

## 5 Attribute semantics and syntax
### 5.1 CMS defined basic signed attributes
- **content-type** (clause 5.1.1): Signed attribute indicating type of signed content. As defined in CMS (RFC 5652 [7], clause 11.1).
- **message-digest** (clause 5.1.2): Signed attribute specifying message digest of content. As defined in CMS (RFC 5652 [7], clause 11.2). Calculation as per clause 5.4.

### 5.2 Basic attributes for CAdES signatures
#### 5.2.1 signing-time
- Signed attribute specifying time signer claims to have performed signing. As defined in CMS (RFC 5652 [7], clause 11.3).

#### 5.2.2 Signing certificate reference attributes
- **General requirements**: Attributes **shall** contain one reference to signing certificate; **may** contain references to certificates in signing path. Each **shall** contain digest value.
- **ESS signing-certificate** (clause 5.2.2.2): Signed attribute using SHA-1 hash. As defined in IETF RFC 2634 [3], clause 5.4. policies field **shall not** be used. If issuerAndSerialNumber present, **shall** match.
- **ESS signing-certificate-v2** (clause 5.2.2.3): Signed attribute using hash algorithm different from SHA-1. As defined in IETF RFC 5035 [5], clause 4. policies field **shall not** be used. If issuerAndSerialNumber present, **shall** match.

#### 5.2.3 commitment-type-indication
- Signed attribute that qualifies the signed data. Indicates commitment made by signer. Contains exactly one CommitmentTypeIndication value. OID: id-aa-ets-commitmentType. Syntax in annex C.

#### 5.2.4 Attributes for identifying the signed data type
- **content-hints** (clause 5.2.4.1): Signed attribute. **Shall not** be used in countersignature. Provides information on innermost signed content. As defined in ESS (RFC 2634 [3], clauses 1.3.4 and 2.9).
- **mime-type** (clause 5.2.4.2): Signed attribute. **Shall not** be used in countersignature. Indicates MIME type of signed data. Exactly one MimeType value. OID: id-aa-ets-mimeType. Syntax in annex C.

#### 5.2.5 signer-location
- Signed attribute specifying address of signer. Exactly one SignerLocation value. OID: id-aa-ets-signerLocation. At least one of countryName, localityName, postalAddress **shall** be present.

#### 5.2.6 Incorporating attributes of the signer
- **signer-attributes-v2** (clause 5.2.6.1): Signed attribute encapsulating signer attributes (claimed, certified, or signed assertions). Exactly one SignerAttributeV2 value. OID: id-aa-ets-signerAttrV2. Empty attribute **shall not** be created.
- **claimed-SAML-assertion** (clause 5.2.6.2): Claimed assertion including SAML assertion. Used in claimedAttributes field of signer-attributes-v2. OID: id-aa-ets-claimedSAML. Value is OCTET STRING containing SAML assertion.

#### 5.2.7 countersignature
- Unsigned attribute including counter signature on the CAdES signature. As defined in CMS (RFC 5652 [7], clause 11.4).

#### 5.2.8 content-time-stamp
- Signed attribute encapsulating one time-stamp token of the signed data content before signing. Exactly one ContentTimestamp value. OID: id-aa-ets-contentTimestamp. messageImprint hash of raw data (without ASN.1 tag and length).

#### 5.2.9 signature-policy-identifier and SigPolicyQualifierInfo
- **signature-policy-identifier** (clause 5.2.9.1): Signed attribute containing explicit identifier of signature policy. Exactly one SignaturePolicyIdentifier value. OID: id-aa-ets-sigPolicyId. signaturePolicyImplied choice **shall not** be used.
- **SigPolicyQualifierInfo** (clause 5.2.9.2): Further qualifies signature policy. Three qualifiers: SPuri (URI), SPUserNotice (notice text), SPDocSpecification (technical spec). Syntax in annex C.

#### 5.2.10 signature-policy-store
- Unsigned attribute containing signature policy document or URI to local store. Exactly one SignaturePolicyStore value. OID: id-aa-ets-sigPolicyStore.

#### 5.2.11 content-reference
- Signed attribute linking one SignedData to another. As defined in ESS (RFC 2634 [3], clauses 1.3.4 and 2.11).

#### 5.2.12 content-identifier
- Signed attribute providing identifier for signed content. As defined in ESS (RFC 2634 [3], clauses 1.3.4 and 2.7). Should contain user-specific information, GeneralizedTime, and random number.

### 5.3 signature-time-stamp
- Unsigned attribute encapsulating time-stamp token on signature value for a specific signer. Exactly one SignatureTimeStampToken value. OID: id-aa-signatureTimeStampToken. messageImprint hash of signature field (without tag and length).

### 5.4 Attributes for validation data values
- **5.4.1 Introduction**: Specifies places to incorporate missing validation data before CAdES-A.
- **5.4.2 OCSP responses**: OCSP response **shall** be incorporated using OCSPResponse or BasicOCSPResponse type. OCSPResponse **should** be used. Within RevocationInfoChoices, OCSP responses **shall** be included in other field, **should** use encoding per RFC 5940 [13].
- **5.4.3 CRLs**: As defined in RFC 5280 [6].

### 5.5 Archive validation data
#### 5.5.1 Introduction
- Specifies archive-time-stamp (ATSv3) using ats-hash-index-v2, both unsigned. If long-term-validation attribute already present, ATSv3 cannot be used (see A.2.5).

#### 5.5.2 ats-hash-index-v2
- Unsigned attribute providing imprint of essential components for archive time-stamp. Exactly one ATSHashIndexV2 value, DER encoded. OID: id-aa-ATSHashIndex-v2. Contains hashIndAlgorithm, certificatesHashIndex, crlsHashIndex, unsignedAttrsHashIndex. Hash values for every instance present at time of archive request.

#### 5.5.3 archive-time-stamp-v3
- Unsigned attribute. Time-stamp token of signed document and signature, including signed attributes and essential components protected by ats-hash-index-v2. Exactly one ArchiveTimeStampToken value. OID: id-aa-ets-archiveTimestampV3. Input for message imprint: concatenation of eContentType, signed data hash, specified SignerInfo fields, and ATSHashIndexV2 instance. **Shall** include single ats-hash-index-v2 attribute.

## 6 CAdES baseline signatures
### 6.1 Signature levels
- **B-B**: requirements for signed and some unsigned attributes at generation.
- **B-T**: adds trusted token proving signature existed at a date/time.
- **B-LT**: adds all validation material for long-term availability.
- **B-LTA**: adds time-stamp tokens for long-term validity of validation material.

### 6.2 General requirements
- **6.2.1 Algorithm requirements**: Algorithms and key lengths **should** comply with ETSI TS 119 312 [i.8]. MD5 **shall not** be used.
- **6.2.2 Notation for requirements**: Table 1 uses notations: "shall be present", "shall not be present", "may be present", "shall be provided", "conditioned presence", "*" (not present). Cardinality: 0, 1, 0 or 1, ≥0, ≥1.
