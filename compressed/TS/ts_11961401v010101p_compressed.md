# ETSI TS 119 614-1: Testing Conformance and Interoperability of Trusted Lists – Part 1: Specifications for testing conformance of XML representation of Trusted Lists
**Source**: ETSI | **Version**: V1.1.1 | **Date**: 2016-06 | **Type**: Normative Technical Specification
**Original**: [ETSI TS 119 614-1 V1.1.1](http://www.etsi.org/standards-search)

## Scope (Summary)
Defines checks to test conformance of Trusted Lists as specified in ETSI TS 119 612. Checks are constrained to elements defined in ETSI TS 119 612, ETSI TS 101 903 (XAdES), and XMLSig. Differentiates structural requirements from XML Schema and other requirements. Agnostic to software technology or implementation strategies.

## 1 Overview
The document specifies tests in 5 categories:
1. Tests derived from XML Schema (Annex A)
2. Tests on structure not defined by XML Schema
3. Tests on values not testable by XML Schema validator
4. Tests on interrelationships (e.g., URIs pointing to elements)
5. Tests on computations (e.g., digests)

Each test assertion includes: unique ID (prefix TL_EUMS or TL_COMM or TL_NO_EUMS or LOL_EU, etc.), normative source, target, prerequisite (optional), predicate, prescription level (mandatory/preferred/permitted), and tag (EUMS Trusted List / non-EUMS / LOL).

## 5 ServiceInformation Element (Clause 5.4)

### 5.1 ServiceTypeIdentifier
Value must be one of the URIs listed in [1] Clause 5.5.1, grouped into three categories (QC-related, non-QC, and others like RA, ACA, etc.). All permitted; exact list in original document.

### 5.2 ServiceName
Apply InternationalNamesType tests.

### 5.3 ServiceDigitalIdentity

#### Common to any ServiceDigitalIdentity
- **No PKI**: Single `<tsl:DigitalId>` with `<tsl:Other>` containing valid URI (mandatory).
- **PKI**: `<tsl:DigitalId/ds:KeyInfo/ds:KeyValue>` permitted (preferred).

#### Specific to ServiceInformation (child of TSPService)
- Must contain one or more `<tsl:DigitalId/tsl:X509Certificate>` (base-64 encoded DER) – mandatory.
- If multiple X509 certificates: same public key and same subject name – mandatory.
- Organization attribute of certificates must match TSPName or be listed in TSPTradeName – mandatory.
- `<tsl:X509SubjectName>` (optional, preferred) – if present, must match subject of certificates.
- `<ds:KeyValue>` must represent public key of certificates – mandatory.
- `<tsl:X509SKI>` permitted – if present, must represent public key – mandatory.
- Same public key must not appear in multiple services unless different ServiceTypeIdentifier with different supervision systems – see specific TAs (prescription varies).

### 5.4 ServiceStatus

#### Common to any ServiceStatus
- If ServiceType is from QC list (TL_EUMS/TSL/SVCTYPE/1), status must be "granted" or "withdrawn".
- If ServiceType is from non-QC lists (SVCTYPE/2 or 3), status must be "recognisedatnationallevel" or "deprecatedatnationallevel".

#### Specific to TSPService (current status)
- Must follow permitted transitions from first history instance (see [1] Clause 5.5.1).

### 5.5 StatusStartingTime
- Must contain UTC value (mandatory).
- In TSPService, must be later than first history entry's StatusStartingTime (mandatory).

### 5.6 SchemeServiceDefinitionURI
Apply NonEmptyMultiLangURIListType tests.

### 5.7 ServiceSupplyPoints
Each `<tsl:ServiceSupplyPoint>` (attribute `type`) must be one of the ServiceTypeIdentifier values from 5.4.1 (mandatory).

### 5.8 TSPServiceDefinitionURI
- Must be present if ServiceType = `NationalRootCA-QC` (mandatory).
- Permitted otherwise.

### 5.9 ServiceInformationExtensions

#### 5.9.1 ExpiredCertsRevocationInfo
- Permitted for certain CA/QC/OCSP/CRL service types (listed).
- Critical attribute must be "false" (if present).

#### 5.9.2 sie:Qualifications
- For CA/QC service type: each `<sie:Qualifier>` must be one of the 13 specific URIs (e.g., QCWithSSCD, QCNoSSCD, etc.) – permitted values.
- For non-CA/QC: none of those same URIs are allowed – mandatory.
- For `<sie:PolicyIdentifier>`: `Qualifier` attribute must be "OIDAsURN" and value must be OID encoded as URN per RFC 3061 – mandatory.
- For `<tslx:CertSubjectDNAttribute>`: same OIDAsURN requirement.

#### 5.9.3 tslx:TakenOverBy
- URI: scheme+authority must match scheme+authority of taking-over TSP's electronic address, TSP info URI, or Scheme info URI (preferred).
- TSPName: English name must match a TSP in the TL (mandatory).
- SchemeOperatorName: English name must match SchemeOperatorName in SchemeInformation (mandatory).
- SchemeTerritory: must equal SchemeTerritory in SchemeInformation (mandatory).

#### 5.9.4 AdditionalServiceInformation
- URI must be one of four specific values (ForSignatures, ForSeals, ForWebSiteAuthentication, RootCA-QC) – permitted.
- If service type is not CA/QC, URI must not be RootCA-QC – mandatory.
- May also be a registered URI further qualifying a QC component service or nationally defined qualification – permitted.

## Annex A: Test Assertions Derived from XML Schema (Normative)

This annex specifies structural tests for XML types defined in ETSI TS 119 612 and its schemas. All TAs in Annex A have `TL_SCH/` prefix. They are not repeated in full here; key points:

- Every type must have non-empty values where applicable.
- Attributes are mandatory as per schema (e.g., `lang` for multilanguage types, `Critical` for extensions).
- Structure (order and cardinality) must exactly match schema definitions.
- Specific types covered: NonEmptyNormalizedString, NonEmptyString, MultiLangStringType, MultiLangNormStringType, InternationalNamesType, NonEmptyURIType, NonEmptyURIListType, NonEmptyMultiLangURIType, NonEmptyMultiLangURIListType, AttributedNonEmptyURIType, AddressType (postal and electronic), ExtensionType, ExtensionsListType, TrustServiceStatusList, SchemeInformation, PolicyOrLegalNoticeType, OtherTSLPointersType, ServiceDigitalIdentityListType, DigitalIdentityListType, DigitalIdentityType, AdditionalInformationType, TrustServiceProviderList, TSPType, TSPInformationType, TSPServicesListType, TSPServiceType, TSPServiceInformationType, ServiceSupplyPointsType, ServiceHistoryType, ServiceHistoryInstanceType, AdditionalServiceInformationType, sie:QualificationsType, sie:QualificationElement, sie:Qualifiers, sie:CriteriaListType, tslx:ExtendedKeyUsageType, tslx:TakenOverByType, tslx:CertSubjectDNAttribute, tslx:MimeType, and xades:ObjectIdentifierType.

**Example structural TA:**
- `TL_SCH/TL/1`: Inner structure of `tsl:TrustServiceStatusList` = `tsl:SchemeInformation, tsl:TrustServiceProviderList?, ds:Signature?` (mandatory).
- `TL_SCH/SCHI/1`: Inner structure of `tsl:SchemeInformation` = defined sequence with optional elements (mandatory).
