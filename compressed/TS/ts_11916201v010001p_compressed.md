# ETSI TS 119 162-1 V1.0.1: Electronic Signatures and Infrastructures (ESI); Associated Signature Containers (ASiC); Part 1: Building blocks and ASiC baseline containers
**Source**: ETSI | **Version**: V1.0.1 | **Date**: 2015-08 | **Type**: Normative (Technical Specification)
**Original**: (not provided)

## Scope (Summary)
This document specifies Associated Signature Containers (ASiC) based on ZIP [5] that bind detached digital signatures or time-assertions with file objects. It defines building blocks and a limited set of baseline containers (ASiC-S and ASiC-E) supporting CAdES, XAdES, time-stamp tokens (RFC 3161/RFC 5816), and evidence records (RFC 4998/RFC 6283). Four levels of ASiC baseline containers address incremental long-term validity requirements.

## Definitions and Abbreviations
- **ASiCArchiveManifest file**: container file matching "*ASiCArchiveManifest*.xml" with one ASiCManifest element per clause A.7
- **ASiCEvidenceRecordManifest file**: file in ASiC-E referencing a set of files to which an evidence record applies; name matches "META-INF/ASiCEvidenceRecordManifest*.xml", conformant to clause A.4
- **ASiCManifest file**: file matching "*ASiCManifest*.xml" with one ASiCManifest element per clause A.4
- **Container**: ZIP file holding files, manifest, metadata, and associated signature(s) under a folder hierarchy
- **Media type**: method to label arbitrary content (per IETF RFC 6838)
- **Metadata**: data describing context, content, and structure of data objects and their management over time (per ISO 15489-1)
- **Time assertion**: time-stamp token or evidence record
- **Abbreviations**: ASiC (Associated Signature Container), ER (Evidence Record), OCF (Open Container Format), ODF (Open Document Format), TST (Time Stamp Token), ZIP (format specified in [5])

## 4 General Syntax
### 4.1 Main Features
- ASiC is a data container using ZIP [5] format, holding file objects and associated digital signatures/time assertions.
- Internal structure includes a root folder (for content) and a `META-INF` folder (for metadata, signatures, time assertions).
- **Container types**: ASiC Simple (ASiC-S) associates one file object with one signature file or one time assertion file; may include `mimetype` file and later add signatures/ASiCArchiveManifests. ASiC Extended (ASiC-E) associates multiple file objects with XAdES or CAdES signatures and/or time assertions; allows adding signatures, time assertions, data without invalidating previous.

### 4.2 General Requirements
1. **R1**: Container format **shall** comply with ZIP [5].
2. **R2**: ZIP limitations:
   - a) Multiple volumes split **shall not** be used.
   - b) File names and comments **shall** be UNICODE UTF-8.
   - c) Only no compression (0) or Flate (8) **should** be used (as per [5] and [10]).
3. **R3**: At least one container type (clauses 4.3 or 4.4) **shall** be supported.

### 4.3 ASiC Simple (ASiC-S)
- Three types: with XAdES, with CAdES, with time assertions.
- **General requirements**: comply with clause 4.2 and file structure of clause 4.3.3.2.

#### 4.3.3 Detailed format for ASiC-S
##### 4.3.3.1 Media type identification
- If `mimetype` file present, media type **shall** be either `application/vnd.etsi.asic-s+zip` (if file extension is .asics or .scs, or no specific media type) or the media type of the signed file object.
- File extension **shall** be `.asics`, `.scs` (3‑char limit), or `.zip` (if manual handling).
- Archive comment may contain `mimetype=` followed by original media type.

##### 4.3.3.2 Contents of the container
- **R4**: `mimetype` file may be present per clause A.1.
- **R5**: **Shall** contain one signed data file at root level (only file object besides optional `mimetype`).
- **R6**: **Shall** contain one `META-INF` folder at root.
- **R7**: `META-INF` folder **shall** contain one of:
   - a) `timestamp.tst` (RFC 3161 time-stamp token)
   - b) `signature.p7s` (detached CAdES per [1] or [11])
   - c) `signatures.xml` (root element `asic:XAdESSignatures` per clause A.5 with one or more XAdES signatures per [2] or [12]; reference rules per clause A.6)
   - d) `evidencerecord.ers` (ERS format)
   - e) `evidencerecord.xml` (XMLERS format)
- **R8**: `META-INF` may contain additional files: ASiCArchiveManifest files and related time-stamp tokens; revocation/certificate info; other application-specific info.

#### 4.3.4 Long term validity of ASiC-S
- For XAdES/CAdES: mechanisms from [1], [2], [11], [12] **shall** be used.
- For time-stamp token: add ASiCArchiveManifest files and time-stamp tokens per clause A.7.
- For ER: internal mechanisms of [8] or [9] **shall** be used.

### 4.4 ASiC Extended (ASiC-E)
- Two types: with XAdES (may include ERS) and with CAdES – time assertions (signatures and/or time assertions).
- Allows nesting of containers.

#### 4.4.2 General Requirements
- **R9**: ASiC-E **shall** use ZIP format per clause 4.2 with file structures from clauses 4.4.3.2 or 4.4.4.2.
- **R10**: One or more digitally signed files **shall** be present outside `META-INF`.

#### 4.4.3 ASiC-E with XAdES
##### 4.4.3.1 Media type identification
- File extension **shall** be `.asice` or `.sce`.
- `mimetype` content **shall** be `application/vnd.etsi.asic-e+zip` or the original media type of the container.
- Archive comment may contain `mimetype=application/vnd.etsi.asic-e+zip`.

##### 4.4.3.2 Contents of Container
- **R11**: `mimetype` file may be present (clause A.1, content per 4.4.3.1 item 2).
- **R12**: One or more `*signatures*.xml` files **shall** be in `META-INF/`, each containing one or more XAdES signatures per [2] or [12]; signed data **shall** be directly referenced via `ds:Reference` or indirectly via signed `ds:Manifest` (subject to clause 4.4.3.4).
- **R13**: Root element of each signature file **shall** be either: `asic:XAdESSignatures`, `document-signatures` (ODF), `signatures` (OCF), any element in any namespace whose valid content is a sequence of `ds:Signature` siblings, or `ds:Signature` directly. Item a) **should** be used.
- **R14**: One or more `ASiCEvidenceRecordManifest` files may be present; each references an ER file in `META-INF` named `evidencerecord.ers` or `.xml`.
- **R15**: Other files may be present: `container.xml` (per OCF), `manifest.xml` (per ODF), `metadata.xml`.

##### 4.4.3.4 XAdES use in ASiC-E with XAdES
- Rules from clause A.6 apply. `ds:Reference` should be preferred over `ds:Manifest`.
- **If `ds:Manifest` used**: (1) **shall** be signed (referenced in `ds:SignedInfo`); (2) **shall not** reference other `ds:Manifest` elements; (3) `Id` attribute should be used; (4) validation application **shall** raise a warning on digest mismatch even if signature value match.

#### 4.4.4 ASiC-E with CAdES – time assertions
##### 4.4.4.1 Media type identification
- File extension **shall** be `.asice` or `.sce`.
- `mimetype` content **shall** be `application/vnd.etsi.asic-e+zip`.
- Comment may contain `mimetype=application/vnd.etsi.asic-e+zip`.

##### 4.4.4.2 Contents of Container
- **R16**: `mimetype` file may be present (clause A.1).
- **R17**: One or more `ASiCManifest` and/or `ASiCEvidenceRecordManifest` files **shall** be present.
- **R18**: For each `ASiCManifest`, one time-stamp token (`*timestamp*.tst`) or one signature file (`*signature*.p7s`) **shall** be present in `META-INF`.
- **R19**: For each `ASiCEvidenceRecordManifest`, one ER file (`*evidencerecord*.ers` or `.xml`) **shall** be present.
- **Verification rules**: verifiers **shall** validate CAdES signature/time-stamp against the `ASiCManifest` content; for `ASiCEvidenceRecordManifest`, validate ER against `DataObjectReference` digests; error on mismatch.

#### 4.4.5 Long term validity of ASiC-E
- For XAdES: mechanisms from [2], [12] or evidence records [8], [9] **shall** be used.
- For CAdES – time assertions: either add `ASiCArchiveManifest` files per clause A.7, or `ASiCEvidenceRecordManifest` files covering all relevant data.
- For ER: internal mechanisms of [8] or [9] **shall** be used.

## 5 ASiC Baseline containers
### 5.1 ASiC Levels
- Four levels: B-B, B-T, B-LT, B-LTA.
- All levels incorporate requirements from clause 4 plus level-specific signature profiles.
- **B-B**: signatures complying with B-B level of [1] or [2].
- **B-T**: signatures with trusted token (date/time).
- **B-LT**: signatures include all validation material.
- **B-LTA**: signatures with additional time-stamp tokens for long-term validation.
- When multiple signatures with different levels, container level **shall** be the lowest level signature.

### 5.2 General requirements
#### 5.2.1 Algorithm requirements
- Algorithms and key lengths **should** comply with ETSI TS 119 312 [i.4].
- MD5 **shall not** be used.
- For container signatures, clauses 6.2.1 of [1] and [2] **shall** apply.

#### 5.2.2 Notation for requirements
- Tables specify presence, cardinality, references, additional notes.

### 5.3 Requirements for ASiC baseline containers
#### 5.3.1 ASiC conformance (Table 1)
- **Container format**: ZIP **shall** be supported.
- **ZIP limitations**: encryption **shall not** be used; compression method **shall** be only 0 or 8 (no compression or Flate).

#### 5.3.2 Requirements for ASiC-S
##### 5.3.2.1 General requirements (Table 2)
- File extension `.asics` **shall** be present.
- `mimetype` may be present.
- Signed file **shall** be present (one, outside `META-INF`).

##### 5.3.2.2 ASiC-S with CAdES (Table 3)
- `META-INF/signature.p7s` **shall** be present.
- CAdES signature as per [1] clause 6 per required level.
- No other elements besides `mimetype`, signed file, and signature file.

##### 5.3.2.3 ASiC-S with XAdES (Table 4)
- `META-INF/signatures.xml` **shall** be present.
- `asic:XAdESSignatures` **shall** be present.
- Each XAdES signature **shall** reference signed file explicitly via `ds:Reference`.
- No other elements besides `mimetype`, signed file, and signature file.

#### 5.3.3 Requirements for ASiC-E with XAdES (Table 5)
- File extension `.asice` **shall** be present.
- `mimetype` may be present.
- At least one signed file object **shall** be present outside `META-INF`.
- At least one ASiC-E XAdES signature **shall** be present (in `META-INF`).
- `asic:XAdESSignatures` **shall** be present per signature file.
- `META-INF/manifest.xml` **shall** be present.
- Each XAdES signature **shall** reference all signed file objects directly via `ds:Reference`.
- `META-INF` folder **shall** contain only files specified in this clause.

## Annex A (normative): ASiC metadata specification, data naming and referencing
- **A.1 mimetype file**: first file in ZIP, no extra fields, no compression, first 4 bytes `50 4B 03 04`. Not compressed/encrypted.
- **A.2 Media type registrations**: tables for `vnd.etsi.asic-s+zip`, `vnd.etsi.asic-e+zip`, `vnd.etsi.timestamp-token`.
- **A.3 ASiC XML Schema**: namespace `http://uri.etsi.org/02918/v1.2.1#`. Schema file attached; digest SHA-256: `38:37:e6:95:db:f0:d5:3d:38:e0:3e:75:26:02:8f:5d:8e:3b:30:66:c8:35:56:b0:ba:1b:df:74:af:ca:d2:01`.
- **A.4 ASiCManifest element**: references one signature/time assertion file via `SigReference` and one or more data files via `DataObjectReference` (with digest, mime type, optional extensions). Syntax provided.
- **A.5 XAdESSignatures element**: root element containing one or more `ds:Signature` children.
- **A.6 Naming and referencing data**: relative URIs **shall** be used; references in `META-INF` relative to root directory; references outside container **shall not** be allowed.
- **A.7 ASiCArchiveManifest file**: used for long-term validity. Rules for adding first and subsequent manifests: naming, referencing all signed/time-stamped data, signatures, previous manifests, and time-stamp tokens. Latest manifest always named `ASiCArchiveManifest.xml`.
