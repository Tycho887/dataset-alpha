# ETSI TS 102 778-6 V1.1.1 (2010-07): Electronic Signatures and Infrastructures (ESI); PDF Advanced Electronic Signature Profiles; Part 6: Visual Representations of Electronic Signatures
**Source**: ETSI | **Version**: V1.1.1 | **Date**: July 2010 | **Type**: Normative Technical Specification
**Original**: DTS/ESI-000085-6

## Scope (Summary)
The present document specifies requirements and recommendations for the visual representations of advanced electronic signatures (AdES) in PDFs. It covers (a) **signature appearance** – the visual representation of the human act of signing placed within a PDF document at signing time and linked to an AdES, and (b) **signature verification representation** – the visual representation of the verification of an AdES. The goal is to aid untrained human understanding and consistency between the two representations.

## Definitions and Abbreviations
### Definitions
- **certificate image**: image that is part of an X509 certificate as specified in draft-ietf-pkix-certimage-04 [5]
- **certified identity**: information about the signer certified by a trusted source
- **claimed signing time**: time of signing claimed by the signer which on its own does not provide independent evidence of the signing time
- **conforming signature handler**: software application, or part of a software application, that knows how to perform digital signature operations (e.g. signing and/or verifying) in conformance with ISO 32000-1 [1] and the requirements of the appropriate profile
- **off page display**: information displayed by a conforming signature handler that is unmistakably separated from the page content of a PDF document
- **PDF Signature**: binary data object based on the CMS (RFC 3852 [9]) or related syntax containing a digital signature placed within a PDF document structure as specified in ISO 32000-1 [1] clause 12.8 with other information about the signature applied when it was first created
- **signature dictionary**: PDF data structure, of type dictionary, as described in ISO 32000-1 [1], clause 12.8.1, Table 252 that contains all the information about the Digital Signature
- **signer**: entity that creates an electronic signature
- **validation data**: data that may be used by a verifier of electronic signatures to determine that the signature is valid (e.g. certificates, CRLs, OCSP responses)
- **verifier**: entity that validates an electronic signature
- **signature appearance**: visual representation of the human act of signing placed within a PDF document at signing time and linked to an advanced electronic signature
- **signature verification representation**: visual representation of the verification of an advanced electronic signature
- **may**: means that a course of action is permissible within a profile
- **shall**: means that the definition is an absolute requirement of a profile (must be strictly followed to conform)
- **should**: means that among several possibilities one is recommended, in a profile, as particularly suitable, without mentioning or excluding others, or that a certain course of action is preferred but not necessarily required

## General Concepts (Condensed)
An advanced electronic signature (AdES) in a PDF is based on a cryptographic algorithm protecting integrity and signer identity. A **signature appearance** is a human-readable visual representation (via the AP key in a widget annotation dictionary) linked to an AdES, sealed by the digital signature at signing time. It may include signer identity (textual name, graphical image) and other attributes (time, reason, location) – the time is a claimed signing time. Identification in the signature appearance is not directly verifiable by the AdES but can be visually compared to the **signature verification representation**, which is an off-page report recreated at validation time from the AdES, containing authenticated information (e.g., certificate identity, validation results, time-stamps). The report should be hierarchical: top level for overall result and signer common name, lower levels for detailed validation data, organized similarly to DSS-X verification reports [3]. The present document provides requirements for both representations and their relationship to the AdES, aiming for consistency to aid human comparison.

## 5 The Signature Appearance
### 5.1 Recommended Information in Signature Appearances
- A conforming signature handler **should** not use information identifying the signer (e.g., name, graphics) that is not derived from the certificate when creating the signature appearance.
- The signature appearance **may** include additional contextual information not included in the certificate (e.g., signing time, location).
- A conforming signature handler **should** include the certified identity of the signatory and the claimed signing time into the signature appearance.
  - If the signatory’s certificate contains a certificate image, that image (draft-ietf-pkix-certimage-04 [5]) **should** be included.
  - If no certificate image, the following **should** be included in order:
    a) Name of signatory (as in CN)
    b) Affiliation of signatory, if relevant (as in O)
    c) Any logo image in the signatory’s certificate (RFC 3709 [4])
    d) Any image of a handwritten signature in certificate (RFC 3739 [6], section 3.2.5)
- A signature appearance **may** contain information aiding the untrained user to validate the signature, e.g., that the signature appearance is not a source of trust.

### 5.2 Encoding of Signature Appearances in PDF
- A signature field’s field type (FT) **shall** be `Sig`.
- The field value (V) **shall** be a signature dictionary containing the signature and specifying attributes as defined in TS 102 778 (PAdES) [2].
- The annotation rectangle (Rect) in the widget annotation dictionary **shall** give the position of the field on its page.
- Signatures complying with this document **shall** specify a rectangle with height and width greater than zero.
- To ensure visibility, the signature field **shall not** have the Hidden bit or the NoView bit of the F entry set to true.
- The conforming signature handler **shall** ensure that all text fits into the annotation rectangle.
- To ensure compliance with PDF/A (ISO 19005 [7]), all signature fields **shall** have an appearance dictionary (AP).
- When constructing the appearance, a single content stream **shall** be used, though it may refer to external objects (Images, Form XObjects) as necessary.
- It is **strongly recommended** that fonts used for any text in the content stream be embedded.

## 6 The Visual Representation of AdES Signature Verification
### General Principles
- A conforming signature handler **shall** display the result of AdES signature verification and the content of signed attributes in a context clearly separated from the page display (**off-page display**). The handler **shall** ensure that such a separated display cannot be faked by active document content.
- A conforming signature handler **shall not** display the result of the signature validation inside the page content.
- The visual representation **should** be displayed in a hierarchical manner, initially showing basic information in a simple form but enabling the user to obtain further details.
- The outline **should** contain a visual representation of the verification result of all AdES signatures and document timestamps that conform to PAdES [2].
- A conforming signature handler **shall** enable the user to display the visual representations of each and every individual AdES signature and document timestamps in a document (e.g., in a list), displayed such that it can be easily related to the signature appearances.

### 6.1 The Visual Representation of Individual AdES Signature Verification
For any AdES signature, a conforming signature handler **shall** at least enable the user to display the following at the top level:
a) Status of the signature: valid, invalid, or indeterminate.
b) The reasons of an invalid or indeterminate result.
c) The document revision that the signature applies to, and give the opportunity to display the PDF revision as applicable at the time of signing.
d) The signing time and an indication of its potential trustworthiness (e.g., claimed by signer or from trusted signature time-stamp).
e) Summary information of certified identity (see clause 6.2) that can be expanded in the next level.
- A conforming signature handler **may** display further information at the top level, avoiding redundancy with other clauses.

### 6.2 Summary Representation of Certified Identity Verification
- If the certified identity has been successfully validated, the following **shall** also be displayed at top level:
  - If the signatory’s certificate contains a certificate image, that image (draft-ietf-pkix-certimage-04 [5]).
  - If no certificate image, the following **should** be included in order:
    a) Name of signatory (as in CN)
    b) Affiliation of signatory (as in O)
    c) Any logo image in the signatory’s certificate (RFC 3709 [4])
    d) Any image of a handwritten signature in certificate (RFC 3739 [6], section 3.2.5)
    e) The identity of the trusted CA used as basis for certificate path validation.
- The visual representation of certified identity validation **should** be easily relatable to the signature appearance layout recommended in clause 5.1.
- Additionally, the following **shall** also be displayed:
  - Whether the certificate is valid at the time of signing (see clause 6.1 item d)).

### 6.3 Detailed Representation of AdES Signature Verification
- The user **should** be able to display all elements of the signer and all CA certificates used in the validation path.
- The signature details **shall** contain the following if available in the verification result:
  a) The algorithm validity (as in section 3.5.2 of [3])
  b) The certificate path validity (as in section 3.5.3 of [3]), with further levels of detail for all data used to determine path validity, following the same rules for visual representation.
  c) Any signed properties included from the AdES signature (attributes defined in clauses 4.4 and 4.5 of TS 102 778-3 [10] and information from keys M, Location, Reason, ContactInfo in the signature dictionary of ISO 32000-1 [1]).
  d) The signature value.
  e) If the AdES is a PAdES-EPES (Part 3 [10]), the value of the signature-policy-identifier attribute and the commitment-type attribute if present.
- A conforming signature handler **may** allow creation and print of a separate document containing the visual representation as a verification protocol. Verification results **shall not** print with the verified document.

### 6.4 The Visual Representation of the Document Timestamp Verification
- Visual representation of document time-stamp verification **should** be similar to individual AdES signature verification where applicable.
- It **should** contain at least the following at top level:
  a) The result of the verification of the timestamp.
  b) The time included in the timestamp.
  c) At least the common name and organisation of the server providing the timestamp.
