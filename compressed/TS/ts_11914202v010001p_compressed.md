# ETSI TS 119 142-2 V1.0.1: Electronic Signatures and Infrastructures (ESI); PAdES digital signatures; Part 2: Additional PAdES signatures profiles
**Source**: ETSI | **Version**: V1.0.1 | **Date**: 2015-07 | **Type**: Normative
**Original**: Provided text (chunk 1 of 1)

## Scope (Summary)
This document defines multiple profiles for PAdES digital signatures embedded within a PDF file. It includes a profile for CMS-based PDF signatures (interoperability beyond ISO 32000-1), extended PAdES signature levels (BES, EPES, LTV) that build on Part 1, and profiles for using XAdES signatures on XML content embedded in PDF (including signed XML documents and XFA forms). It does not repeat base requirements of referenced standards but aims to maximize interoperability.

## Definitions and Abbreviations
- **certificate**: public key of a user, together with some other information, rendered un-forgeable by encipherment with the private key of the certification authority which issued it
- **certificate policy (CP)**: named set of rules that indicates the applicability of a certificate to a particular community and/or class of application with common security requirements
- **Certificate Revocation List (CRL)**: signed list indicating a set of public key certificates that are no longer considered valid by the certificate issuer
- **Certification Authority (CA)**: authority trusted by one or more users to create and assign public key certificates; optionally, the certification authority may create the users' keys
- **certification signature**: digital signature used in conjunction with Modification Detection Permissions (MDP) as defined by ISO 32000-1, clause 12.8.2.2
- **electronic time-stamp**: data object that binds a representation of a datum to a particular time, thus establishing evidence that the datum existed at that time (NOTE: in RFC 3161, referring to timeStampToken field within TimeStampResp)
- **PAdES signature**: digital signature that satisfies the requirements specified within this document
- **PDF serial signature**: specific digital signature where the second (and subsequent) signers of a PDF not only sign the document but also the signature of the previous signer and any modification (e.g. form fill-in)
- **PDF signature**: DER-encoded binary data object based on PKCS #7 or CMS (RFC 5652) or related syntax containing a digital signature and other information necessary to validate the electronic signature, placed within a PDF document structure as specified in ISO 32000-1, clause 12.8
- **relying party**: natural or legal person that relies upon electronic identification or trust service
- **seed value dictionary**: PDF data structure, of type dictionary, as described in ISO 32000-1, clause 12.7.4.5, table 234, containing information constraining the properties of a digital signature applied to a specific signature field
- **signature dictionary**: PDF data structure, of type dictionary, as described in ISO 32000-1, clause 12.8.1, table 252, containing all information about the digital signature
- **signature handler**: software application, or part of a software application, that knows how to perform digital signature operations (e.g. signing and/or validating) in conformance with ISO 32000-1 and the requirements of the appropriate profile
- **signer**: natural or legal person who creates a digital signature
- **Time-Stamping Authority (TSA)**: trusted third party that creates electronic time-stamps in order to indicate that a datum existed at a particular point in time
- **validation data**: data that can be used by a verifier of digital signatures to determine that a digital signature is valid (e.g. certificates, CRLs, OCSP responses)

## 4 Profile for CMS digital signatures in PDF
### 4.1 Features
- Encoded in CMS as per PKCS #7 1.5 (RFC 2315)
- Support serial signatures
- Optionally include signature time-stamps, revocation information
- Protect integrity and authenticate signer identity
- May include reasons, location, contact info
- Legal content attestation may be used to indicate PDF capabilities affecting signed document (e.g. JavaScript)

### 4.2 Requirements of Profile for CMS Signatures in PDF
#### 4.2.1 Requirements on PDF signatures
- **R1**: PDF signatures **shall** be as specified in ISO 32000-1, clause 12.8.
- **R2**: The signature information **shall** be embedded into the document itself and the ByteRange **shall** be the entire file, including the signature dictionary but excluding the PDF Signature itself.
- **R3**: The PDF Signature (DER-encoded PKCS#7) **shall** be placed into the Contents entry of the signature dictionary.
- **R4**: The PKCS#7 object **shall** conform to RFC 2315. At minimum, it **shall** include the signer's X.509 signing certificate. (NOTE: PKCS#1 not supported by this profile.)
- **R5**: Timestamping and revocation information **should** be included in the PDF Signature. This revocation info and as much of the complete chain of certificates as available **should** be captured and validated before completing creation.
- **R6**: If present, any revocation information **shall** be a signed attribute of the PDF Signature.
- **R7**: RFC 5755 attribute certificates associated with the signer certificate **should not** be used. (NOTE: reduces interoperability.)
- **R8**: There **shall** only be a single signer (one “SignerInfo” component) in any PDF Signature.

#### 4.2.2 Requirements on PDF signature handlers
- **R9**: A PDF reader **may** substitute a different signature handler than specified in Filter, as long as it supports the specified SubFilter format.
- **R10**: Only the two SubFilter values from ISO 32000-1, clause 12.8.3.3.1 (adbe.pkcs7.detached and adbe.pkcs7.sha1) **shall** be used. (NOTE: SHA-1 being phased out; other hashing algorithms should be used.)

#### 4.2.3 Requirements on signature validation
- When a user requests validation, a reader **shall** invoke the appropriate signature handler that **shall**:
  - **R11**: Validate that the document digest matches that in the signature as per ISO 32000-1, clause 12.8.1.
  - **R12**: Validate the path of certificates as per RFC 5280 at the time indicated by electronic time-stamp (clause 4.2.4) or other trusted signing time. Revocation status **shall** be checked as per clause 4.2.5.
  - **R13**: The signing certificate reference attribute itself **should** be ignored during validation if present. (NOTE: certificate hash not required by this profile; applications needing it can use PAdES baseline profiles or clauses 5.3/5.4.)

#### 4.2.4 Requirements on Time Stamping
##### 4.2.4.1 Creation
- **R14**: An electronic time-stamp from a trusted TSA **should** be applied to the digital signature as soon as possible after signing.
- **R15**: If embedded, it **shall** be embedded as described in ISO 32000-1, clause 12.8.3.3.1.

##### 4.2.4.2 Validation
- **R16**: A signature handler **shall** take the signature field of the PKCS#7 signature, encode it and compute the digest using the algorithm indicated in the electronic time-stamp.
- **R17**: A signature handler **shall** check if the computed digest is the same as the digest present in the electronic time-stamp.

#### 4.2.5 Requirements on revocation checking
- **R18**: When validating, a signature handler **may** ignore any embedded revocation information in favour of alternative storage or referenced data as per its own policies.

#### 4.2.6 Requirements on Seed Values
- **R19**: Seed values that would require a signature handler to violate this profile **shall not** be used (e.g., specifying PKCS#1 is not permitted).

#### 4.2.7 Requirements on encryption
- **R20**: The requirements in PAdES Part 1 [4], clause 5.5 **shall** apply.

## 5 Extended PAdES signature profiles
### 5.1 Features
- Define three levels of PAdES extended signatures: BES, EPES, LTV.
- PAdES-E-BES: basic signatures, unambiguous connection to signer certificate.
- PAdES-E-EPES: built on BES, includes signature policies.
- PAdES-E-LTV: built on BES or EPES, maintains long-term validity.

### 5.2 General Requirements
#### 5.2.1 Requirements from Part 1
- **R21**: Requirements from clause 4.1 of [4] (PAdES Part 1) **shall** apply to all profiles in this clause.

### 5.5 PAdES-E-LTV Level
- **R22**: Signature handlers creating/validating PAdES-LTV **shall** support PDF documents with:
  - Document security store information as per clause 5.4.2 of [4].
  - Document time-stamps as per clause 5.4.3 of [4].
- **R23**: Signed PDF documents **should** contain DSS followed by a document time-stamp.
- **R24**: Validation data **shall** be carried by values within the DSS. (NOTE: external references not supported.)
- **R25**: Systems **shall** support creation/validation of signatures with one or more DSS entries and document time-stamps. (NOTE: Object IDs should not be reused when adding new validation data.)

## 6 Profiles for XAdES Signatures signing XML content in PDF
### 6.1 Features
- Allow signing of XML Data embedded in PDF.
- Two sets of profiles: one for signed XML documents (basic and long-term) and one for XFA forms (basic and long-term).

### 6.2 Profiles for XAdES signatures of signed XML documents embedded in PDF containers
#### 6.2.1 Overview
- Defines profile for arbitrary XML document signed with XAdES, embedded in PDF for integrity, authentication, non-repudiation.
- Basic profile: levels XAdES-E-BES, -EPES, -T.
- Long-term profile: levels XAdES-E-C, -X, -XL, -A.
- Scenario: XML document signed outside PDF, then embedded; verification extracts and augments.

#### 6.2.2 Profile for Basic XAdES signatures of XML documents embedded in PDF containers
##### 6.2.2.1 Features
- Signed XML document created independent of PDF container.
- Supports enveloped, enveloping, or detached references.
- Profiles: XAdES-E-BES, -EPES, -T.
- Supports serial and parallel signatures.

##### 6.2.2.2 General syntax and requirements
- **R26**: Signatures **shall** be XAdES signatures (BES, EPES, or T) with syntax from [6] and restrictions in this document.
- **R27**: Unsigned properties not found in this profile **may** be present and **may** be ignored unless used with other profiles.

##### 6.2.2.3 Requirements for applications generating signed XML document to be embedded
- **R28**: The signed XML document **shall** be created independently of the final PDF container.
- **R29**: The signed XML document **shall** contain at least one XAdES signature and one or more signed data objects.
- **R30**: Signed data objects **shall** satisfy one of:
  1) All embedded within the signed XML document; or
  2) If detached, a `ds:Reference` element **shall** reference it according to W3C XML-Signature 1.1.

##### 6.2.2.4 Mandatory operations
###### 6.2.2.4.1 Protecting the signing certificate
- **R31**: The signing certificate **shall** be referenced in the `SigningCertificate` element or included in `ds:KeyInfo` and covered by the signature. (NOTE: recommend using `SigningCertificate` qualifying property.)

##### 6.2.2.5 Requirements on XAdES optional properties
- Table 3 lists allowed properties; if present, syntax/usage shall be as in [6].

**(Table 3 condensed: allowed properties include SigningTime, SignaturePolicyIdentifier, SignatureProductionPlaceV2, SignerRoleV2, DataObjectFormat, CommitmentTypeIndication, AllDataObjectsTimeStamp, IndividualDataObjectsTimeStamp, SignatureTimeStamp, SignaturePolicyStore; all referenced to [6] clauses.)**

##### 6.2.2.6 Serial Signatures
- **R32**: Supports serial signatures by mechanisms in [6] clause 5.2.7.
- **R33**: `CounterSignature` element is optional; if present, syntax/usage as per [6] clause 5.2.7.2.
- **R34**: Detached XAdES may countersign; countersignature should also be present within the XML content.

##### 6.2.2.7 Parallel Signatures
- **R35**: A signed XML document may include several XAdES signatures signing in parallel the same data objects.

##### 6.2.2.8 PAdES Signatures
- **R36**: CMS digital signatures (clause 4) or PAdES-E-BES/EPES (optionally augmented via LTV) can be applied to a document with XAdES signature. (NOTE: XAdES cannot be augmented after a PDF signature is applied.)

#### 6.2.3 Profile for long-term XAdES signatures of signed XML documents embedded in PDF containers
##### 6.2.3.1 Features
- Long-term signatures production.
- Levels: XAdES-E-C, -X, -XL, -A.

##### 6.2.3.2 Augmentation mechanism
- **R37**: Conforming readers **shall** detach the signed XML document, add suitable unsigned XAdES properties to upgrade the signature, then embed again.

##### 6.2.3.3 Optional properties
- No additional requirements; if present, syntax/usage as per [6].

##### 6.2.3.4 Validation Process
- **R38**: Document Secure Store (from [4]) **should not** be accessed during validation.

### 6.3 Profiles for XAdES signatures on XFA Forms
#### 6.3.1 Overview
- Two profiles: basic (XAdES-E-BES, -EPES, -T) and long-term (using DSS/VRI from PAdES-B-LTV).
- Scenarios: signing only XML data of XFA form, or any XML content allowed by XFA spec.

#### 6.3.2 Profile for Basic XAdES signatures on XFA forms
##### 6.3.2.1 Features
- Signs XFA data or any XFA XML content.
- Levels: BES, EPES, T.
- Supports serial and parallel signatures.

##### 6.3.2.2 General syntax and requirements
- **R39**: Signatures **shall** be XAdES signatures with syntax from [6] and restrictions in this profile.
- **R40**: Signatures **shall** respect XMLDSig requirements from XFA spec except where conflicting.
- **R41**: Unsigned properties not found may be present and ignored unless used with other profiles.
- **R42**: A time-stamp **should** be applied immediately after signing.
- **R43**: Signature handlers **shall** sign the `SignedProperties` element and the `ds:SignatureProperties` containing signing time and reasons without listing them in the signature manifest.

##### 6.3.2.3 Mandatory operations
###### 6.3.2.3.1 Protecting the signing certificate
- **R44**: This profile recommends using `SigningCertificate` qualifying property. At least one of `SigningCertificate` element and signed certificate in `ds:KeyInfo` **shall** be present.

##### 6.3.2.4 Requirements on XAdES optional properties
- Table 4 specifies presence: `SigningTime` **shall not be present**; `SignaturePolicyIdentifier` **may be present**; `CommitmentTypeIndication` conditioned on policy; others may be present; all referenced to [6].

**(Table 4 condensed: SigningTime: shall not be present; SignaturePolicyIdentifier: may be present; SignatureProductionPlaceV2: may be present; SignerRoleV2: may be present; DataObjectFormat: may be present; CommitmentTypeIndication: conditioned (shall be present only if SignaturePolicyIdentifier is present); AllDataObjectsTimeStamp: may be present; IndividualDataObjectsTimeStamp: may be present; SignatureTimeStamp: may be present (should be present); SignaturePolicyStore: may be present. NOTE: XFA processors include additional elements in ds:SignatureProperties; signing time is in CreateDate element.)**

- **R45**: Signatures **shall** sign the `ds:SignatureProperties` element containing additional XFA-specific elements.
- **R46**: If `SignaturePolicyIdentifier` is absent, `CommitmentTypeIndication` **shall not** be present.
- **R47**: If `SignaturePolicyIdentifier` is present, `CommitmentTypeIndication` **may** be present; for XAdES-E-EPES signatures, `ds:SignatureProperties` **shall not** include reason for signing in `description` element (Dublin Core).

##### 6.3.2.5 Serial Signatures
- **R48**: Supported by mechanisms in [6] clause 5.2.7. `CounterSignature` optional; if present, syntax per [6] 5.2.7.2. Detached countersignature should be present within XFA dynamic form.

##### 6.3.2.6 Parallel Signatures
- **R49**: A dynamic XFA form may include several XAdES signatures signing in parallel the XML content.

#### 6.3.3 Profile for long-term validation XAdES signatures on XFA forms
##### 6.3.3.1 Overview
- Profiles XAdES-E-BES, -EPES, -T to support long-term validation using LTV mechanisms from [4] (DSS/VRI dictionaries) to achieve equivalence to XAdES-E-XL and -A.

##### 6.3.3.2 Features
- Includes features a), b), d), e) from clause 6.3.2.
- Equivalent to XAdES-E-XL and -A via LTV-related PDF objects (DSS and VRI) as per clause 5.3.2 of [4].

##### 6.3.3.3 General Requirements
- **R50**: Signature handlers **shall** be able to sign/validate signed XFA dynamic forms with XAdES signatures aligned with this profile.
- **R51**: Signature handlers **shall** support PDF documents with:
  - Document security store as per clause 5.4.2 of [4].
  - Document time-stamps as per clause 5.4.3 of [4].
- **R52**: Validation data carried by value within DSS is supported.
- **R53**: Signature handlers **shall** support generation/validation with one or more DSS entries and document time-stamps.

### 6.3.4 Extensions Dictionary
- **R54**: The extensions dictionary **should** include an entry `<</ESIX<</BaseVersion /1.7 /ExtensionLevel 1>>>>` to identify extensions per clause 5.4 of [4].

## Informative Annexes (Condensed)
- **Annex A (General Features)**: Provides background on PDF signatures, signature types (certification, usage rights), signature handlers (Filter/SubFilter pair), serial signatures (each signature contains one signer; multiple signature dictionaries), validation and time-stamping, ISO 19005-1 (PDF/A-1) limitations (based on PDF 1.4, no embedded revocation/time-stamps), ISO 19005-2 (PDF/A-2) (based on ISO 32000-1, supports PAdES profiles), and seed values vs. signature policies (seed values are workflow constraints; signature policies are general endorsement rules).
