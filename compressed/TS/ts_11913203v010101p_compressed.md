# ETSI TS 119 132-3 V1.1.1 (2021-01): Electronic Signatures and Infrastructures (ESI); XAdES digital signatures; Part 3: Incorporation of Evidence Record Syntax (ERS) mechanisms in XAdES
**Source**: ETSI Technical Committee ESI | **Version**: V1.1.1 | **Date**: January 2021 | **Type**: Normative Technical Specification
**Original**: https://www.etsi.org/deliver/etsi_ts/119100_119199/11913203/01.01.01_60/ts_11913203v010101p.pdf

## Scope (Summary)
Specifies the semantics and syntax for a new unsigned XAdES qualifying property (`xadesen:SealingEvidenceRecords`) that contains evidence records (ERS in XML or ASN.1 format). Defines rules for incorporating evidence records within XAdES signatures or legacy XAdES signatures, including a new XAdES-E-ERS level. Signatures specified are not baseline XAdES signatures.

## 4 General requirements

### 4.1 XML Namespaces
Three namespaces are used:
- `http://www.w3.org/2001/XMLSchema` (prefix `xsd`)
- `urn:ietf:params:xml:ns:ers` (prefix `ers`)
- `http://uri.etsi.org/19132/v1.1.1#` (prefix `xadesen`) — defined by this TS.

ETSI has generated XML Schema file `XAdES1913203v111-202101.xsd`. The `xsd:schema` element is:

```xml
<xsd:schema elementFormDefault="qualified" targetNamespace="http://uri.etsi.org/19132/v1.1.1#"
xmlns="http://uri.etsi.org/19132/v1.1.1#" xmlns:ers="urn:ietf:params:xml:ns:ers"
xmlns:xsd="http://www.w3.org/2001/XMLSchema">
```

### 4.2 Electronic time-stamps in evidence-records
All electronic time-stamps within evidence-records incorporated into XAdES signatures specified in this document **shall** be instances of `TimeStampToken` type as per RFC 3161 updated by RFC 5816.

### 4.3 Placement of validation data of time-stamp tokens
When including validation data for a time-stamp token encapsulated within an `ArchiveTimeStamp`, one of the following methods **shall** be used:
1. Add information in the `SignedData` of the `TimeStampToken` itself.
2. If the time-stamp token is within an XML Evidence Record (RFC 6283), validation data **may** be added within `ers:CryptographicInformation` children of `ers:CryptographicInformationList` that shares the same `ers:ArchiveTimeStamp` as the `ers:TimeStamp` element encapsulating the time-stamp token.

Method 1 **should** be used.

If the XAdES signature contains any qualifying property that encapsulates a time-stamp token (e.g., `xades:SignatureTimeStamp`, `xadesv141:ArchiveTimeStamp`, etc.), before adding any unsigned qualifying property encapsulating an evidence record, the validation material for those time-stamp tokens not already present **shall** be incorporated using `xadesv141:TimeStampValidationData` elements as per EN 319 132-1 and TS 101 903.

## 5 The `xadesen:SealingEvidenceRecords` qualifying property

### 5.1 Semantics

#### 5.1.1 Data time-stamped by time-stamp tokens
- The `xadesen:SealingEvidenceRecords` qualifying property **shall** be an unsigned qualifying property.
- It **shall** contain either a sequence of `ers:EvidenceRecord` (RFC 6283) or a sequence of `EvidenceRecord` (RFC 4998) (base64Binary encoded as `ASN1EvidenceRecord`).
- This qualifying property **may** be incorporated only in XAdES signatures using direct or indirect incorporation.
- Evidence records within the same `xadesen:SealingEvidenceRecords` **shall** act as parallel evidence records protecting the same parts of the signature, except the other evidence records in the same property.
- The first time-stamp token in the initial `ArchiveTimeStamp` within each evidence record **shall** be generated as per clause 5.2.2.
- After the initial `ArchiveTimeStamp`, each evidence record **may** follow its own path, and time-stamp token renewal or hash-tree renewal **may** occur. Both renewals **shall** be performed as per RFC 6283 and RFC 4998.
- Once a `xadesen:SealingEvidenceRecords` is incorporated, the only allowed changes to the XAdES signature are:
  - renewal (time-stamp or hash-tree) of the evidence records within the same property;
  - incorporation of a new evidence record within the same property;
  - incorporation of a new `xadesen:SealingEvidenceRecords` unsigned qualifying property.
- New `xadesen:SealingEvidenceRecords` properties allow serial evidence records.

#### 5.1.2 Incorporation of validation data during the life cycle of evidence records
- Before incorporating the first evidence record, any validation data required for validating any signed component not already present **shall** be incorporated as per EN 319 132-1.
- If validation data contains a Delta CRL, the whole set of CRLs **shall** be included.
- Before adding a new time-stamp token within an evidence record (during renewal), missing validation data for the latest time-stamp token in that evidence record **shall** be incorporated using one of the two methods in clause 4.3.
- If the XAdES signature already contains one or more `xadesen:SealingEvidenceRecords`, before adding a new one, validation data for the last time-stamp tokens of all evidence records in the last `xadesen:SealingEvidenceRecords` **shall** be incorporated using one of the two methods in clause 4.3.

### 5.2 Syntax

#### 5.2.1 Introduction
The `xadesen:SealingEvidenceRecords` unsigned qualifying property **shall** be defined as in XML Schema file `XAdES1913203v111-202101.xsd` (location in clause A.1). The schema definition:

```xml
<!-- targetNamespace="http://uri.etsi.org/19132/v1.1.1#"-->
<xsd:element name="SealingEvidenceRecords" type="EvidenceRecordsType"/>
<xsd:complexType name="EvidenceRecordsType">
<xsd:choice>
<xsd:sequence minOccurs="1" maxOccurs="unbounded">
<xsd:element ref="ers:EvidenceRecord"/>
</xsd:sequence>
<xsd:sequence minOccurs="1" maxOccurs="unbounded">
<xsd:element name="ASN1EvidenceRecord" type="xsd:base64Binary"/>
</xsd:sequence>
</xsd:choice>
</xsd:complexType>
```

It **shall** contain either one or more `ers:EvidenceRecord` elements (RFC 6283) or one or more `ASN1EvidenceRecord` elements (base64 encoded RFC 4998). All such properties in one XAdES signature **shall** have the same type (either XMLERS or ASN.1 ERS).

#### 5.2.2 Requirements for incorporating the first time-stamp token in the initial ArchiveTimestamp within an evidence-record

##### 5.2.2.1 General requirements
The initial time-stamp token encapsulated within the first `ArchiveTimeStamp` of any evidence record **shall** incorporate a `HashTree` whose first child contains the digest of the concatenation (in order per RFC 6283 or RFC 4998) of the following data objects:

1. Data objects from processing each `ds:Reference` in `ds:SignedInfo` per XMLDSIG [7] clause 4.4.3.2; if result is XML node set, canonicalize using `ds:CanonicalizationMethod`.
2. Canonicalized `ds:SignedInfo`, `ds:SignatureValue`, and (if present) `ds:KeyInfo`.
3. All unsigned qualifying properties incorporated except the `xadesen:SealingEvidenceRecords` under construction, canonicalized per clause 4.5 of EN 319 132-1.
4. For each `xadesv141:TimeStampValidationData` added for validation data of time-stamp tokens (excluding those in `xadesen:SealingEvidenceRecords`), the canonicalized object as per steps a)-d) of clause 5.5.2.2 step 4 of EN 319 132-1.
5. All `ds:Object` elements except the one containing `QualifyingProperties` (per step 5 of clause 5.5.2.2 of EN 319 132-1).
6. Objects from signed `ds:Manifest` elements: for each `ds:Reference` in each signed `ds:Manifest`, process the referenced data object per reference processing model; if it is a `ds:Manifest`, recurse.

##### 5.2.2.2 Contribution of unsigned qualifying properties directly incorporated
In non-distributed XAdES signatures using direct incorporation, the set of unsigned qualifying properties contributing to the message imprint **shall** be all unsigned qualifying properties present within the sequence in `xadesen:UnsignedSignatureProperties`.

##### 5.2.2.3 Contribution of unsigned qualifying properties indirectly incorporated
In distributed XAdES signatures using indirect incorporation, the set **shall** be formed by:
1. Unsigned qualifying properties that are children of `xadesen:UnsignedSignatureProperties` descendant of the root `ds:Signature`.
2. Then, for each `xadesen:QualifyingPropertiesReference` child of `ds:Object` within the XAdES root element, in order of appearance: retrieve the referenced `xadesen:QualifyingProperties` and add all unsigned qualifying properties from its `xadesen:UnsignedSignatureProperties`.

## 6 XAdES signature level including ERS

### 6.1 Overview
Specifies a new level (XAdES-E-ERS) created by combining attributes from EN 319 132-1 and this document.

### 6.2 General requirements
- The XAdES signature **shall** incorporate signed and unsigned qualifying properties as per clauses 4 and 5 of EN 319 132-1.
- Algorithms and key lengths **should** be as per ETSI TS 119 312, but can be superseded by national recommendations.

### 6.3 XAdES-E-ERS

**Additional requirements (letters a-r)**:
- a) Generator **shall** include claimed UTC time of signature generation in `xades:SigningTime`.
- b) `xades:SigningCertificateV2` **shall** be present if signing certificate is not in `ds:KeyInfo` or if present but not signed by the signature; otherwise **may** be absent.
- c) References to certificates **should** not include `IssuerSerialV2`.
- d) `xadesv141:SignaturePolicyStore` **may** be incorporated only if `xades:SignaturePolicyIdentifier` with `SigPolicyHash` is present; otherwise **shall not** be incorporated.
- e) Each `xades:SignatureTimeStamp` **may** contain one or more time-stamps from different TSAs.
- f) Time-stamps in `xades:SignatureTimeStamp` **shall** be created before signing certificate is revoked or expired.
- g) Duplication of certificate values **should** be avoided.
- h) Incorporation of `xades:CertificateValues` **shall** be determined by steps 4) a) of algorithms in EN 319 132-1 clauses 5.5.2.2 and 5.5.2.3.
- i) Incorporation of `xades:AttrAuthoritiesCertValues` **shall** be determined by steps 4) c) of those algorithms.
- j) Duplication of revocation values **should** be avoided.
- k) Incorporation of `xades:RevocationValues` **shall** be determined by steps 4) b) of those algorithms.
- l) Incorporation of `xades:AttributeRevocationValues` **shall** be determined by steps 4) d) of those algorithms.
- m) If time-stamp token is within an evidence record, validation data **shall** be incorporated as per clause 4.3 of this document; otherwise as per EN 319 132-1.
- n) Validation data for time-stamps in listed qualifying properties **shall** be included in `xadesv141:TimeStampValidationData` as per EN 319 132-1.
- o) Each `xadesv141:ArchiveTimeStamp` **may** contain more than one time-stamp from different TSAs.
- p) Before adding a new `xadesv141:ArchiveTimeStamp`, all validation material required for validating the signature **shall** be included, including certificates and status information for signing certificate, countersignatures, attribute certificates, and previous time-stamp certificates.
- q) References in `CompleteCertificateRefsV2` and `AttributeCertificateRefsV2` **should** not include `IssuerSerialV2`.
- r) `xadesv141:AttributeCertificateRefsV2` and `xades:AttributeRevocationRefs` **may** be used only when at least an attribute certificate or signed assertion is incorporated; otherwise **shall not** be used.

## 7 Legacy XAdES baseline signatures
When an `ers:SealingEvidenceRecords` unsigned qualifying property is incorporated into legacy XAdES extended signatures, this unsigned qualifying property **shall** comply with the present document.
