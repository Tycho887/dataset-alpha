# ETSI TS 119 472-3 V1.1.1: Profiles for Electronic Attestation of Attributes; Part 3: Profiles for issuance of EAA or PID
**Source**: ETSI | **Version**: V1.1.1 | **Date**: 2026-03 | **Type**: Normative Technical Specification (TS)
**Original**: ETSI TS 119 472-3 V1.1.1 (2026-03)

## Scope (Summary)
Specifies a protocol based on OpenID4VC-HAIP [1] and OpenID4VCI [2] for EUDI Wallets to request issuance of PID or EAA from PID/EAA Providers. Extends OpenID4VC-HAIP to meet EUDI Wallet ARF requirements, including access/registration certificates, Embedded Disclosure Policy, Wallet Instance/Unit Attestations, proof-of-possession, and support for X509-AC and JSON-LD W3C-VC EAA formats.

## Definitions and Abbreviations
- **PID/EAA Provider's Registrar**: entity registering PID/EAA Provider's information for electronic identification and authentication towards EUDI Wallets.
- **Wallet Instance Attestation (WIA)**: JWT signed by Wallet Provider attesting app integrity (based on EUDI Wallet TS03 [i.2]).
- **Wallet Unit Attestation (WUA)**: JWT signed by Wallet Provider describing Wallet Unit components for authentication and validation (based on EUDI Wallet TS03 [i.2]).
- **Abbreviations**: ARF, EAA, EDP, EUDI, HAIP, JWS, JWT, PID, WIA, WSCA, WSCD, WU, WUA, X509-AC.

## 4 EAA issuance flows

### 4.1 General requirements
- **GEN-REQ-4.1-01**: EUDI Wallet **shall** implement OpenID4VCI [2] per OpenID4VC-HAIP [1] clause 4, unless specified otherwise.
- **GEN-REQ-4.1-02**: PID/EAA Providers **shall** implement OpenID4VCI [2] per OpenID4VC-HAIP [1] clause 4, unless specified otherwise.
- **GEN-REQ-4.1-03**: EUDI Wallet **shall** support Authorisation Code Flow (clause 3.4 of [2]) and Pre-Authorised Code Flow (clause 3.5 of [2]).
- **GEN-REQ-4.1-04**: PID/EAA Providers implementing clause 4 **shall** implement at least one of the two flows above.
- **GEN-REQ-4.1-05**: PID Providers implementing Pre-Authorised Code Flow **shall** perform user authorisation with physical presence. (NOTE: excluded for online LoA HIGH due to hijacking risk.)
- **GEN-REQ-4.1-06**: Custom URL scheme "eu-eaa-offer://" **shall** be used to invoke the EUDI Wallet.

### 4.2 Requirements for Issuer Metadata (PID/EAA Provider Metadata)

#### 4.2.1 General requirements
- **ISS-MDATA-4.2.1-01**: Issuer Metadata **shall** be signed metadata per clause 12.2.3 of OpenID4VCI [2] with JWS (RFC 7515 [8]).
- **ISS-MDATA-4.2.1-02**: Signing certificate **shall** be the access certificate of the PID/EAA Provider.
- **ISS-MDATA-4.2.1-03**: `credential_configurations_supported` format values **shall** include one or more values from Annex A of OpenID4VCI [2].

#### 4.2.2 Provision of access certificates to EUDI Wallet
- **ISS-MDATA-ACC_CERT-4.2.2-01**: JWS Protected Header **shall** contain `x5c` parameter (RFC 7515 [8]).
- **ISS-MDATA-ACC_CERT-4.2.2-02**: `x5c` **shall** contain base64 (RFC 4648 [7]) DER-encoded access certificate (which is also signing certificate).
- **ISS-MDATA-ACC_CERT-4.2.2-03**: `x5c` **should** contain base64 DER-encoded certificates in signing certificate's path up to (excluding) trust anchor.

#### 4.2.3 Provision of registration certificates to EUDI Wallet
- **ISS-MDATA-REG_CERT-4.2.3-01**: Issuer Metadata **shall** contain `issuer_info` parameter.
- **ISS-MDATA-REG_CERT-4.2.3-02**: `issuer_info` **shall** be at top level of Signed JWT payload.
- **ISS-MDATA-REG_CERT-4.2.3-03**: `issuer_info` **shall** have same structure as `verifier_info` in OpenID4 VP [9] clause 5.1.
- **ISS-MDATA-REG_CERT-4.2.3-04**: One element of `issuer_info` array **may** contain PID/EAA Provider's registration certificate.
- **ISS-MDATA-REG_CERT-4.2.3-05**: `format` member of that element **shall** be `"registration_cert"`.
- **ISS-MDATA-REG_CERT-4.2.3-06**: `data` member of that element **shall** be the registration certificate.
- **ISS-MDATA-REG_CERT-4.2.3-07**: One element **shall** contain PID/EAA Provider's registration information provided by Registrar.
- **ISS-MDATA-REG_CERT-4.2.3-08**: `format` member of that element **shall** be `"registrar_dataset"`.
- **ISS-MDATA-REG_CERT-4.2.3-09**: `data` member of that element **shall** be a non-empty JSON Object, which:
  - **ISS-MDATA-REG_CERT-4.2.3-10**: **Shall** contain `identifier` member (clause B.2.2 of ETSI TS 119 475 [4]), value = PID/EAA Provider's identifier.
  - **ISS-MDATA-REG_CERT-4.2.3-11**: **Shall** contain `srvDescription` member (array of JSON MultiLangString objects per clause B.2.6 of [4]), contents = registered user-friendly descriptions.
  - **ISS-MDATA-REG_CERT-4.2.3-12**: **Shall** contain `registryURI` member (clause B.2.1 of [4]), value = URI of Registrar.
  - **ISS-MDATA-REG_CERT-4.2.3-13**: **Shall** contain `providesAttestations` member (info on attestation types intended to issue for Wallet Units).
- **Notes**: `providesAttestations` indicates registered attestation types; `credential_configurations_ids` in Credential Offer may differ for active offers.

#### 4.2.4 Provision of PID/EAA reuse policy

##### 4.2.4.1 General requirements
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-01**: `credential_metadata` may contain `credential_reuse_policy` parameter with PID/EAA Provider's preferences to limit reuse.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-02**: Absence means no limitation.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-03**: `credential_reuse_policy` **shall** be a JSON Object.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-04**: It **shall** have `id` member identifying the policy.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-05**: `id` **shall** be a JSON string.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-06**: May have `options` member for specific details.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-07**: `options` **shall** be a JSON Array of objects.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.1-08**: If `id` = `"arf_annex_ii"`, policy **shall** be as per clause 4.2.4.2.

##### 4.2.4.2 ARF pre-defined PID/EAA reuse policy
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-01**: If `id` = `"arf_annex_ii"`, `options` array **shall** be present.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-02**: If `id` = `"arf_annex_ii"`, each element in `options` **shall** contain `"details"` mapped to an array of strings.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-03**: The `"details"` array **shall** contain either `"once_only"` or `"limited_time"`.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-04**: The `"details"` array **may** contain `"rotating-batch"` or `"per-relying-party"` or both.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-05**: If array has single element, Wallet Unit **shall** behave as specified by that string.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-06**: If array has multiple strings, Wallet Unit **should** follow the method whose string appears first and is supported.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-07**: If array contains `"once_only"`, `"rotating-batch"`, or `"per-relying-party"`, the object **shall** also contain a JSON number mapped to `"batch_size"`.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-08**: `"batch_size"` **shall** specify max array size for `proofs` in Credential Request and **shall** be ≥ 2.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-09**: If `"details"` contains `"once_only"`, the object **shall** also contain `"reissue_trigger_unused"` (JSON number).
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-10**: `"reissue_trigger_unused"` **shall** specify lower limit for unused PIDs/EAAs.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-11**: `"reissue_trigger_unused"` **shall** be lower than `"batch_size"`.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-12**: If `"details"` contains `"limited-time"`, `"rotating-batch"`, or `"per-relying-party"`, the object **shall** also contain `"reissue_trigger_lifetime_left"` (JSON number).
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-13**: `"reissue_trigger_lifetime_left"` **shall** specify seconds before expiration triggering reissue.
- **ISS-MDATA-EAA-REUSE-POL-4.2.4.2-14**: If both `batch_credential_issuance` and `credential_reuse_policy` are present, `batch_credential_issuance` **shall** be ignored.

#### 4.2.5 Provision of Embedded Disclosure Policy

##### 4.2.5.1 Introduction (informative)
EDP is a set of rules embedded in an EAA by its provider, indicating conditions for wallet-relying party to access the EAA. Wallet providers must ensure processing of common EDPs from Annex III of [i.3]. Wallet instances verify compliance. EDP metadata is associated with EAA at issuance; not revealed to relying party during presentation.

##### 4.2.5.2 Requirements for the data model of Embedded Disclosure Policy
- **ISS-MDATA-EBD-4.2.5.2-01**: EDP **shall** be identified by a unique URI.
- **ISS-MDATA-EBD-4.2.5.2-02**: EDP **may** be accessible via this URI.
- **ISS-MDATA-EBD-4.2.5.2-03**: EDP identifier **shall** be included with the data set or just identifier if pre-loaded. EAA provider **shall** include either the identifier with data set or just identifier if pre-loaded.
- **ISS-MDATA-EBD-4.2.5.2-04**: EDP **may** include description of applicability to a particular community/class.
- **ISS-MDATA-EBD-4.2.5.2-05**: EDP **may** include authority identifier responsible for the policy.
- **ISS-MDATA-EBD-4.2.5.2-06**: EDP **may** indicate no policy restrictions.
- **ISS-MDATA-EBD-4.2.5.2-07**: EDP **may** contain a list of authorised relying parties: by subject distinguished name (LDAP string per RFC 4514 [6]) and/or URI-encoded entitlements per ETSI TS 119 475 [4], held in registration certificate.
- **ISS-MDATA-EBD-4.2.5.2-08**: EDP **may** define a specific list of roots of trust for access certificates.
- **ISS-MDATA-EBD-4.2.5.2-09**: Each root of trust element **shall** include issuer's distinguished name (LDAP string) and issuer's certificate serial number.
- **ISS-MDATA-EBD-4.2.5.2-10**: Other information **may** be included in an EDP Extension; wallet **may** ignore.
- **ISS-MDATA-EBD-4.2.5.2-11**: Wallet Units **should** still cope with extensions present even if ignored.
- **ISS-MDATA-EBD-4.2.5.2-12**: An EDP Extension **may** define alternative policy rules for specific attributes subject to selective disclosure.
- **ISS-MDATA-EBD-4.2.5.2-13**: EDP **should** contain a link to a website explaining the policy in layman's terms.

### 4.3 Requirements for the Credential Offer
- **ISS-CRED-OFFER-4.3-01**: PID/EAA Providers **shall** support at least one of these Grant Types: `authorisation_code` or `urn:ietf:params:oauth:grant-type:pre-authorised_code` (as defined in clause 4.1.1 of OpenID4VCI [2]).

### 4.4 Requirements for the Pushed Authorisation Request

#### 4.4.1 Scope of application
- **AUTH-REQ-4.4.1-01**: Requirements in clause 4.4 only apply to PID/EAA Providers implementing Authorisation Code Flow (clause 3.5 of [2]).

#### 4.4.2 Requirements for contents
- **AUTH-REQ-4.4.2-01**: **Shall** include WIA as `OAuth-Client-Attestation` parameter (per Appendix E of [2] and [5]).
- **AUTH-REQ-4.4.2-02**: **Shall** include Proof-of-Possession of public key in `cnf` claim (per same references).

#### 4.4.3 Requirements for processing
- **AUTH-REQ-PROC-4.4.3-01**: PID/EAA Provider **shall** verify WIA signature under Wallet Provider's public key from Trusted List.
- **AUTH-REQ-PROC-4.4.3-02**: **Shall** check WIA has not expired.
- **AUTH-REQ-PROC-4.4.3-03**: **Shall** verify PoP signature under public key from WIA `cnf` claim.

### 4.5 Requirements for the Token Request

#### 4.5.1 Requirements for contents
- **TOKEN-REQ-4.5.1-01**: **Shall** include WIA as `OAuth-Client-Attestation` parameter.
- **TOKEN-REQ-4.5.1-02**: **Shall** include Proof-of-Possession of public key in `cnf` claim.

#### 4.5.2 Requirements for processing
- **TOKEN-REQ-PROC-4.5.2-01**: Verify WIA signature under Wallet Provider's public key from Trusted List.
- **TOKEN-REQ-PROC-4.5.2-02**: Check WIA not expired.
- **TOKEN-REQ-PROC-4.5.2-03**: Verify PoP signature under public key from WIA `cnf` claim.
- **TOKEN-REQ-PROC-4.5.2-04**: Refresh tokens for credential refresh **may** be supported.

### 4.6 Requirements for the Credential Request

#### 4.6.1 Requirements for contents

##### 4.6.1.1 Common requirements
- **CRED-REQ-4.6.1.1-01**: If PID/EAA is cryptographically bound to device, **shall** include `proofs` parameter.
- **CRED-REQ-4.6.1.1-02**: `proofs` **shall** contain either `jwt` or `attestation` child member.

##### 4.6.1.2 Using proofs->jwt mechanism
- **CRED-REQ-4.6.1.2-01**: `jwt` array **shall** have 1 element.
- **CRED-REQ-4.6.1.2-02**: Element **shall** contain `nonce` claim in JWT Body (from Nonce Endpoint).
- **CRED-REQ-4.6.1.2-03**: Element **shall** contain `key_attestation` parameter in Protected Header.
- **CRED-REQ-4.6.1.2-04**: `key_attestation` **shall** be the WUA, signed by Wallet Provider.
- **CRED-REQ-4.6.1.2-05**: WUA **shall** be a key attestation in JWT format as per clause D.1 of OpenID4VCI [2].
- **CRED-REQ-4.6.1.2-06**: `attested_keys` claim in WUA **shall** contain one or more public keys owned by the WU.
- **CRED-REQ-4.6.1.2-07**: Element **shall** be signed by Wallet Unit using private key corresponding to first public key in `attested_keys`.

##### 4.6.1.3 Using proofs->attestation mechanism
- **CRED-REQ-4.6.1.3-01**: `attestation` array **shall** contain only one element.
- **CRED-REQ-4.6.1.3-02**: Element **shall** contain `nonce` claim in JWT Body.
- **CRED-REQ-4.6.1.3-03**: Element **shall** be the WUA (does not prove possession).

#### 4.6.2 Requirements for processing

##### 4.6.2.1 Processing with proofs->jwt
- **CRED-REQ-PROC-4.6.2.1-01**: PID/EAA Provider **shall** verify WUA signature under Wallet Provider's public key from Trusted List.
- **CRED-REQ-PROC-4.6.2.1-02**: **Shall** verify jwt element signature under public key in first position of `attested_keys`.
- **CRED-REQ-PROC-4.6.2.1-03**: **Shall** verify `nonce` claim has valid nonce from its Nonce endpoint.
- **CRED-REQ-PROC-4.6.2.1-04**: **Shall** generate as many PIDs/EAAs as elements in `attested_keys`.
- **CRED-REQ-PROC-4.6.2.1-05**: Each PID/EAA **shall** be bound to one of those public keys.
- **CRED-REQ-PROC-4.6.2.1-06**: No two PIDs/EAAs bound to same public key.

##### 4.6.2.2 Processing with proofs->attestation
- **CRED-REQ-PROC-4.6.2.2-01**: PID/EAA Provider **shall** verify WUA signature under Wallet Provider's public key from Trusted List.
- **CRED-REQ-PROC-4.6.2.2-02**: **Shall** generate as many PIDs/EAAs as elements in `attested_keys`.
- **CRED-REQ-PROC-4.6.2.2-03**: Each PID/EAA **shall** be bound to one of those public keys.
- **CRED-REQ-PROC-4.6.2.2-04**: No two PIDs/EAAs bound to same public key.

### 4.7 Requirements for the Notification Request
- **NOT-REQ-4.7-01**: If EUDI Wallet supports notifications, it **shall** send one or more Notification Requests per `notification_id` received.

## 5 Crypto suites
- **CRYPTO-5-01**: In addition to crypto suites in OpenID4VC-HAIP [1], EUDI Wallet and PID/EAA Providers **shall** support A128GCM and A256GCM (NIST SP 800.38D [10]).

## 6 Security considerations
- **SEC-6-01**: Security considerations in clause 13 of OpenID4VCI [2] **shall** apply.
- **SEC-6-02**: Privacy considerations in clause 15 of OpenID4VCI [2] **shall** apply.

## Annex A (normative): Requirements for issuance of X509-AC EAAs
- **A.1 Introduction**: This annex defines specific requirements for X509-AC EAAs per clause 8 of ETSI TS 119 472-1 [3].
- **A.2 Requirements for Issuer Metadata**
  - **ISS-MDATA-A.2-01**: If issuer can issue X509-AC EAAs, `credential_configurations_supported` format **shall** include `"x509_attr"`.
- **A.3 Rules for path in Issuer Metadata for X509-AC EAAs**
  - **ISS-MDATA-X509-AC EAA-A-01**: Attributes in X509-AC EAA **shall** be identified by `path` member within `claims` of `credential_metadata`.
  - **ISS-MDATA-X509-AC EAA-A-02**: `path` **shall** be array of 2 or 3 strings: (1) type of EAA, (2) OID of attribute (dot-separated integers), (3) optionally, name of attribute.

## Annex B (normative): Requirements for issuance of JSON-LD W3C-VC EAAs secured with enveloping proofs
- **B.1 Introduction**: Defines requirements for JSON-LD W3C-VC EAAs secured with enveloping proofs per [11] and profiled in clause 7 of [3].
- **B.2 Requirements for Issuer Metadata**
  - **ISS-MDATA-B.2-01**: If able to issue JSON-LD W3C VC JOSE EAAs, format **shall** include `"vc+jwt"`.
  - **ISS-MDATA-B.2-02**: If able to issue JSON-LD W3C VC SD-JWT EAAs, format **shall** include `"vc+sd-jwt"`.

## Annex C (informative): Change history
Condensed: Document history from V0.0.1 to V1.1.1 (published March 2026). Key changes include redefining transport of WIA and WUA, supporting pre-authorised code flow, two proofs methods (jwt and attestation), PID/EAA reuse policy structure based on ARF Annex II, editorial adjustments, and removal of optional elements.
