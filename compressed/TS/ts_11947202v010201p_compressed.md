# ETSI TS 119 472-2: Electronic Signatures and Trust Infrastructures (ESI); Profiles for Electronic Attestation of Attributes; Part 2: Profiles for EAA/PID Presentations to Relying Party
**Source**: ETSI | **Version**: V1.2.1 | **Date**: 2026-03 | **Type**: Technical Specification (Normative)
**Original**: ETSI TS 119 472-2 (2026-03)

## Scope (Summary)
Specifies three realizations for Presentations of Electronic Attestation of Attributes (EAAP): SD‑JWT VC, ISO/IEC‑mdoc, and JSON‑LD W3C VC. Also specifies two protocol profiles for Relying Parties (RPs) to request EAAPs/PIDs from EUDI Wallets: the ISO/IEC‑mdoc profile (non‑API mediated only) and the OpenID4VC‑HAIP profile (supporting both non‑API and API‑mediated transmission mechanisms).

## Definitions and Abbreviations
- **Electronic Attestation of Attributes Presentation (EAAP)**: data derived from an EAA that is presented to a specific Relying Party (NOTE: This data can be generated in such a way that it is subject‑bound.)
- **relying party registrar**: entity in charge of registering the Relying Party’s information necessary to allow for their electronic identification and authentication towards European Digital Identity Wallets
- **Abbreviations**: BLE, CBOR, COSE, DCQL, EAA, EAAP, EUDI, GEN, HAIP, JOSE, JSON, JSON‑LD W3C VC, JWS, JWT, KB, KB‑JWT, LD, mDL, NFC, OIDFVP, PID, REQ, RO, RP, SD, SD‑JWT VC, SD‑JWT, SD‑JWT+KB, URI, URL, USB, VP, W3C VC, WU, X509‑AC

## 4 Implementation of Electronic Attestation of Attributes Presentations

### 4.1 EAAP implementation based on SD‑JWT VC
- **EAAP‑SD‑JWT VC‑01**: If the SD‑JWT VC EAA contains the `cnf` claim, the corresponding SD‑JWT VC EAAP **shall** be a SD‑JWT+KB as specified in IETF SD‑JWT [2].
- **EAAP‑SD‑JWT VC‑02**: If the SD‑JWT VC EAA does not contain the `cnf` claim, the corresponding SD‑JWT VC EAAP **shall** be a SD‑JWT VC EAAP as specified in IETF SD‑JWT [2].
- **EAAP‑SD‑JWT VC‑03**: A SD‑JWT VC EAAP **shall** be serialized using either the Compact Serialization (clause 5.2 of [2]) or the Flattened JSON Serialization (clause 8.2 of [2]).
- **EAAP‑SD‑JWT VC‑04**: The Key Binding JSON Web Token (KB‑JWT) of a SD‑JWT VC EAAP (which is a SD‑JWT+KB) **shall** be signed by the EAA subject.

### 4.2 EAAP implementation based on ISO/IEC‑mdoc
- **EAAP‑ISO/IEC‑mdoc‑01**: Each element in the `documents` member of a `DeviceResponse` (clause 10.3.3 of [10]) **shall** be an ISO/IEC‑mdoc EAAP if it does not contain the `errors` member.
- **EAAP‑ISO/IEC‑mdoc‑02**: The `issuerSigned` member of the `DeviceResponse` message **shall** contain all the disclosed attributes signed by the EAA/PID Provider.
- **EAAP‑ISO/IEC‑mdoc‑03**: The `deviceSigned` member of the `DeviceResponse` message **may** contain attributes not present in `issuerSigned` iff their inclusion has been explicitly authorized by the EAA/PID Provider.

## 5 ISO/IEC‑mdoc profile

### 5.1 Introduction
Defines a profile for non‑API mediated transmission built on ISO/IEC 18013‑5 [10].

### 5.2 Requirements on EUDI Wallet and RP support
- **ISO/IEC 18013‑SUPPORT‑01**: Wallet Units, PID Providers, Attestation Providers, Wallet Providers, and Relying Parties **shall** not support server retrieval as specified in [10] for requesting/presenting PID or attestation attributes.
- **ISO/IEC 18013‑SUPPORT‑02**: The EUDI Wallet **shall** meet the requirements defined in clause 5.3.
- **ISO/IEC 18013‑SUPPORT‑03**: A Relying Party **shall** meet the requirements defined in clause 5.3.

### 5.3 Protocol requirements

#### 5.3.1 General requirements
- **ISO/IEC 18013‑GEN‑01**: All mandatory requirements for device retrieval data structures defined in [10] **shall** apply.
- **ISO/IEC 18013‑GEN‑02**: All optional requirements for device retrieval data structures defined in [10] **shall** remain optional unless stated otherwise in the present document.
- **ISO/IEC 18013‑GEN‑03**: If the present document modifies a requirement in [10], the modified requirement **shall** prevail.

#### 5.3.2 ISO/IEC‑mdoc EAAP Request contents
- **ISO/IEC 18013‑REQ‑01**: All elements of the `docRequests` array **shall** contain the `readerAuth` member.
- **ISO/IEC 18013‑REQ‑02**: The digital signature in `readerAuth` **shall** be generated with the private key whose corresponding public key is enclosed within the RP access certificate.
- **ISO/IEC 18013‑REQ‑03**: The `x5chain` unprotected header parameter **shall** contain the RP access certificate in its first element, and its certificate path up to, but excluding, the trust anchor.
- **ISO/IEC 18013‑REQ‑04**: Instances of `ItemsRequest` in `docRequests` **shall** contain a non‑empty `requestInfo` member of type `RequestInfo`.
- **ISO/IEC 18013‑REQ‑05**: The `RequestInfo` type **shall** be as the following CDDL:
```cddl
RequestInfo = {
    ?"eUWrprc": bstr,
    "euWrpRegistrarInfo": EUWrpRegistrarInfo
}
```
- **ISO/IEC 18013‑REQ‑06**: If the RP has a registration certificate, the `requestInfo` member **shall** contain a member with label `"euWrprc"`.
- **ISO/IEC 18013‑REQ‑07**: The member with label `"euWrprc"` **shall** map to a CBOR byte string.
- **ISO/IEC 18013‑REQ‑08**: That CBOR byte string **shall** contain the serialization of the RP registration certificate.
- **ISO/IEC 18013‑REQ‑09**: The `requestInfo` member **shall** contain a member with label `"euWrpRegistrarInfo"`.
- **ISO/IEC 18013‑REQ‑10**: That member **shall** map to a CBOR map which **shall** be an instance of `EUWrpRegistrarInfo`.
- **ISO/IEC 18013‑REQ‑11**: The `EUWrpRegistrarInfo` type **shall** be as the following CDDL:
```cddl
EUWrpRegistrarInfo = {
    "identifier": [+Identifier],
    "srvDescription": ServiceDescription,
    "registryURI": tstr,
    "intendedUseIdentifier": tstr,
    "purpose": Purposes,
    "policyURI": tstr,
    ?"credential": [+Credential]
}
Identifier = {
    "type": tstr,
    "identifier": tstr
}
ServiceDescription = [+ MultiLangString]
Purposes = [+ MultiLangString]
MultiLangString = {
    "lang": tstr,
    "content": tstr
}
Credential = {
    "format": tstr,
    "meta": tstr,
    ?"claim": [+Claim]
}
Claim = {
    "path": tstr,
    ?"values": [+(tstr / int / bool)]
}
```

#### 5.3.3 ISO/IEC‑mdoc EAAP Response profile
- **ISO/IEC 18013‑RESP‑01**: In response to a `DeviceRequest`, the WU **shall** generate a `DeviceResponse` as specified in [10].

## 6 Profile built on OpenID4VC‑HAIP

### 6.1 Introduction
Defines a profile supporting both non‑API mediated (sections 5, 5.1, 5.3, 7, 8 of [11]) and API‑mediated (sections 5, 5.2, 5.3, 7, 8 of [11]) transmission mechanisms.

### 6.2 Requirements on EUDI Wallet and RP support
- **OIDFVP‑HAIP‑SUPPORT‑01**: The EUDI Wallet **shall** meet the requirements of clauses 6.3 and 6.4 (non‑API mediated).
- **OIDFVP‑HAIP‑SUPPORT‑02**: The EUDI Wallet **may** meet the requirements of clause 6.5 (API‑mediated).
- **OIDFVP‑HAIP‑SUPPORT‑03**: The EUDI Wallet **shall** support cross‑device flow using only the API‑mediated mechanism.
- **OIDFVP‑HAIP‑SUPPORT‑04**: A Relying Party **shall** meet the requirements of clauses 6.3 and 6.4.
- **OIDFVP‑HAIP‑SUPPORT‑05**: A Relying Party **may** meet the requirements of clause 6.5.

### 6.3 Common requirements for all transmission mechanisms

#### 6.3.1 General requirements
- **OIDFVP‑HAIP‑GEN‑01**: All mandatory requirements in clauses 5, 5.3, 7, and 8 of [11] **shall** apply.
- **OIDFVP‑HAIP‑GEN‑02**: All optional requirements in those clauses **shall** remain optional unless stated otherwise.
- **OIDFVP‑HAIP‑GEN‑03**: If the present document modifies a requirement in [11], the modified requirement **shall** prevail.

#### 6.3.2 Authorization Request (EAAP request) profile

##### 6.3.2.1 General requirements
- **OIDFVP‑HAIP‑COMMON‑REQ‑01**: The Authorization Request **shall** use the Client Identifier Prefix `x509_hash`.

##### 6.3.2.2 Requirements for the Request Object
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑01**: The RO JWT body **shall** contain the `verifier_info` parameter.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑02**: The `verifier_info` parameter **shall** contain RP Registrar‑provided data.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑03**: The element in `verifier_info` array enclosing RP Registrar‑provided data **shall** be a JSON Object which **shall not** contain the `credential_ids` member.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑04**: The `format` member of that element **shall** be `"registrar_dataset"`.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑05**: The `data` member of that element **shall** be a non‑empty JSON Object.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑06**: That data object **shall** contain `identifier` (clause B.2.2 of [14]), whose value is the RP’s identifier.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑07**: **Shall** contain `srvDescription` (clause B.2.2 of [14]), an array of `MultiLangString` objects.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑08**: **Shall** contain `registryURI` (clause B.2.1 of [14]), the URI of the RP Registrar’s API.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑09**: **Shall** contain `intendedUseIdentifier` (clause B.2.7 of [14]).
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑10**: **Shall** contain `purpose` (clause B.2.7 of [14]), an array of `MultiLangString` objects.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑11**: **Shall** contain `policyURI` (clause B.2.8 of [14]).
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑12**: **May** contain `credential` (clause B.2.7 of [14]), an Array of `Credential` objects.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑13**: If the RP has a registration certificate, one element of `verifier_info` **shall** include it.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑14**: That element **shall** be a JSON Object which **shall not** contain `credential_ids`.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑15**: Its `format` member **shall** be `"registration_cert"`.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑16**: Its `data` member **shall** be the base64url encoding of the serialized RP registration certificate.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑17**: The Authority Key Identifier **shall** use the ETSI Trusted Lists mechanism (`"etsi_tl"` type).
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑18**: The RO JWT body **shall** contain the `client_metadata` parameter.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑19**: The `client_metadata` **shall** contain the `jwks` member.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑20**: The `jwks` member **shall** contain `kid` and `use` parameters.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑21**: The `kid` parameters **shall** uniquely identify one key.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑22**: The RO JWT body **shall** contain the `aud` parameter.
- **OIDFVP‑HAIP‑COMMON‑REQ‑RO‑23**: The RO **shall** be signed by the RP using the private key whose corresponding public key is enclosed within the RP access certificate.

#### 6.3.3 Authorization Response (EAAP response) profile
- **OIDFVP‑HAIP‑COMMON‑RESP‑01**: The EUDI Wallet **shall** encrypt the authorization response.

### 6.4 Specific requirements for non‑API mediated transmission mechanism

#### 6.4.1 General requirements
- **OIDFVP‑HAIP‑REDIRECTS‑01**: All mandatory requirements in clause 5.1 of [11] **shall** apply.
- **OIDFVP‑HAIP‑REDIRECTS‑02**: All optional requirements in clause 5.1 of [11] **shall** remain optional unless stated otherwise.
- **OIDFVP‑HAIP‑REDIRECTS‑03**: The EUDI Wallet **shall** support at least `"eu-eaap://"` as a custom URL scheme for its `authorization_endpoint`.
- **OIDFVP‑HAIP‑REDIRECTS‑04**: The Authorization Request **shall** contain the `request_uri` parameter and **shall not** contain the Request Object (RO).

#### 6.4.2 Requirements for the Request Object
- **OIDFVP‑HAIP‑REDIRECTS_RO‑01**: The JWS Protected Header of the RO **shall** incorporate the `x5c` header parameter.
- **OIDFVP‑HAIP‑REDIRECTS_RO‑02**: The `x5c` parameter **shall** contain the RP access certificate in its first element, and its certificate path up to, but excluding, the trust anchor.
- **OIDFVP‑HAIP‑REDIRECTS_RO‑03**: The JWS Protected Header **shall** incorporate the `iat` header parameter.

### 6.5 Specific requirements for API‑mediated transmission mechanism

#### 6.5.1 OpenID4VC‑HAIP‑related requirements
- **OIDFVP‑HAIP‑API‑01**: All mandatory requirements in clause 5.2 of [11] **shall** apply.
- **OIDFVP‑HAIP‑API‑02**: All optional requirements in clause 5.2 of [11] **shall** remain optional unless stated otherwise.
- **OIDFVP‑HAIP‑API‑03**: The EUDI Wallet **shall not** support Authorization Requests with exchange protocol value `"openid4vp-1-unsigned"`.

#### 6.5.2 Additional requirements
- **OIDFVP‑HAIP‑ADD‑API‑01**: The EUDI Wallet **shall** by default disclose the presence of all stored EAAs’ type to the mediating API (clause 5.2 of [11]), but **shall not** disclose the attributes and their values.
- **OIDFVP‑HAIP‑ADD‑API‑02**: If a PID/EAA is deleted on user request, the EUDI Wallet **shall** disclose to the mediating API that it no longer stores that PID/EAA.
- **OIDFVP‑HAIP‑ADD‑API‑03**: If the Wallet is uninstalled, the EUDI Wallet **shall** disclose to the mediating API that it no longer stores any previously disclosed PID(s)/EAA(s).
- **OIDFVP‑HAIP‑ADD‑API‑04**: The EUDI Wallet **shall** provide a global user setting to disable disclosure of stored EAAs via the mediating API. When disabled, the Wallet **shall not** advertise or respond to API‑mediated requests.
- **OIDFVP‑HAIP‑ADD‑API‑05**: Where supported by browser and OS, the EUDI Wallet **shall** use CTAP 2.2 hybrid flow (section 11.5 of [13]) for cross‑device communication.
- **OIDFVP‑HAIP‑ADD‑API‑06**: Browsers and OS **shall** use short‑range CTAP transport bindings (BLE, NFC, USB) to transfer CTAP2 messages.

### 6.6 Security considerations
The security considerations in clause 14 of OpenID4 VP [7] apply.

## Annex A (normative): EAAP implementation based on JSON‑LD W3C VC

### A.1 Introduction
Specifies requirements for generating EAAPs for JSON‑LD W3C VC EAAs (clause 7 of [5]) using JWS signatures. These EAAPs are designated as JSON‑LD W3C VP JOSE EAAPs.

### A.2 JOSE‑signed JSON‑LD W3C EAAPs
- **EAAP‑JSON‑LD W3C VC JOSE‑01**: A JSON‑LD W3C VC JOSE EAAP **shall** meet requirements from clauses 4.13 of [1], 3.1.2 of [3], and the present clause.
- **EAAP‑JSON‑LD W3C VC JOSE‑02**: It **shall** be a JWS signature generated by the EAA subject.
- **EAAP‑JSON‑LD W3C VC JOSE‑03**: The payload **shall** meet clause 4.13 of [1].
- **EAAP‑JSON‑LD W3C VC JOSE‑04**: The payload **shall** have the `verifiableCredential` property.
- **EAAP‑JSON‑LD W3C VC JOSE‑05**: Each element in `verifiableCredential` **shall** have `type`, `@context`, and `id`.
- **EAAP‑JSON‑LD W3C VC JOSE‑06**: The `type` of each element **shall** be `EnvelopedVerifiableCredential`.
- **EAAP‑JSON‑LD W3C VC JOSE‑07**: The `id` of each element **shall** contain one or more data URIs (IETF RFC 2397 [4]).
- **EAAP‑JSON‑LD W3C VC JOSE‑08**: Data URIs **shall** be separated by `;`.
- **EAAP‑JSON‑LD W3C VC JOSE‑09**: If the URI encapsulates JSON‑LD W3C VC JOSE EAAs, the media type **shall** be `application/vc+jwt`.
- **EAAP‑JSON‑LD W3C VC JOSE‑10**: If the URI encapsulates JSON‑LD W3C VC SD‑JWT EAAs, the media type **shall** be `application/vc+sd-jwt`.
- **EAAP‑JSON‑LD W3C VC JOSE‑11**: All signed JSON‑LD W3C VC EAAs in the data URI **shall** use the same serialization (Compact or base64‑encoded Flattened JSON).
- **EAAP‑JSON‑LD W3C VC JOSE‑12**: If using Flattened JSON, the string `;base64,` **shall** be inserted between the media type and the data part.

## Annex D (normative): Requirements for requesting presentation of X509‑AC EAAs

### D.2 Authorization Request
- **X509_AC_EAA‑GEN‑REQ‑01**: For requesting an X509‑AC EAAP (Annex B), the `format` claim within `dcql_query` **shall** have the value `"x509_attr"`.

## Annex E (informative): Change history (Condensed)
Records major restructuring (v1.1.2 to v1.2.1) including: incorporation of API‑mediated transmission mechanisms for both profiles; deletion of incorrect HAIP requirements from ISO/IEC‑mdoc profile; addition of `requestInfo` with Registrar data and optional registration certificate; modifications to HAIP profile making `verifier_info` mandatory, adding registrar‑provided data and registration certificate elements; making API‑mediated support optional; editorial updates.
