# ETSI TS 119 152-1 V1.1.1: Electronic Signatures and Trust Infrastructures (ESI); CB AdES (CBOR-AdES) digital signatures; Part 1: Building blocks and CB-AdES baseline signatures
**Source**: ETSI | **Version**: V1.1.1 | **Date**: 2026-03 | **Type**: Normative Technical Specification
**Original**: DTS/ESI-0019152-1

## Scope (Summary)
Specifies a CBOR format for AdES signatures (CB-AdES) built on COSE (IETF RFC 9052), defining new header parameters and mechanisms to provide equivalent features to CAdES/XAdES/JAdES. Defines four baseline signature levels (B-B, B-T, B-LT, B-LTA) with incremental requirements for long-term validity.

## Definitions and Abbreviations
- **byte string**: sequence of octets (8 bits)
- **CB-AdES signature**: COSE signature meeting the requirements specified in this document.
- **CBOR-bstr-wrapped**: CBOR object wrapped in a CBOR byte string (bstr).
- **COSE signature**: CBOR structure containing a digitally signed message (per IETF RFC 9052).
- **COSE Payload**: binary data object signed by a COSE signature.
- **COSE signature value**: digital signature cryptographic value calculated over protected headers, payload, and externally supplied data, encapsulated in a bstr.
- **electronic time-stamp**: data binding other electronic data to a particular time, per IETF RFC 3161.
- **payload field**: the payload field of COSE_Sign or COSE_Sign1 arrays.
- **Abbreviations**: CBOR, CDDL, COSE, CRL, IETF, OCSP, OID, PKI, URI, UTC, etc.

### Terminology
- "COSE Signature" denotes the CBOR structure for digital signatures per IETF RFC 9052.
- "CBOR data item" – any data item per IETF RFC 8949.
- "header parameter" – member of protected or unprotected headers map (IETF RFC 9052).
- "CB-AdES component" – any constituent (header parameter, map member, array element, etc.).
- Two structures: COSE_Sign (multiple signers) and COSE_Sign1 (single signer).

## 4 General Requirements
### 4.1 CDDL definitions
New types and components defined using CDDL [5].
### 4.2 Requirements on CB-AdES supporting COSE structures
CB-AdES may be built on COSE_Sign or COSE_Sign1.
### 4.3 Encoding
May be untagged or tagged (COSE_Sign_Tagged, COSE_Sign1_Tagged).
### 4.4 Requirements on CB-AdES headers
- Unprotected headers map shall contain only one member: **uHeaders** (a CBOR array).
- Rationale: CBOR maps have no order; uHeaders array ensures order for long-term protection.
- For COSE_Sign: protected headers in body and signer layers allowed; no uHeaders in body layer; uHeaders may be in signer layer.
- For COSE_Sign1: all headers in body layer.
- New header parameters shall be incorporated as specified; other header parameters not profiled may be added.
### 4.5 Requirements on Payload
May be attached or detached. Detached may be one or concatenated objects (see sigD).
### 4.6 Map keys and data tags
- Map keys: integers.
- Tags for tagged data items: integers.
### 4.7 Encapsulation in CBOR byte strings
Encapsulate CBOR components in CBOR byte strings per IETF RFC 9052 clause 9.

## 5 Header Parameters Semantics and Syntax
### 5.1 Header parameters defined by IETF
#### 5.1.1 Introduction
Additional requirements for IETF header parameters.
#### 5.1.2 The alg (algorithm) header parameter
- **Semantics**: Signed header parameter qualifying the signature. Values per IETF RFC 9052.
- **Syntax**: per IETF RFC 9052. Should use algorithms recommended in ETSI TS 119 312. Identifier from IANA "COSE Algorithms".
- In COSE_Sign: placed at signer layer.
#### 5.1.3 The content type (content type) header parameter
- **Semantics**: Signed, qualifies COSE Payload. Not present if sigD present; not present if content type implied; not present if payload is a counter-signed signature.
- **Syntax**: per IETF RFC 9052. In COSE_Sign: signer layer.
#### 5.1.4 The kid (key identifier) header parameter
- **Semantics**: Signed, qualifies signature. Value should be DER-encoded IssuerSerial per IETF RFC 5035.
- **Syntax**: per IETF RFC 9052. In COSE_Sign: signer layer.
#### 5.1.5 The x5u (X.509 URL) header parameter
- **Semantics**: Signed, qualifies signature. Hint for certificate location.
- **Syntax**: per IETF RFC 9360. In COSE_Sign: signer layer.
#### 5.1.6 CBOR component for counter signatures
- **Semantics**: Element of uHeaders. May be either header parameter per IETF RFC 9338 or a CB-AdES signature.
- **Syntax**: per IETF RFC 9338. Signature value computed per RFC 9338 clause 3.3.
#### 5.1.7 The x5t (cert hash) header parameter
- **Semantics**: per IETF RFC 9360.
- **Syntax**: COSE_CertHash = [hashAlg: (int/tstr), hashValue: bstr]. Hash identifier from IANA COSE Algorithms.
- In COSE_Sign: signer layer.
#### 5.1.8 The x5chain (X.509 chain) header parameter
- **Semantics**: May be signed or unsigned. If unsigned, included in uHeaders.
- **Syntax**: COSE_X509 = bstr / [2*certs: bstr]. In COSE_Sign: signer layer.
#### 5.1.9 The iat (issued at) header parameter
- **Semantics**: May be signed or unsigned. Value is the claimed signing time.
- **Syntax**: NumericDate per IETF RFC 8932. Incorporated as CWT claim per IETF RFC 9597. In COSE_Sign: signer layer.
#### 5.1.10 The crit (critical) header parameter
- **Semantics**: Signed, qualifies signature.
- **Syntax**: per IETF RFC 9052. If sigD present, crit shall also be present and include sigD label.
- In COSE_Sign: signer layer.

### 5.2 New signed header parameters
#### 5.2.1 Labels and tags
Signed headers identified by integer labels (Table 1): x5ts=261, srCms=262, sigPl=263, srAts=264, adoTst=265, sigPId=266, sigD=267. CDDL assigns identifiers.
#### 5.2.2 The x5ts (X.509 certificates Thumbprints) header parameter
- **Label**: 261. Signed.
- **Semantics**: Contains references to certificates in the certification path, first is signing certificate.
- **Syntax**: x5ts = [2*x5t: COSE_CertHash]. At least one of x5t, x5ts, or x5chain must be in protected headers.
- In COSE_Sign: signer layer.
#### 5.2.3 The srCms (signer commitments) header parameter
- **Label**: 262. Signed. Qualifies COSE Payload.
- **Semantics**: Indicates commitment type via URI, may have qualifiers.
- **Syntax**: srCms = [+SrCm]; SrCm map with keys: 1=commId (oId), 2=commQuals. commId's id member URI.
- In COSE_Sign: signer layer.
#### 5.2.4 The sigPl (signature production place) header parameter
- **Label**: 263. Signed. Qualifies signer.
- **Semantics**: Address associated with signer.
- **Syntax**: sigPl map with optional keys: 1=addressCountry, 2=addressLocality, 3=addressRegion, 4=postOfficeBoxNumber, 5=postalCode, 6=streetAddress. At least one member.
- In COSE_Sign: signer layer.
#### 5.2.5 The srAts (signer attributes) header parameter
- **Label**: 264. Signed. Qualifies signer.
- **Semantics**: Encapsulates signer attributes (claimed, certified, signed assertions).
- **Syntax**: srAts map with keys: 1=certified (CertifiedAttrs), 2=signedAssertions (AttrArrays), 3=claimed (AttrArrays). CertifiedAttrs: [+CertifiedAttr] with choice x509AttrCert or otherAttrCert. NotCertifiedItem: [mediaType, *label=>value]. Empty not allowed.
- In COSE_Sign: signer layer.
#### 5.2.6 The adoTst (COSE payload time-stamp) header parameter
- **Label**: 265. Signed. Qualifies COSE Payload.
- **Semantics**: Encapsulates one or more electronic time-stamps generated before signature production, message imprint is the COSE Payload.
- **Syntax**: adoTst = tstContainer. Message imprint computation depends on sigD presence.
- In COSE_Sign: signer layer.
#### 5.2.7 The sigPId (signature policy identifier) header parameter
##### 5.2.7.1 Semantics and syntax
- **Label**: 266. Signed. Qualifies signature.
- **Syntax**: sigPId map: 1=id (oId), 2=digAlgVal ([hashAlg, hashValue]), 3=digPSp (bool, default false), 4=sigPQuals ([+SigPQual]).
##### 5.2.7.2 Signature policy qualifiers
- **Semantics**: Three qualifiers: spURI (tag 1), spUserNotice (tag 2), spDSpec (tag 3). Others via otherQuals.
- **Syntax**: SigPQual map: 1=#6.32(tstr) for spURI, 2=SpUserNotice, 3=SpDesc (obId), *label=>value. SpUserNotice: 1=NoticeRef (org, noticeNumbers), 2=explText.
#### 5.2.8 The sigD header parameter
##### 5.2.8.1 Semantics and syntax
- **Label**: 267. Signed. Only for detached payload, at most one per protected header.
- **Semantics**: References one or more detached data objects and specifies how they build COSE Payload. Chaining not allowed.
- **Syntax**: sigD map: 1=mId (URI), 2=pars ([+tstr]), 3=hashM (int/tstr), 4=hashV ([+bstr]), 5=ctys ([+tstr/null]).
- mId and pars shall be present. Conditions for hashM/hashV.
- In COSE_Sign: signer layer.
##### 5.2.8.2 Mechanisms supported by URI-references
###### 5.2.8.2.1 General requirements
- Use URI-references; each element of pars is an URI-reference (resolved to locator per RFC 3986). HTTP scheme shall be supported.
###### 5.2.8.2.2 Mechanism ObjectIdByURI
- mId = "http://uri.etsi.org/19152/ObjectIdByURI". No hashV/hashM. ctys optional. COSE Payload built by concatenating dereferenced objects.
###### 5.2.8.2.3 Mechanism ObjectIdByURIHash
- mId = "http://uri.etsi.org/19152/ObjectIdByURIHash". hashV and hashM shall be present. COSE Payload contributes as empty stream; digest values provide indirect signing.

### 5.3 New unsigned header parameter
#### 5.3.1 The uHeaders header parameter
- **Semantics**: CBOR array in unprotected headers map containing unsigned CBOR values. Order reflects incorporation order. All unsigned attributes placed here.
- **Syntax**: uHeaders = [+bstr .cbor UHeaderInstance]. uHeaders tag = 268. Unsigned attributes labels: sigTst=1, valData=2, arcTst=3, refs=4, sigRTst=5, rfsTst=6, sigPSt=7. Counter signatures with labels 11, 12; x5chain=33.
- uHeaders shall be non-empty array. In COSE_Sign: signer layer. Should be the only member of unprotected headers.
#### 5.3.2 The sigPSt CBOR map
- **Semantics**: Contains signature policy document or a local URI reference.
- **Syntax**: sigPSt map: 1=DocOrLocalURI (sigPolDoc bstr or sigPolLocalURI #6.32(tstr)), 2=spDSpec (oId).
#### 5.3.3 The sigTst CBOR map
- **Semantics**: Encapsulates electronic time-stamps on COSE signature value.
- **Syntax**: sigTst = tstContainer. Message imprint input = COSE signature value.
#### 5.3.4 The valData CBOR map
- **Semantics**: Contains certificates and/or revocation data for validating any signature in CB-AdES.
- **Syntax**: valData map: 1=xVals ([+X509OrOther]), 2=rVals (with crlVals, ocspVals, otherVals). X509OrOther: 1=pkiOb (x509Cert), 2=pkiOb (otherCert). rVals: crlVals, ocspVals each [+pkiOb].
- Empty valData not allowed. xVals at least one member.
#### 5.3.5 The arcTst CBOR map
##### 5.3.5.1 Semantics and syntax
- **Semantics**: Encapsulates electronic time-stamps on COSE Payload, protected headers, signature value, externally supplied data, and uHeaders.
- **Syntax**: arcTst = tstContainer. All required validation material for counter signatures shall be present before first arcTst.
##### 5.3.5.2 Generation and incorporation
Steps: (1) add missing certificates/revocation in valData; (2) compute message imprint per clause 5.3.5.3; (3) request time-stamp tokens; (4) build arcTst; (5) wrap and append to uHeaders.
##### 5.3.5.3 Computation of message-imprint for arcTst
Input is a CBOR array encoded as bstr, containing: context string ("Signature" or "Signature1" or counter-signature context), protected headers (body and signer layer), externally supplied data, COSE Payload (retrieving via sigD if needed), other_fields for version 2 counter signatures, signature bstr, and elements of uHeaders (all elements before the arcTst being validated). Steps 1-12 are defined.

### 5.4 Generally useful syntax
#### 5.4.1 The obId data type
- **Semantics**: Unique permanent identifier for a data object.
- **Syntax**: oId map: 1=id (#6.32(tstr) URI), 2=desc (tstr), 3=docRefs ([#6.32(tstr)]).
#### 5.4.2 The pkiOb data type
- **Semantics**: Incorporates non-CBOR PKI objects (e.g., X.509 certificates).
- **Syntax**: pkiOb map: 1=val (bstr), 2=encoding (#6.32(tstr) URI), 3=specRef (#6.32(tstr) URI). If encoding absent, val is DER-encoded.
#### 5.4.3 Container for electronic time-stamps
##### 5.4.3.1 Introduction
Defines tstContainer type.
##### 5.4.3.2 Containers for electronic time-stamps
Lists: adoTst, sigTst, arcTst, rfsTst, sigRTst.
##### 5.4.3.3 The tstContainer type
- **Semantics**: Encapsulates one or more time-stamp tokens.
- **Syntax**: tstContainer map: 1=tstTokens ([+TstToken]). TstToken map: 1=val (bstr), 2=type (tstr), 3=encoding (#6.32(tstr)), 4=specRef (#6.32(tstr)). For RFC 3161 tokens, type/encoding/specRef not present.

## 6 CB-AdES Baseline Signatures
### 6.1 Signature levels
Four levels: B-B (signed headers), B-T (adds sigTst), B-LT (adds validation data), B-LTA (adds arcTst). Each builds on previous.
### 6.2 General requirements
#### 6.2.1 Algorithm requirements
Algorithms should be per ETSI TS 119 312. MD5 shall not be used.
#### 6.2.2 Notation for requirements
Table 14 uses columns: "shall be present", "shall not be present", "may be present", "shall be provided", "conditioned presence", "*". Cardinality: 0, 1, 0 or 1, ≥0, ≥1.
