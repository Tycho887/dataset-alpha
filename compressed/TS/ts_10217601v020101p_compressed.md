# ETSI TS 102 176-1: Electronic Signatures and Infrastructures (ESI); Algorithms and Parameters for Secure Electronic Signatures; Part 1: Hash functions and asymmetric algorithms
**Source**: ETSI | **Version**: V2.1.1 | **Date**: 2011-07 | **Type**: Normative
**Original**: ETSI TS 102 176-1 V2.1.1 (2011-07)

## Scope (Summary)
Defines lists of hash functions, signature schemes, and recommended combinations ("signature suites") for advanced electronic signatures and related infrastructure. Criteria: algorithm is considered secure, commonly used, and referenceable (e.g., by OID). Annex A specifies algorithms for specific data structures (certificates, CRLs, OCSP, TSTs, etc.). Not a general-purpose catalog; interoperability with security is the main goal.

## Definitions and Abbreviations
- **bit length**: of an integer p is r if 2^(r-1) ≤ p < 2^r
- **signature policy**: set of rules for creation and validation of electronic signature, defining technical and procedural requirements
- **signature scheme**: triplet of signature creation, verification, and key generation algorithms
- **signature suite**: combination of signature scheme with padding method and hash function
- **Abbreviations**: AA, AC, ANSI, CA, CRL, CRT, CSP, CWA, DRNG, DSA, ECC, ECDSA, ECGDSA, IETF, ISO, MGF, NIST, NRNG, OCSP, OID, RfC, RNG, RSA, SCDev, TST, TSU, URI, URN

## 4 Maintenance of the Document
- ETSI shall review the document at most every two years.
- In case of new attacks, ETSI shall update as soon as possible; affected algorithms should no longer be used even before revision.
- Transition periods allowed for phasing out algorithms/parameters; affected certificates and signature products should be withdrawn if security implications are serious.

## 5 Hash Functions
### 5.1 General
Hash functions require: pre-image resistance, 2nd pre-image resistance, collision resistance.
### 5.2 Recommended One Way Hash Functions
- **5.2.1 SHA-1**: No more recommended. SHOULD NOT be used for document signatures. MAY be used for certificate creation/verification if sufficient entropy included. SHA-224 or SHA-256 SHOULD be implemented for new products.
- **5.2.2 RIPEMD-160**: No more recommended due to short hash length.
- **5.2.3 SHA-224**: MAY be used; NOT RECOMMENDED if SHA-256 can be used.
- **5.2.4 SHA-256**: MAY be used.
- **5.2.5 WHIRLPOOL**: MAY be used for time-stamp tokens and with RSA.
- **5.2.6 SHA-384**: MAY be used; NOT RECOMMENDED if SHA-512 can be used.
- **5.2.7 SHA-512**: MAY be used.
- **5.2.8 SHA-3**: NIST competition ongoing; finalist selection in 2012.

#### 6.1.2 Recommended Signature Algorithms
- **6.1.2.1 RSA**: Computations SHALL be as per RFC 3447. Modulus bit length must be at least MinModLen. p and q should have roughly same length. Private key d and public exponent e; ed ≡ 1 mod lcm(p-1,q-1). d must not be too small. Small public exponent (e=3) MAY be used if performance critical, otherwise e ≥ 2^16+1 RECOMMENDED.
- **6.1.2.2 DSA**: Computations SHALL be as per FIPS 186-3. pMinLen and qMinLen as per table 7. SHA-1 not recommended for new applications. RECOMMENDED to use qMinLen=256.
- **6.1.2.3 ecdsa-Fp**: SHOULD be applied as per ANSI X9.62. Class number of endomorphism ring SHALL be at least MinClass=200. r0 SHALL be > r0Min=104. h = n/q ≤ 4.
- **6.1.2.4 ecdsa-F2m**: SHALL be applied as per ANSI X9.62. Class number condition same. h ≤ 4.
- **6.1.2.5 ecgdsa-Fp**: SHALL be applied as per ISO/IEC 14888-3. Parameters same as ecdsa-Fp.
- **6.1.2.6 ecgdsa-F2m**: SHALL be applied as per ISO/IEC 14888-3. Parameters same as ecdsa-F2m.

## 10 Time Period Resistance of Hash Functions and Keys
- **10.1 Hash functions**: SHOULD resist as long as signature verification needed. If not, signature maintenance process SHALL be performed.
- **10.2 Signer's key**: SHOULD resist during certificate validity; else revocation or time-stamping.
- **10.3 Trust anchors**: SHOULD remain secure during whole time signatures need verification. Stronger margin recommended.
- **10.4 Other keys (TSU, CA, CRL issuer, OCSP)**: SHOULD resist during own certificate validity and relying certificates; stronger parameters than signer keys.

## 11 Practical Ways to Identify Hash Functions and Signature Algorithms
- Objects MUST be referenced using an OID and/or URN.
- **11.1 OIDs**: Tables 15-17 provide OIDs for hash functions, signature algorithms, and signature suites.
- **11.2 URNs**: Tables 18-19 provide URNs for hash functions and signature suites.
- **11.3 & 11.4** Address algorithms without OID/URN yet (Whirlpool, ECDSA with RIPEMD-160, ECGDSA).

## Annex A (normative): Algorithms for Various Data Structures
Specifies algorithms for each data structure (CAdES, XAdES, signer certificates, CRLs, OCSP, CA certificates, self-signed, TSTs, TSU certificates, AA certificates). Each table lists issuer/user requirements (SHALL/SHOULD/MAY). Key points: SHA-256 recommended; SHA-1 phased out; RSA, DSA, ECDSA supported.
