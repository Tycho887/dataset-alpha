# ETSI TS 102 778-4: PDF Advanced Electronic Signature Profiles; Part 4: PAdES Long Term - PAdES-LTV Profile
**Source**: ETSI | **Version**: V1.1.2 | **Date**: 2009-12 | **Type**: Technical Specification (Normative)
**Original**: http://www.etsi.org/deliver/etsi_ts/102700_102799/10277804/01.01.02_60/ts_10277804v010102p.pdf

## Scope (Summary)
This document profiles the electronic signature formats in ISO 32000-1 [1] to support Long Term Validation (LTV) of PDF Signatures. It specifies how to include validation information in a PDF Document and protect it with time-stamps so that PDF Signatures can be verified long after signing. The profile supports functionality equivalent to CAdES-X Long and CAdES-A (see TS 101 733 [2]) and XAdES-XL and XAdES-A (see TS 101 903 [3]) in a single profile PAdES-LTV. It also specifies extensions to ISO 32000-1 [1] (see Annex A) to provide LTV features.

## Definitions and Abbreviations

### Definitions
- **conforming signature handler**: software application that performs digital signature operations in conformance with ISO 32000-1 [1] and the appropriate profile.
- **Document Security Store (DSS)**: information appended to a PDF document relating to its security including VRI and indirect references to validation data for all signatures.
- **document time-stamp**: time-stamp applied to a document along with any document security-related information.
- **PDF Signature**: binary data object based on PKCS#7 (RFC 2315 [4]) or related syntax containing a digital signature placed within a PDF document structure as per ISO 32000-1 [1], clause 12.8.
- **signature dictionary**: PDF data structure of type dictionary as described in ISO 32000-1 [1], clause 12.8.1, table 252.
- **signer**: entity that creates an electronic signature.
- **validation data**: data used to determine signature validity (e.g., certificates, CRLs, OCSP responses).
- **Validation Related Information (VRI)**: indirect references to validation data used to validate a specific signature.
- **verifier**: entity that validates an electronic signature.

**Keywords**: "shall" – absolute requirement; "should" – recommended; "may" – permissible.

### Abbreviations
BER, BES, CA, CAdES, CMS, CRL, DSS, EPES, GSM, LTV, OCSP, PAdES, PDF, SIM, VRI, XAdES, XFA.

## 4 Profile for PAdES-LTV

### 4.1 Overview
- Long-term validation requires CA certificates, CRLs, or OCSP responses (validation data). This profile uses two extensions to ISO 32000-1 [1]:
  1. **Document Security Store (DSS)** – carries validation data and optional VRI relating data to a specific signature (see Annex A.1).
  2. **Document Time-stamp** – extends protection lifetime and binds DSS to the document (see Annex A.2).
- The DSS is collected and the signature first verified before the first Document Time-stamp; that indicated time is used as the assumed signing time for re-verification.
- Protection can be extended by adding further DSS information and a new Document Time-stamp (see Figure 3).

### 4.2 General Requirements
- Conforming signature handlers creating/verifying PAdES-LTV documents **shall** support:
  a) Document security store as per clause A.1.
  b) Document time-stamps as per clause A.2.
- It is **recommended** that signed PDF documents contain DSS followed by a document Time-stamp.
- This profile supports validation data carried by value within the DSS.
- **NOTE**: Use of validation data referencing external sources is not supported.
- Systems supporting this profile **shall** support creation and/or validation of signatures with one or more DSS entries and document Time-stamps.

### 4.3 Validation Process
It is **recommended** that the validation process be:
1. Validate the "latest" document Time-stamp at current time with current validation data.
2. Validate the "inner" document Time-stamp at previous document Time-stamp time with validation data present in the previous DSS.
3. Validate the signature and signature Time-stamp at the latest innermost LTV document Time-stamp time using validation data stored in the DSS and time-stamped by successive enveloping time-stamps.
- Validation of documents without document Time-stamps is outside scope.

### 4.4 Extensions Dictionary
The extensions dictionary (ISO 32000-1 [1] clause 7.12) **should** include:
```
<</ESIC
  <</BaseVersion /1.7
    /ExtensionLevel 1
```

## Annex A (normative): ISO 32000-1 LTV Extensions

### A.1 Document Security Store

**Background**: To support long-term validation, PDF allows inclusion of validation data (certificates, CRLs, OCSP responses) in a separate security store (DSS) that can be added after signing. VRI entries identify validation data for specific signatures.

**Catalog Addition** (to ISO 32000-1 [1] Table 28):
| KEY   | TYPE    | VALUE |
|-------|---------|-------|
| DSS   | dictionary | (Optional) Document-wide security-related information. |

**DSS Dictionary**:
- If present, **shall** contain validation-related information only for PKCS#7 (and derivatives) signatures or for XAdES signatures in dynamic XFA [i.2].
- The VRI entry for a signature **shall** be located in an incremental update section after the section with that signature.

**Entries in DSS Dictionary**:
| KEY    | TYPE      | VALUE |
|--------|-----------|-------|
| Type   | name      | (Optional) **shall** be `DSS` if present. |
| VRI    | dictionary | (Optional) Contains Signature VRI dictionaries keyed by base-16-encoded (uppercase) SHA1 digest of the signature. |
| Certs  | array     | (Optional) Array of indirect references to streams, each containing one BER-encoded X.509 certificate (RFC 5280 [7]). |
| OCSPs  | array     | (Optional) Array of indirect references to streams, each containing a BER-encoded OCSP response (RFC 2560 [8]). |
| CRLs   | array     | (Optional) Array of indirect references to streams, each containing a BER-encoded CRL (RFC 5280 [7]). |

**Signature VRI Dictionary**:
- Relates validation data to a specific signature.
- Validation data **shall** be that used by the party adding the DSS to verify the signature (or known to be appropriate).
- Any values in Cert, CRL, OCSP arrays of a VRI **shall** also be present in the applicable DSS dictionary. If a type of data is not used, the corresponding key **shall not** be present.
- A Signature VRI dictionary **shall not** record unsuccessful validation attempts.

**Entries in Signature VRI Dictionary**:
| KEY     | TYPE        | VALUE |
|---------|-------------|-------|
| Type    | name        | (Optional) **Shall** be `VRI`. |
| Cert    | array       | (Optional, not empty) Array of indirect references to streams containing X.509 certificates used in signature validation. **Should** contain all used certificates. |
| CRL     | array       | (Optional, not empty) Array of indirect references to streams containing CRLs used. **Should** represent all used CRLs. |
| OCSP    | array       | (Optional, not empty) Array of indirect references to streams containing OCSP responses used. **Should** represent all used OCSPs. |
| TU      | date        | (Optional) Date/time when VRI was obtained. **Shall** be absent when TS is present. **Should** be ignored per this profile. |
| TS      | stream      | (Optional) Stream containing BER-encoded Time-stamp (RFC 3161 [6]) representing secure time when VRI was obtained. **Shall** be absent when TU is present. **Should** be ignored per this profile. |

- **DocMDP restrictions** (ISO 32000-1 [1] clause 12.8.2.2) **shall not** apply to incremental updates containing DSS, VRI, Certs, CRLs, OCSPs.

**Usage of DSS/VRI in Signature Creation and Validation**:
- Preferred order of search for validation components (from verifier's trust anchors):
  1. Validation data referenced in VRI component in DSS.
  2. Validation data referenced in DSS.
  3. Validation data embedded in the signature.
  4. Validation data from local repository.
  5. Validation data retrieved from online sources.
- For backward compatibility, conforming writers **should** embed validation-related information into the signature itself when possible.

### A.2 Document Time-stamp
- A Document Time-stamp dictionary is a standard Signature dictionary (ISO 32000-1 [1], 12.8.1) with modifications.

**Modifications to Table 252**:
| KEY       | TYPE      | VALUE |
|-----------|-----------|-------|
| Type      | name      | (Optional) If present, **shall** be `DocTimeStamp`. |
| SubFilter | name      | (Required) When Type is DocTimestamp, **shall** be `ETSI.RFC3161`. Other values must be prefixed with registered developer identification (ISO 32000-1 [1] annex E). |
| Contents  | byte string | (Required) Hexadecimal string representing the value of the byte range digest. When SubFilter is `ETSI.RFC3161`, **shall** be the TimeStampToken per RFC 3161 [6] with messageImprint being a hash of the bytes indicated by ByteRange. |
| V         | integer   | (Optional) Version; if present **shall** be 0. |

- The following keys **shall not** be present: Cert, Reference, Changes, R, Prop_AuthTime, Prop_AuthType.
- The following keys **should not** be present: Name, M, Location, Reason, ContactInfo (conforming reader should ignore these).
- When evaluating DocMDP restrictions, the presence of a Document Time-stamp dictionary item **shall** be ignored.
