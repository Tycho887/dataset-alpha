# ETSI TS 119 534-2: Electronic Signatures and Infrastructures (ESI); Testing Conformance and Interoperability of Registered Electronic Mail Services; Part 2: Test suites for interoperability testing of providers using same format and transport protocols

**Original**: https://www.etsi.org/deliver/etsi_ts/119500_119599/11953402/01.02.01_60/ts_11953402v010201p.pdf

## Scope (Summary)
This document defines test suites for interoperability testing of Registered Electronic Mail (REM) services as specified in ETSI EN 319 532 series. It covers black-box, 4-corner, and extended models, including Store&Forward and Store&Notify styles. It also provides a mechanism for documenting and expanding test cases.

## 4 Technical Approach

### 4.1 Components of test cases and their identifiers
- Test cases are built from parameterized scenarios (Clause 5), REM message instances (Clause 6), named sets of entities, and additional requirements.
- Each test case is defined by: `TestCase#i = Scenario_id(<Receiving side identifier>, <REM message id 1>, …, <REM message id N>, <additional requirements set>, <failure reasons>?)`

### 4.2 Adding new test cases
Follow steps: identify entities, define REMSs, define scenario if new, identify all REM messages and MIME sections, define new header/body combinations as needed, define new ERDS evidence sets, and assign identifiers.

## 6 REM Messages Instances

### 6.1 Introduction and technical approach
Instances of REM payloads, REMS notifications, REMS receipts, and REM dispatches are defined using compositions of MIME headers and sections. At least one instance per type includes all optional headers.

### 6.2 Combinations of fields for headers in REM envelopes
#### 6.2.1 Introduction
Defines combinations for REMS relay metadata, signed data, introduction, free text, HTML, original message, extension, ERDS evidence, and signature MIME headers.

#### 6.2.2 REMS relay metadata MIME Header
Tables 10 and 11 define parameterized combinations (e.g., RFCFields#1, NewFields#1 to #11). Table 12 provides composite instances with assurance level and consignment mode parameters. Additional requirements (a-d) govern date relationships.

#### 6.2.3 Signed data MIME header
Shall be as specified in ETSI EN 319 532-3 [5] clause 6.2.2. Identifier: SIGDATA_COMB.

#### 6.2.4 REMS introduction section
- 6.2.4.2: REMS introduction MIME header shall include "REM-Section-Type" with value "rem_message/introduction" and "REM-Content-Type". Identifier: REMS_IntrComb.
- 6.2.4.3: Free text subsection header combinations FreeText#1 (with Content-Disposition) and FreeText#2 (without).
- 6.2.4.4: HTML subsection header shall include "Content-Type" and "Content-Transfer-Encoding". Identifier: HTMLComb.

#### 6.2.5 Original message MIME section header
Table 14: combinations OrMess#1 (all optional), OrMess#2 (without Content-Description), OrMess#3 (without any optional).

#### 6.2.6 Extension MIME section header
Table 15: combinations Ext#1 (all optional), Ext#2 (without REM-Extension-Code and REM-Extension-Namespace-URI), Ext#3 (without any optional).

#### 6.2.7 ERDS evidence MIME section header

#### 6.2.8 REMS signature MIME section header

## 7 Other named sets

Pattern: REMSs(I,J) where I is number of IREMS, J is number of RREMS.

### 7.2 Named sets of additional requirements

Pattern: RecSide(I,J,K) where I is number of recipients, J number of delegates, K number of recipients per delegate (K ≤ I, if I≠0 then K≠0).

## 8 Test cases definition

### 8.1 Introduction
Test cases are defined by parametrizing scenarios with specific REM message instances and additional requirements. Example: REM_SF#3 scenario with parameters.

### 8.2 Test cases for black-box model
#### 8.2.1 Store&Forward style
- 8.2.1.1: Rules: REM-RelayDate absent (NewFields #9-#11). Test cases cover various entity sets (RecSide) and additional requirements.
- 8.2.1.2–8.2.1.14: Define test case sets for each scenario REM_SF#1 to #13 with parameter ranges for I, J, AssLevelComb, ConsignmentModeId, K, L, M, N, O, and FailureReason codes from ETSI EN 319 522-2.

#### 8.2.2 Store&Notify style
- 8.2.2.1: Rules for REM messages: dispatches, receipts, and notifications use specific parameter sets.
- 8.2.2.2: Failure reasons for REMS_SN#2 (RC08) and REMS_SN#3 (RC09).

### 8.3 Test cases for 4-corner model
