# ETSI TS 102 778-2: PDF Advanced Electronic Signature Profiles; Part 2: PAdES Basic - Profile based on ISO 32000-1
**Source**: ETSI | **Version**: V1.2.1 | **Date**: 2009-07 | **Type**: Normative Technical Specification
**Original**: [ETSI TS 102 778-2 V1.2.1](http://www.etsi.org)

## Scope (Summary)
This document profiles the use of PDF signatures as specified in ISO 32000-1 and based on CMS (RFC 3852), for any application where PDF is appropriate for digital document exchange, including interactive forms. It adds additional restrictions to maximize interoperability without repeating base requirements.

## Definitions and Abbreviations
### Definitions
- **certification signature**: Signature used with Modification Detection Permissions (MDP) per ISO 32000-1, clause 12.8.2.2.
- **conforming signature handler**: Software that performs digital signature operations (signing/verifying) in conformance with ISO 32000-1 and this profile.
- **PDF serial signature**: Signature workflow where subsequent signers sign the document plus previous signature and modifications (e.g. form fill-in).
- **PDF signature**: DER-encoded PKCS#7 binary data object containing a digital signature and information to verify it (certificate, revocation info).
- **seed value dictionary**: PDF dictionary (ISO 32000-1, table 234) constraining properties of a signature applied to a signature field.
- **signature dictionary**: PDF dictionary (ISO 32000-1, table 252) containing all information about the digital signature.
- **Conformance keywords**: 
  - **shall**: absolute requirement of this profile.
  - **should**: recommended course of action; alternatives may be valid with careful consideration.
  - **may**: permissible course of action.

### Abbreviations
- CAdES: CMS Advanced Electronic Signatures (per TS 101 733)
- CMS: Cryptographic Message Syntax
- CRL: Certificate Revocation List
- OCSP: Online Certificate Status Protocol
- PDF: Portable Document Format

## 4 Description of Profile (Informative Summary)
This profile specifies a PDF signature as per ISO 32000-1:2008 with additional restrictions for greater interoperability. Key features include: CMS-based signature (PKCS#7 v1.5), support for serial signatures, optional time-stamp and revocation information, integrity and authentication protection, optional reasons/location/contact info. "Legal content attestation" can indicate PDF capabilities affecting signed content (e.g., JavaScript).

### 4.3 Time Stamping
A trusted timestamp should be applied immediately after signing to reflect the actual signing time and prevent repudiation. Time-stamping process per RFC 3161; if embedded, it **shall** be done as described in ISO 32000-1, clause 12.8.3.3.1.

### 4.4 Revocation Checking
Revocation information **should** be embedded as a signed attribute (adbe-revocationInfoArchival) to save time and protect non-repudiation. When validating, a conforming handler **may** ignore embedded revocation info per its own policies. Revocation checking may use CRL (RFC 3280) or OCSP (RFC 2560).

### 4.5 Seed Values and Signature Policies
A seed value dictionary (ISO 32000-1, table 234) can specify rules for signing (digest methods, revocation, timestamp authorities, certificates) as requirements or recommendations. Seed values that would require violation of this profile **shall not** be used.

## 5 Requirements of Profile for CMS Signatures in PDF

### 5.1 Requirements on PDF Signatures
- **a)** PDF Signatures **shall** be as specified in ISO 32000-1, clause 12.8.
- **b)** Signature information **shall** be embedded in the document; ByteRange **shall** cover the entire file including the signature dictionary but excluding the PDF Signature itself.
- **c)** The PDF Signature (DER-encoded PKCS#7) **shall** be placed in the Contents entry of the signature dictionary.
- **d)** The PKCS#7 object **shall** conform to RFC 2315 and **shall** include the signer's X.509 signing certificate at minimum. (Note: PKCS#1 format not supported by this profile.)
- **e)** Timestamping and revocation information **should** be included. Revocation information and the complete certificate chain **shall** be captured and validated before completing signature creation. Revocation information **shall** be a signed attribute.
- **f)** If present, any revocation information **shall** be a signed attribute.
- **g)** Use of RFC 3281 attribute certificates associated with the signer certificate is **not recommended** (reduces interoperability).
- **h)** There **shall** be only a single signer (single "SignerInfo") in any PDF Signature.

### 5.2 Requirements on PDF Conforming Signature Handlers
- **a)** A PDF reader **may** substitute a different conforming signature handler for verification as long as it supports the specified SubFilter format.
- **b)** Only the two SubFilter values listed in ISO 32000-1, clause 12.8.3.3.1 (`adbe.pkcs7.detached` and `adbe.pkcs7.sha1`) **shall** be used. (Note: actual algorithm support per ISO 32000-1, table 257; consult TS 102 176-1 for guidance. SHA-1 being phased out, other hash algorithms recommended.)

### 5.3 Requirements on Signature Validation
When verifying signatures, a reader **shall** invoke the appropriate handler to:
- **a)** Verify document digest matches that in the signature (per ISO 32000-1, clause 12.8.1).
- **b)** Validate the certificate path (RFC 3280) at the time indicated by a time-stamp (clause 4.3) or other trusted indication of signing time. Revocation status **shall** be checked as per clause 4.4.

### 5.4 Requirements on Time Stamping
- **a)** A timestamp from a trusted timestamp server **should** be applied immediately after signature creation.
- **b)** If a timestamp is embedded, it **shall** be embedded as described in ISO 32000-1, clause 12.8.3.3.1.

### 5.5 Requirements on Revocation Checking
- **a)** When validating, a conforming signature handler **may** ignore embedded revocation information in favour of alternative storage or referenced data per its own policies.

### 5.6 Requirements on Seed Values
- **a)** Seed values that would require a conforming signature handler to violate this profile **shall not** be used.

## Informative Content (Condensed)
- **Introduction**: Electronic signatures are essential for trust in digital documents. This profile supports advanced electronic signatures as per European Directive and builds on ISO 32000-1 and CMS. It does not repeat base standard requirements.
- **Clause 4 Description**: Provides an informative overview of features: CMS-based, serial signatures, optional time-stamp/revocation. Explains time stamping (RFC 3161), revocation checking (CRL/OCSP), and seed values (similar to CAdES signature policies).
- **Use of attribute certificates (5.1(g) note)**: Not recommended due to reduced interoperability.
- **Algorithm selection**: Consult TS 102 176-1; SHA-1 phase-out noted.
