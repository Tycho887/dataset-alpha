# ETSI TS 119 411-5 V2.1.1: Electronic Signatures and Trust Infrastructures (ESI); Policy and security requirements for Trust Service Providers issuing certificates; Part 5: Implementation of qualified certificates for website authentication as in amended Regulation 910/2014
**Source**: ETSI | **Version**: V2.1.1 | **Date**: 2025-02 | **Type**: Normative Technical Specification
**Original**: https://www.etsi.org/deliver/etsi_ts/119400_119499/11941105/02.01.01_60/ts_11941105v020101p.pdf

## Scope (Summary)
Defines two approaches for issuing, deploying, and consuming qualified certificates for website authentication (QWACs). Approach #1 (1-QWAC) uses the QWAC directly as a TLS certificate. Approach #2 (2-QWAC) uses a separate TLS certificate bound to the QWAC via a TLS Certificate Binding. Both aim to meet Articles 45 and 45a and Recital 65 of Regulation (EU) 2024/1183.

## Definitions and Abbreviations
- **1-QWAC**: qualified certificate for website authentication based on Approach #1 in clause 1 of this document.
- **2-QWAC**: qualified certificate for website authentication based on Approach #2 in clause 1 of this document.
- **Transport Layer Security (TLS)**: protocol standardized by IETF RFC 8446 [5] used by web browsers to establish encrypted connections to websites.
- **TLS Certificate**: X.509 public key certificate with an extended key usage extension containing id-kp-serverAuth KeyPurposeId, used to authenticate a website in accordance with a TLS protocol.
- **web browser**: software application that acts on behalf of a user to retrieve, render, and interact with websites.
- **web browser vendor**: legal entity who provides a web browser.
- **web browser vendor root store**: set of TLS root certificates that TLS certificates have to chain up to in order to be trusted by the web browser.
- **web browser vendor root store policy**: policy defining individual rules on the issuers of the TLS root certificates in the root store by the web browser vendor.
- **QTSP**: Qualified Trust Service Provider.
- **QWAC**: Qualified Website Authentication Certificate.

## 4 Trust Service Provider Requirements for issuing Qualified Certificates for Website Authentication

### 4.1 1-QWAC Approach
#### 4.1.1 Policy Requirements
- The 1-QWAC certificate **shall** be issued in accordance with one of the following certificate policies as specified in ETSI EN 319 411-2 [3]:
  a) QEVCP-w; or
  b) QNCP-w.

#### 4.1.2 Certificate Profile Requirements
- The 1-QWAC certificate **shall** be issued in accordance with ETSI EN 319 412-4 [4] for the relevant certificate policy as identified in clause 4.1.1.

### 4.2 2-QWAC Approach
#### 4.2.1 Policy Requirements
- The 2-QWAC certificate **shall** be issued in accordance with the QNCP-w-gen certificate policy as specified in ETSI EN 319 411-2 [3].

#### 4.2.2 Certificate Profile Requirements
- The 2-QWAC certificate **shall** be issued in accordance with ETSI EN 319 412-4 [4] for the relevant certificate policy as identified in clause 4.2.1, except: the extKeyUsage value **shall** only assert the extendedKeyUsage purpose of id-kp-tls-binding as specified in Annex A.

## 5 Website Operator Guidance for using Qualified Certificates for Website Authentication

### 5.1 Usage of 1-QWACs
- The website operator requests a 1-QWAC from a QTSP as defined in clause 4.1 and configures its use (e.g., provisions the 1-QWAC on a TLS terminator).

### 5.2 Usage of 2-QWACs
- The website operator requests:
  - a 2-QWAC from a QTSP; and
  - a TLS certificate from a TLS Certificate issuer whose corresponding root CA certificate is included in the desired Web Browser Vendor Root Store(s).
- When using a 2-QWAC, website operators **shall**:
  1. Enumerate the TLS certificate(s) they wish relying parties to use to establish secure connections with their website.
  2. Produce a TLS Certificate Binding over their selected TLS certificate(s) with their 2-QWAC as described in Annex B and place it on their website.
  3. Configure their website to serve:
     a) an HTTP 'Link' response header (as defined in IETF RFC 8288 [6]) with a relative reference to the TLS Certificate Binding, and a rel value of `tls-certificate-binding`; and
     b) the TLS certificate(s).
- Website operators **should** deploy appropriate HTTP caching directives for their TLS Certificate Binding file. Cache lifetime **should not** extend beyond the lifetime of the binding. Operators **may** rely upon a unique binding filename/location to reduce potential cache errors.
- When rolling over from an existing QWAC to a new QWAC, the operator produces a fresh TLS Certificate Binding and deploys it using the ordinary mechanism. Operators **should** remove no-longer-used TLS certificates from the binding to optimize validation.

## 6 Web Browser Requirements for consuming Qualified Certificates for Website Authentication

### 6.1 General Procedures
#### 6.1.1 Determining Usage
- If a certificate is received as part of the TLS Handshake protocol which has QCStatements indicating that it is a QWAC, it **shall** be processed under procedures for 1-QWAC.
- If a certificate is received as part of the TLS Certificate Binding which has QCStatements indicating that it is a QWAC, it **shall** be processed under procedures for 2-QWAC.

#### 6.1.2 Validation of QWACs
- For 1-QWACs and 2-QWACs, validation **shall** include:
  1. that the QWAC includes QCStatements as specified in clause 4.2 of ETSI EN 319 412-4 [4] and the appropriate Policy OID specified in ETSI EN 319 411-2 [3];
  2. that the QWAC chains back through appropriate & valid digital signatures to an issuer on the EU Trusted List which is authorized to issue Qualified Certificates for Website Authentication as specified in ETSI TS 119 615 [1];
  3. that the QWAC's validity period covers the current date and time;
  4. that the website domain name appears in the QWAC's subject alternative name(s); and
  5. that the QWAC's certificate profile conforms with:
     a) For a 1-QWAC, clause 4.1.2; or
     b) For a 2-QWAC, clause 4.2.2.
- The web browser **may** also perform further checks (e.g., revocation status).

### 6.2 Usage of QWACs
#### 6.2.1 Usage of 1-QWACs
- Web browsers **shall**:
  1. Establish a secure TLS connection with the site using the web browsers' procedures and configuration, and evaluate the presented TLS Certificate with the security requirements of the web browser vendor and their policies for web security, domain authentication and encryption as outlined in Recital 65 of Regulation (EU) 2024/1183 [i.3].
     - If this step fails, the procedure finishes negatively.
  2. Evaluate the presented TLS certificate with the validation criteria in clause 6.1.2 to establish if it is considered a 1-QWAC.
     - If this step fails, the procedure finishes negatively.
  3. Display the outcome as specified in clause 6.3.

#### 6.2.2 Usage of 2-QWACs with TLS Certificate Binding
- Web browsers **shall**:
  1. Establish a secure TLS connection with the site using the web browsers' procedures and configuration, and evaluate the presented TLS Certificate with the security requirements of the web browser vendor and their policies (as in 6.2.1 step 1).
     - If this step fails, the procedure finishes negatively.
  2. Examine HTTP headers in any main frame navigation response for a HTTP 'Link' response header with rel value `tls-certificate-binding`.
     - If absent, the procedure finishes negatively.
  3. Fetch the resource at that link and evaluate it for conformance with Annex B.
     - If non-conformant, the procedure finishes negatively.
  4. Examine the QWAC presented in the binding with validation criteria in clause 6.1.2.
     - If fails or not considered a 2-QWAC, the procedure finishes negatively.
  5. Validate the JAdES signature on the TLS Certificate binding according to ETSI EN 319 102-1 [2].
     - If fails or not valid, the procedure finishes negatively.
  6. Validate that the TLS Certificate used in Step 1 appears in the list contained in the validated binding.
     - If fails, the procedure finishes negatively.
  7. Display the outcome as specified in clause 6.3.

### 6.3 Validation outcomes
- If the applicable procedure (clause 6.2) finishes **successfully** (with a valid QWAC), the web browser:
  - **shall** indicate that the QWAC is validated as being qualified as specified in Regulation (EU) 2024/1183 [i.3] using the EU Trust Mark if technically feasible. When using the EU Trust Mark, the web browser **may** provide a link to the relevant Trusted List entry.
  - **shall** display the identity data contained in qualified certificates for website authentication in a user-friendly manner. The web browser **should** take accessibility requirements defined in ETSI EN 301 549 [i.6] into account.
- If the procedure finishes **negatively** (without a valid QWAC), the web browser:
  - **may** use its discretion in presenting the website, but **shall not** display a "EU Qualified Status" indicator (e.g., EU Trust Mark).
  - **may** include a warning that a QWAC failed to validate.

## Annex A (normative): id-kp-tlsBinding EKU specification
Defines the KeyPurposeID `id-kp-tls-binding` using ASN.1 module. Object identifier arc: `{ itu-t(0) identified-organization(4) etsi(0) id-qwacImplementation(194115) tls-binding(1) id-kp-tls-binding(0) }`.

## Annex B (normative): TLS Certificate Binding Profile
A website operator uses a public key certificate to produce a JAdES signature over specified fields using the private key of the QWAC, serialized using JWS Compact Serialization. Header parameters: `alg`, `kid`, `cty` (must be "TLS-Certificate-Binding-v1"), `x5t#S256`, `x5c` (chain ending in EU Trusted List trust anchor), `iat`, `exp`, `sigD` (which includes hashes of TLS certificates). Signature produced per ETSI TS 119 182-1 [7]. Future versions distinguished by different `cty`.

## Annex C (informative): Web Browser Recommendations (Condensed)
- **C.1**: Web browsers should support both 1-QWAC and 2-QWAC. Vendors should make additional requirements available to 1-QWAC issuers and substantiate positive impact on security/privacy.
- **C.2**: Vendors running a root store for TLS certificates should publish and maintain a root store policy. If relying on another root store, they should disclose its location. Additional requirements should not weaken CA/Browser Forum Baseline Requirements.
- **C.3**: Root store policy should contain at least: scope, contact, revisions, lists of inclusion/retention/exclusion requirements, application/removal/update process descriptions. Vendors may publish interpretation guidelines.
- **C.4**: TLS certificate issuers should fully comply with the compliant root store policy. Vendors should respond to inquiries within 14 calendar days.
