# ETSI TS 119 122-2 V1.0.1 (2015-07): Electronic Signatures and Infrastructures (ESI); CAdES digital signatures; Part 2: Extended CAdES signatures
**Source**: ETSI | **Version**: V1.0.1 | **Date**: July 2015 | **Type**: Normative (Technical Specification)
**Original**: ETSI TS 119 122-2, available from http://www.etsi.org/standards-search

## Scope (Summary)
This document specifies CAdES digital signatures built on CMS signatures by incorporating signed and unsigned attributes for long-term validity. It defines extended CAdES signature levels (CAdES-E-BES, CAdES-E-EPES, CAdES-E-T, CAdES-E-A, and those with validation data references) which address incremental requirements to maintain signature validity over time, with higher optionality than baseline profiles defined in ETSI TS 119 122-1.

## Normative References
- [1] ETSI TS 119 122-1: "Electronic Signatures and Infrastructures (ESI); CAdES digital signatures; Part 1: Building blocks and CAdES baseline signatures"

## Definitions and Abbreviations
- **CAdES signature**: digital signature that satisfies the requirements specified within the present document
- **digital signature**: data appended to, or a cryptographic transformation of a data unit that allows a recipient to prove the source and integrity and protect against forgery
- **legacy CAdES 101 733 signature**: CAdES signature generated according to ETSI TS 101 733 [i.1]
- Abbreviations: as given in ETSI TR 119 001 [i.3]

## Modal Verbs Terminology
In the present document "shall", "shall not", "should", "should not", "may", "need not", "will", "will not", "can" and "cannot" are to be interpreted as described in clause 3.2 of the ETSI Drafting Rules. "must" and "must not" are not allowed except in direct citation.

## 4 Additional CAdES levels without references to validation data

### 4.1 Overview
Clause 4.3 defines CAdES-E-BES, CAdES-E-EPES, CAdES-E-T, and CAdES-E-A built on CAdES-E-T. Annex A (normative) defines levels incorporating references to validation data and time-stamps on those references. Requirements tables follow the format in clause 6.2.2 of ETSI TS 119 122-1 [1].

### 4.2 General requirements
- The general CMS syntax **shall** be as specified in ETSI TS 119 122-1 [1], clause 4.
- The signature **shall** contain a CMS SignedData (clause 4.4 of [1]) and at least one SignerInfo (clause 4.6 of [1]).
- Algorithms and key lengths **should** comply with ETSI TS 119 312 [i.6].

### 4.3 CAdES-E-BES, CAdES-E-EPES, CAdES-E-T and CAdES-E-A built on CAdES-E-T signatures
These signatures **shall** be CAdES signatures whose attributes satisfy the requirements in the following table. CAdES-E-EPES adds a signature-policy-identifier attribute. CAdES-E-T adds one or more signature-time-stamp attributes. CAdES-E-A is built on CAdES-E-T and on levels defined in Annex A.

**Notes:**
- [a] ESS signing-certificate **shall** be used if SHA‑1 hash algorithm is used.
- [b] ESS signing-certificate-v2 **shall** be used when another hash algorithm than SHA‑1 is used.
- [c] The signature-policy-store attribute **may** be incorporated only if signature-policy-identifier is also incorporated and contains in sigPolicyHash the digest value of the signature policy document; otherwise it **shall not** be incorporated.
- [d] Attribute-certificate-references and attribute-revocation-references **may** be used when at least a certified signer attribute (certifiedAttributesV2) or a signed assertion (signedAssertions) is present; otherwise they **shall not** be used.
- [e] The issuerSerial field **should not** be included in the encoding of ESSCertID, ESSCertIDv2 or OtherCertID.
- [f] The time-stamp tokens encapsulated within signature-time-stamp **shall** be created before the signing certificate has been revoked or expired.
- [g] The generator **shall** include the full set of certificates, including trust anchors when available, that have been used to validate the signature.
- [h] Duplication of certificate values **should** be avoided.
- [i] If certificate values for long-term validation are not yet included elsewhere, they **shall** be included in SignedData.certificate per clause 5.5.3 of [1].
- [j] The generator **shall** include the full set of revocation data (CRL or OCSP responses) used in validation.
- [k] Duplication of revocation values **should** be avoided.
- [l] When revocation data contains CRLs not already included, the CRL values **shall** be included within SignedData.crls.crl.
- [m] When revocation data contains OCSP responses not already included, the OCSP values **shall** be included within SignedData.crls.other.
- [n] When a long-term-validation attribute is not present, applications **shall** generate an archive-time-stamp-v3 attribute whenever a new archive time-stamp is needed.
- [o] When a long-term-validation attribute is already present, applications **shall** generate a long-term-validation attribute whenever renewal is needed.
- [p] Before generating or incorporating an archive-time-stamp-v3 attribute, all validation material required for verifying the signature (including previous archive time-stamp validation material) **shall** be included.
- [q] Before adding an additional long-term-validation attribute, all validation material required for verifying the signature and previous long-term-validation attributes **shall** be included in extraCertificates and extraRevocation of the new attribute.
- Note 1: content-time-stamp, signature-time-stamp, CAdES-C-timestamp, and time-stamped-certs-crls-references can have multiple instances from different TSUs.
- Note 2: The signature policy can establish specific requirements for other attributes.
- Note 3: Further information on signature policies is provided in ETSI TS 119 172-1 [i.5].
- Note 4: Trusted time indications provide the initial step towards long-term validity.
- Note 5: This form also provides protection against a TSP key compromise.

## 5 Legacy signatures
When new attributes are incorporated to legacy CAdES 101 733 signatures, these attributes **shall** comply with ETSI TS 119 122-1 [1].

## Annex A (normative): CAdES levels with references to validation data

### A.1 CAdES-E-C, CAdES-E-X, CAdES-E-X-Long and CAdES-E-X-L signatures
- CAdES-E-C: built on CAdES-E-T by adding complete-certificate-references and complete-revocation-references.
- CAdES-E-X Type 1: built on CAdES-E-C by adding one or more CAdES-C-Timestamp attributes.

- CAdES-E-X-L Type 1: built on CAdES-E-X Type 1 by adding certificate-values and revocation-values.
