# ETSI TS 119 142-1: Electronic Signatures and Infrastructures (ESI); PAdES digital signatures; Part 1: Building blocks and PAdES baseline signatures
**Source**: ETSI | **Version**: V1.0.1 | **Date**: 2015-07 | **Type**: Normative Technical Specification
**Original**: http://www.etsi.org/standards-search

## Scope (Summary)
Specifies PAdES digital signatures building on PDF signatures (ISO 32000-1) with alternative encoding equivalent to CAdES. Defines four levels (B-B, B-T, B-LT, B-LTA) of PAdES baseline signatures for long-term validity. Procedures for creation/validation are out of scope (ETSI TS 119 102).

## Definitions and Abbreviations
- **electronic time-stamp**: data that binds other electronic data to a particular time (RFC 3161/RFC 5816)
- **generator**: any party that creates or adds attributes to a signature (signer, initial validator, or maintainer)
- **Legacy PAdES baseline signature**: signature according to ETSI TS 103 172
- **PAdES signature**: signature satisfying requirements of this document and ETSI TS 119 142-2
- **proof of existence**: information proving data existed before a given time
- **signature handler**: software module implementing signing and/or authentication
- **trust service provider**: body operating one or more electronic Trust Services
- **verifier**: entity that validates a digital signature
- **DSS**: Document Security Store
- **ESS**: Enhanced Security Services
- **TSL**: Trust Status List
- **VRI**: Validation Related Information

## 4 General syntax

### 4.1 General requirements for PAdES signatures based on PDF signatures
- **a)** A DER-encoded SignedData object (CAdES) shall be included as PDF signature in Contents entry of Signature Dictionary (ISO 32000-1 clause 12.8.1). There shall be only one signer (single SignerInfo) in any PDF Signature.
- **b)** ISO 32000-1 clause 12.8 requirements apply except where overridden by this document.
- **c)** For attributes/keys with same/similar meaning, only one should be used as per Table 1 requirements (clause 6.3).

## 5 Attributes syntax and semantics

### 5.1 Introduction
Distinguishes three attribute types: CMS/CAdES defined, ISO 32000-1 defined, and validation/archive validation data attributes. Clause 6.3 specifies how to use them.

### 5.2 CMS and CAdES defined attributes
The following attributes may be used in the DER-encoded SignedData object; syntax as per ETSI TS 119 122-1 clause 5:
- content-type, message-digest, signing certificate reference attributes (ESS signing-certificate, ESS signing-certificate-v2), commitment-type-indication, signer-attributes-v2, content-time-stamp, signature-policy-identifier, signature-time-stamp

### 5.3 ISO 32000-1 defined attributes
- Entries of Signature Dictionary shall be as defined in ISO 32000-1 clause 12.8.1 unless otherwise specified. Keys: M, Contents, Filter, SubFilter, ByteRange, Location, Name, ContactInfo, Reason.

### 5.4 Validation data and archive validation data attributes

#### 5.4.1 Overview
Describes Document Security Store (DSS) extension to carry validation data, optionally with VRI dictionaries relating data to specific signatures. Also defines Document Time-stamp (clause 5.4.3) to extend protection lifetime. Life-cycle allows repeated LTV with new DSS and document time-stamps.

#### 5.4.2 Document Security Store

##### 5.4.2.1 Catalog
Added to ISO 32000-1 Table 28: Optional DSS dictionary key.

##### 5.4.2.2 DSS Dictionary
- **Shall** have key DSS in document catalog. Contains validation-related info for PKCS#7/CMS signatures (or XAdES for dynamic XFA).
- **Entries**: Type (Name, optional, value DSS), VRI (Dictionary, optional - maps SHA1 digest to signature VRI), Certs (Array of indirect refs to DER-encoded X.509 certificates per RFC 5280), OCSPs (Array of DER-encoded OCSP responses per RFC 6960), CRLs (Array of DER-encoded CRLs per RFC 5280).
- For document signatures: bytes hashed are the complete hexadecimal string in Contents entry. For CRL/OCSP signatures: bytes hashed are the signature objects as BER-encoded OCTET STRING (primitive). For XAdES: canonicalize ds:Signature using Exclusive XML Canonicalization then hash.
- **VRI dictionaries shall be in incremental update sections**, later than the signature they apply to.

##### 5.4.2.3 Signature VRI Dictionary
- Contains VRI for a specific signature. **Entries**: Type (Name, optional, value VRI), Cert (Array of indirect refs to certificates used in validation), CRL (Array of indirect refs to CRLs used), OCSP (Array of indirect refs to OCSP responses used), TU (Date, optional, absent if TS present), TS (Stream, optional, absent if TU present; contains DER-encoded time-stamp token per RFC 3161/RFC 5816).
- **Additional requirements**: TU key should not be used; TS key should not be used; exactly one of TU, TS, or a subsequent document time-stamp shall provide VRI generation claimed time.
- For PKCS#7: datum hashed in TS is encryptedDigest field. VRI is optional for optimization; all data can be in DSS.
- **Cert, CRL, OCSP values in VRI shall also be present in DSS.** If signature has no associated items, corresponding key not present.
- **VRI shall not be used for unsuccessful validation attempts.**
- **DocMDP restrictions (ISO 32000-1 clause 12.8.2.2) shall not apply** to incremental updates containing DSS and VRI.

#### 5.4.3 Document Time-stamp
- Document Time-stamp dictionary is a standard Signature Dictionary (ISO 32000-1 clause 12.8.1) with modifications:
  - Type: **Required**, value DocTimeStamp
  - SubFilter: **Required**, value should be ETSI.RFC3161; other values prefixed with registered developer ID.
  - Contents: **Required** (hex string) value of TimeStampToken per RFC 3161/RFC 5816 when SubFilter is ETSI.RFC3161. messageImprint shall be hash of bytes indicated by ByteRange. ByteRange shall cover entire document including the dictionary but excluding Contents.
  - V: **Optional**, if present shall be 0.
- **Keys that shall not be present**: Cert, Reference, Changes, R, Prop_AuthTime, Prop_AuthType.
- **Keys that should not be present**: Name, M, Location, Reason, ContactInfo (conforming reader should ignore).
- **DocMDP restrictions**: presence of Document Time-stamp is ignored. Updates to time-stamp and revocation info use same LTV methodology.

### 5.5 Requirements on encryption
- Encryption of PDF content shall be applied before any signature is incorporated.
- Encryption applies to all strings and streams except:
  - ID entry in trailer
  - Strings in Encrypt dictionary
  - Strings inside streams (already encrypted)
  - Hexadecimal strings representing Contents key in Signature Dictionary.

### 5.6 Extensions dictionary
The extensions dictionary (ISO 32000-1 clause 7.12) should include:
- Entry for ESIC with BaseVersion /1.7, ExtensionLevel 1 (for clause 5.4 extensions)
- Entry for ESIC with BaseVersion /1.7, ExtensionLevel 2 (for clause 6.3 requirement l)
- Equivalent Adobe entry: /ADBE, BaseVersion /1.7, ExtensionLevel 8.

## 6 PAdES baseline signatures

### 6.1 Signature levels
Four levels:
- **B-B**: signed and some unsigned attributes at generation.
- **B-T**: adds trusted token proving signature existed at a certain time.
- **B-LT**: includes all validation material required for signature validation in the document.
- **B-LTA**: incorporates electronic time-stamps allowing validation long after generation; targets long-term availability and integrity of validation material.

### 6.2 General requirements for PAdES baseline signatures

#### 6.2.1 Algorithm requirements
- Algorithms and key lengths should comply with ETSI TS 119 312. MD5 shall not be used as digest algorithm.

#### 6.2.2 Notation for requirements
- Table 1 uses columns: Attributes/Fields/Services, Presence in each level, Cardinality, References, Additional requirements.
- Values: "shall be present", "shall not be present", "may be present", "shall be provided", "conditioned presence", "*" (should not be present in that level; upper levels may specify otherwise).
- Cardinality: 0, 1, 0 or 1, ≥0, ≥1. Example: DSS is * for B-B/B-T, shall be present for B-LT/B-LTA with cardinality B-B,B-T: 0, B-LT,B-LTA: 1 (i.e., zero or more for lower, one or more for higher).

#### Additional requirements (referenced in Table 1):
- **a)** Generator shall include signing certificate in SignedData.certificates.
- **b)** Generators should include all certificates not available to verifiers for path building; for Trusted Lists (TS 119 612), include intermediary chain.
- **c)** content-type shall have value id-data.
- **d)** commitment-type-indication may be present only if signature-policy-identifier is present; otherwise shall not be present.
- **e)** Either ESS signing-certificate or signing-certificate-v2 shall be used based on hash function.
- **f)** Should prefer ESS signing-certificate-v2.
- **g)** M entry shall contain claimed UTC time per ISO 32000-1 clause 7.9.4.
- **h)** Contents key shall contain DER-encoded SignedData (CMS) forming CAdES (TS 119 122-1).
- **i)** ISO 32000-1 clauses 12.8.3.2 (PKCS#1) and 12.8.3.3 (PKCS#7) shall not be used.
- **j)** Verifier may substitute a different signature handler as long as it supports SubFilter format.
- **k)** ByteRange shall cover entire file including Signature Dictionary but excluding Contents.
- **l)** SubFilter value shall be ETSI.CAdES.detached.
- **m)** Reason key may be used only if signature-policy-identifier is not present in CMS.
- **n)** Trusted time provided by either signature-time-stamp or document-time-stamp.
- **o)** Generator shall use DER encoding for signature-time-stamp attribute.
- **p)** PAdES-B-T may contain several signature-time-stamp or document-time-stamp attributes.
- **q)** May reserve space for signature-time-stamp if propagating B-B to higher levels.
- **r)** Applications should include certificate values in DSS; include all certificates used that are not already present (signing, attribute, TSA).
- **s)** Avoid duplication of certificate values.
- **t)** Include full set of revocation data (CRL or OCSP) used in validation of signer/CA.
- **u)** Use DER encoding for certificate and revocation values.
- **v)** VRI dictionary should not be used.
- **w)** PAdES-B-LTA may have more than one document-time-stamp after DSS/VRI.
- **x)** Before adding document-time-stamp, include all validation material required for signing certificate, attribute certificates, and TSA certificates (including previous document-time-stamps). Should be in DSS.
- **y)** SubFilter value shall be ETSI.RFC3161.

## 6.4 Legacy PAdES baseline signatures
- When new unsigned elements (CAdES attributes or PDF dictionary entries) are incorporated to legacy signatures, they shall comply with this document.
