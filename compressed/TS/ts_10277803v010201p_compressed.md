# ETSI TS 102 778-3: PDF Advanced Electronic Signature Profiles; Part 3: PAdES Enhanced - PAdES-BES and PAdES-EPES Profiles
**Source**: ETSI | **Version**: V1.2.1 | **Date**: July 2010 | **Type**: Normative Technical Specification
**Original**: [ETSI TS 102 778-3 V1.2.1](https://www.etsi.org/deliver/etsi_ts/102700_102799/10277803/01.02.01_60/ts_10277803v010201p.pdf)

## Scope (Summary)
Profiles the use of PDF Signatures (ISO 32000-1 [1]) with CMS (RFC 3852 [4]) to support signature formats equivalent to CAdES‑BES, CAdES‑EPES and CAdES‑T (TS 101 733 [2]). Defines PAdES‑BES (basic CMS with signing certificate substitution protection) and PAdES‑EPES (extends BES with signature policies). Incorporates signature time‑stamp as optional for both profiles. Does not repeat base requirements; disambiguates techniques. Intended for signers. Requirements in this document take precedence over ISO 32000-1 [1] and TS 101 733 [2].

## Definitions and Abbreviations
### Definitions
- **conforming signature handler**: software that performs digital signature operations in conformance with ISO 32000‑1 [1] and the appropriate profile.
- **PDF signature**: binary data object based on CMS containing a digital signature placed within a PDF document as specified in ISO 32000‑1 clause 12.8.
- **signature dictionary**: PDF data structure (dictionary) as described in ISO 32000‑1 clause 12.8.1, table 252.
- **signer**: entity that creates an electronic signature.
- **verifier**: entity that validates an electronic signature.
- **shall**: absolute requirement of a profile.
- **should**: recommended, but alternative may be used with understanding.
- **may**: permissible within a profile.

### Abbreviations
CAdES, CMS, CRL, EPES, GSM, LTV, OCSP, PAdES, PAdES‑BES, PAdES‑EPES, PDF, PKCS, SHA, SIM

## 4 PAdES-BES and PAdES-EPES Profiles

### 4.1 Introduction
These profiles describe signature creation and verification in PDF documents with features similar to CAdES‑BES, CAdES‑EPES and CAdES‑T (TS 101 733 [2]). The signature time‑stamp attribute is optional for both profiles, making the signature effectively a CAdES‑T form. Signature dictionary items (ISO 32000‑1) **should** be used in preference to CAdES attributes unless otherwise specified.

### 4.2 General Requirements
- a) Requirements for PDF Signatures per ISO 32000‑1 clause 12.8 apply except where overridden.
- b) A DER‑encoded `SignedData` (CMS, RFC 3852 [4]) **shall** be included as the PDF signature in the entry with key `Content` of the signature dictionary.
- c) The `ByteRange` **shall** cover the entire file including the signature dictionary but excluding the PDF Signature itself.
- d) Requirements in ISO 32000‑1 clauses 12.8.3.2 (PKCS#1) and 12.8.3.3 (PKCS#7) **do not apply**.
- e) The signature dictionary **shall** contain the value `ETSI.CAdES.detached` for the key `SubFilter`.
- f) A verifier **may** substitute a different signature handler (other than `Filter`) provided it supports the specified `SubFilter` format.
- g) The signature dictionary **shall not** contain a `Cert` entry.
- h) Unsigned attributes not described **may be ignored**; handling of unsupported signed attributes is a matter for the verifier. (NOTE: a signature attribute cannot be supported if no specification on processing exists. See TS 102 778‑4 [9] for LTV support.)
- i) A timestamp from a trusted timestamp server **should** be applied immediately after signature creation.

### 4.3 SignerInfo
Only a single `SignerInfo` **shall** be present in any PDF signature for all profiles.

### 4.4 Mandatory Attributes
All profiles require the following signed attributes:

#### 4.4.1 content-type Attribute
Value **shall** always be `id-data`.

#### 4.4.2 message-digest Attribute
Syntax as defined in CMS (RFC 3852 [4]).

#### 4.4.3 Signing Certificate Reference Attribute
The `ESS signing-certificate` attribute or `ESS signing-certificate-v2` attribute (as in CAdES clause 5.7.3 of TS 101 733 [2]) **shall** be used as a signed attribute. The `Cert` key in the signature dictionary **shall not** be used.  
NOTE: When SHA‑1 is used, `signing‑certificate` is required; otherwise `signing‑certificate‑v2` is required (RFC 5035 [7]).

### 4.5 Attributes Optional in CAdES
These attributes may be present, used as defined in CAdES (TS 101 733 [2]) except as qualified below (this document takes precedence).

#### 4.5.1 signature-policy-identifier Attribute
- **PAdES‑EPES**: **shall** be present as a signed attribute. Rules from CAdES clause 5.8.1 apply.
- Not to be confused with "seed values" (ISO 32000‑1 clause 12.7.4.5). Conforming signature handlers **shall** enforce seed values constraints at signing time; **should** enforce signature policies at signing time when possible. During validation, **should not** enforce seed values but **shall** enforce signature policy constraints.
- Defines four new elements in the “signature field seed value dictionary”:

| Key | Type | Value |
|-----|------|-------|
| `SignaturePolicyOID` | ASCII string (Optional) | String representation of the OID of the signature policy to use when signing. |
| `SignaturePolicyHashValue` | Byte string (Optional) | Hash of the signature policy, computed as in CAdES clause 5.8.1. |
| `SignaturePolicyHashAlgorithm` | ASCII string (Optional) | Hash function used for `SignaturePolicyHashValue`. Represented as in ISO 32000‑1 table 257. |
| `SignaturePolicyCommitmentType` | Array of ASCII strings (Optional) | If `SignaturePolicyOID` present, this array defines allowed commitment types. An empty string indicates the default commitment type. |

If `SignaturePolicyOID` is absent, the other three fields **must be ignored**. If `SignaturePolicyOID` is present but `SignaturePolicyCommitmentType` is absent, all commitments defined by the signature policy can be used.

#### 4.5.2 signature-time-stamp Attribute
**Should** be present as an unsigned attribute. Rules from CAdES clause 6.1 apply. Creates a signature functionally equivalent to CAdES‑T.

#### 4.5.3 signing-time Attribute
**Shall not** be used. The time of signing can be indicated by the `M` entry in the signature dictionary.

#### 4.5.4 counter-signature Attribute
**Shall not** be used.

#### 4.5.5 content-reference Attribute
**Shall not** be used (PDF provides its own cross‑reference mechanisms).

#### 4.5.6 content-identifier Attribute
**Shall not** be used.

#### 4.5.7 content-hints Attribute
**Shall not** be used.

#### 4.5.8 commitment-type-indication Attribute
- **PAdES‑EPES**: **may** be present. Seed values may restrict its values (see 4.5.1).
- **PAdES‑BES**: **shall not** be present.

#### 4.5.9 signer-location Attribute
**Shall not** be present. Location can be indicated by the `Location` entry in the signature dictionary.

#### 4.5.10 signer-attributes Attribute
**May** be present. If present, used as defined in CAdES clause 5.11.3 of TS 101 733 [2]. Attribute certificates **shall not** be included (per ISO 32000‑1 section 12.8.3.3.1).

#### 4.5.11 content-time-stamp Attribute
**May** be present. If present, used as defined in CAdES clause 5.11.4 of TS 101 733 [2].

### 4.6 Signature Validation
A conforming signature handler **shall** perform the following steps when verifying a signature:

#### 4.6.1 Signing Certificate Reference Validation
Verifier **shall** compare the hash of the signer’s certificate with the hash value in the `signing‑certificate` or `signing‑certificate‑v2` attribute. If no match, verifier **should** return an incomplete validation response. Validation rules per RFC 5035 [7] clauses 2 and 8 apply.

#### 4.6.2 Document Digest
Verifier **shall** check that the document digest matches that in the signature as specified in ISO 32000‑1 clause 12.8.1.

#### 4.6.3 Certificate Path Validation
Verifier **shall** validate the certificate path per RFC 3280 [10] clause 6. The signature may be verified against a time other than the current time if all validation information (certificates, revocation) is known to have existed at that time (e.g., using LTV Profiles per TS 102 778‑4 [9]). Otherwise, the verifier’s current time **shall** be used. The claimed signing time (`M` entry) is not a trusted indication.

#### 4.6.4 Revocation Checking
A conforming signature handler **shall** use either or both of: CRL (RFC 5280 [5]) or OCSP (RFC 2560 [6]). If the certificate was revoked after the assumed signing time, a warning may be passed to the user.

### 4.7 Extensions Dictionary
The extensions dictionary (ISO 32000‑1 clause 7.12) **should** include:
```pdf
<</ESIC
<</BaseVersion /1.7 /
ExtensionLevel 2
>>
>>
```
to identify that the PDF document includes extensions as defined in this document.
