# ETSI TS 119 172-4 V1.2.1: Electronic Signatures and Trust Infrastructures (ESI); Signature Policies; Part 4: Signature applicability rules (validation policy) for European qualified electronic signatures/seals using trusted lists

**Source**: ETSI | **Version**: V1.2.1 | **Date**: 2026-02 | **Type**: Technical Specification (Normative)
**Original**: ETSI TS 119 172-4 V1.2.1 (2026-02)

## Scope (Summary)
Specifies a set of rules for determining, using EU Member States trusted lists, whether a digital signature meets the requirements of EU qualified electronic signatures/seals under Directive 1999/93/EC or Regulation (EU) No 910/2014 (eIDAS).

## 4 Signature applicability rules for validation of EU qualified electronic signatures/seals

### 4.0 Introduction
Requirements are organized in four categories:
a) Validation constraints and procedures (per EN 319 102-1).
b) Validation and applicability checking practices.
c) Technical applicability checking process.
d) Reporting results.

### 4.1 Signature applicability rules
Two sets of rules are defined (each with an OID):
1. **id-etsi-sarc-realTimeReq** (OID in Annex A): For contexts requiring real-time validation response; basic signatures acceptable.
2. **id-etsi-sarc-realTimeNotReq** (OID in Annex A): For contexts where real-time response not required, up to 24h delay acceptable, or minimum signature class is signature-with-time.

### 4.2 Validation constraints and validation procedures
- **REQ-4.2-01**: The driving/validation application **shall** follow the validation process per EN 319 102-1 clause 5.1.2 and support the process for Signatures providing Long Term Availability and Integrity of Validation Material.
- **REQ-4.2-02**: Minimum requirements for QES. Additional inputs/requirements may be used; if so, they **shall** be indicated in the report.
- **REQ-4.2-03**: Constraints for validation process:
  - X.509 validation:
    - a) SetOfTrustAnchors **shall** be set from 'Service digital identity' per TS 119 615 clause 4.3, using signing certificate, `http://uri.etsi.org/TrstSvc/Svctype/CA/QC`, and NotBeforeDate.
    - b) Constraints in TS 119 172-1 Table A.2 rows (m)1.2 to (m)1.10 **shall not** be used.
    - c) Revocation:
      - i) RevocationCheckingConstraints **shall** be "eitherCheck".
      - ii) RevocationFreshnessConstraints:
        - For `realTimeReq`: max 24h for signing certificate.
        - For `realTimeNotReq`: max 0 for signing certificate.
      - iii) Constraints (m)2.3 and (m)3 **shall not** be used.
    - d) No constraints on end-entity certificates acting as trust anchor.
  - Cryptographic:
    - e) The cryptographic verification process **shall** enable reporting on cryptographic suites and security issues, referencing national rules or TS 119 312.
    - f) If an algorithm is unsupported, the application **shall not** invalidate the signature but return INDETERMINATE with a warning.
  - Signature elements:
    - g) Failure to comply with signature formats in REQ-4.3-01 **shall** result in a warning, not invalidation.
    - h) **Shall** enforce presence of signed reference or signed copy of signing certificate.

### 4.3 Requirements on signature validation and applicability rules checking practices
- **REQ-4.3-01**: The validation application **should** support signature formats per [7], [8], [9], [10], and baseline profiles [4], [5], [6]. *(Aims to support Commission Implementing Decision 2015/1506)*
- **REQ-4.3-02**: **Should** comply with TS 119 101.
- **REQ-4.3-03**: When provided as a service, processes **should** comply with TS 119 441.
- **REQ-4.3-04**: Relying parties **shall** be provided with unambiguous information on any security relevant issues.
- **REQ-4.3-05**: Relying parties **shall** be provided with procedures and facilities to validate signatures and obtain results.
- **REQ-4.3-06**: Relying parties **shall** be provided with procedures and facilities to identify relevance of further actions for preservation of signed data.

### 4.4 Technical applicability (rules) checking process

#### 4.4.1 Overview
Process to determine technical suitability of a digital signature to be an EU qualified electronic signature/seal using TLs. Input: outputs of clause 4.2 process. Output: status indicating technical suitability.

#### 4.4.2 Processing
- **REQ-4.4.2-01**: The TARC process **shall** perform the process specified in TS 119 615 clause 4.4, with CERT = signing certificate, Date-time = best signature time from REQ-4.2-01.
- **REQ-4.4.2-02**: If QC-Status = "PROCESS_PASSED" and QC-Results includes "QC_For_eSig" or "QC_For_eSeal":
  - a) then the signing certificate **shall** be technically determined at that time as an EU qualified certificate for electronic signature/seal;
  - b) otherwise: process stops, signature determined indeterminate, results reflected in report.
- **REQ-4.4.2-03**: Void.
- **REQ-4.4.2-04**: The TARC process **shall** perform TS 119 615 clause 4.5 with same inputs.
- **REQ-4.4.2-05**: If QSCD-Status = "PROCESS_PASSED" and QSCD-Results = "QSCD_YES":
  - a) then the digital signature **shall** be technically determined as created by a qualified signature/seal creation device;
  - b) otherwise: process stops, signature indeterminate, results in report.
- **REQ-4.4.2-06**: If all three conditions are met:
  - a) signing certificate determined as EU qualified certificate,
  - b) digital signature determined as created by qualified device,
  - c) result of clause 4.2 process is TOTAL-PASSED,
  - then the digital signature **shall** be determined as technically suitable to implement an EU qualified electronic signature/seal; otherwise it shall not.

### 4.5 Requirements on applicability rules checking report
- **REQ-4.5-01**: The report **shall** include (presented meaningfully to a natural person):
  - a) Scope text: *“Signature applicability rules checking (validation rules) for European qualified electronic signatures/seals using trusted lists”* plus description.
  - b) Complete signer data from certificate (Subject and, if present, Subject Alt Name).
  - c) Indication if pseudonym used.
  - d) Time reference:
    - i) Absence of trust anchor in EU TL should be expressed via information.
    - ii) Report **shall** indicate whenever applicable: claimed signing time; time of document time-stamp/assertion; time of signature time-stamp/assertion; revocation/suspension time; OCSP/CRL times; best signature time.
    - iii) For each timing info, report **shall** indicate evidential relevance and level of assurance (including policy ID, accuracy, qualified status).
  - e) Presentation of signed data.
  - f) Signed and unsigned attributes.
  - g) Overall status and reasons.
  - h) Cryptographic suite information and security issues; **shall** clearly indicate which national rules or TS 119 312 failed.
  - i) Freshness value of revocation status information relative to best signature time.
  - j) Optionally, detailed outcome of each step.
- **REQ-4.5-02**: OIDs from Annex A may be used to identify digital signature type or compliance.
