# ETSI TS 102 778-5: PDF Advanced Electronic Signature Profiles; Part 5: PAdES for XML Content - Profiles for XAdES signatures
**Source**: ETSI | **Version**: V1.1.2 | **Date**: 2009-12 | **Type**: Normative (Technical Specification)
**Original**: ETSI TS 102 778-5 V1.1.2 (2009-12)

## Scope (Summary)
Defines four profiles for using XAdES signatures (TS 101 903) to sign XML content within PDF containers: (1) signed XML document embedded in PDF, with basic (XAdES-BES/EPES/T) and long-term (XAdES-C to A) profiles; (2) signed dynamic XFA forms, with basic and long-term (XAdES-LTV) profiles using DSS/VRI dictionaries.

## Definitions and Abbreviations
### Definitions
- **conforming signature handler**: software that performs digital signature operations in conformance with ISO 32000-1 and the applicable profile.
- **data object**: actual binary/octet data being operated on (transformed, digested, or signed) by an application (from [3]).
- **PDF Signature**: binary data object based on CMS (RFC 3852) containing a digital signature placed within a PDF document as per ISO 32000-1 clause 12.8.
- **signature dictionary**: PDF dictionary structure (ISO 32000-1, clause 12.8.1, table 252) containing digital signature information.
- **signer**: entity that creates an electronic signature.
- **verifier**: entity that validates an electronic signature.
- **validation data**: data used to determine signature validity (e.g., certificates, CRLs, OCSP responses).
- **XML document**: data object that is well-formed, as defined in XML specification [7].

**Keywords**: 
- **may**: course of action permissible within a profile.
- **shall**: absolute requirement of a profile (must be strictly followed for conformance).
- **should**: recommendation, preferred but not required.

### Abbreviations
BES, CA, CAdES, CMS, DSS, EPES, LTV, OCSP, PAdES, PDF, VRI, XAdES, XFA, XML

## 4 Profiles for XAdES Signatures of Signed XML Documents Embedded in PDF Containers

### 4.1 Overview
Defines two profiles for arbitrary signed XML documents (e.g., UBL invoices) embedded in PDF:
- **Basic profile** (clause 4.2): XAdES-BES, XAdES-EPES, XAdES-T.
- **Long-term profile** (clause 4.3): upgrading to XAdES-C, XAdES-X, XAdES-XL, XAdES-A.

### 4.2 Profile for Basic XAdES Signatures of XML Documents Embedded in PDF Containers

#### 4.2.1 Features
- Signed XML document (including XAdES signatures) created independently of PDF container.
- XAdES signatures protect signed data objects (integrity, authenticity); signature time-stamp allows non-repudiation.
- Supported forms: XAdES-BES, XAdES-EPES, XAdES-T.
- Supports serial signatures via XAdES countersignatures.
- Supports parallel signatures.

#### 4.2.2 General Syntax and Requirements
- **Shall** follow TS 101 903 syntax with restrictions in this profile.
- Unsigned properties not in this profile may be ignored.
- Handling of unsupported signed properties is a matter for the verifier.

#### 4.2.3 Requirements for Applications Generating Signed XML Document to be Embedded
- Signed XML document **shall** be created independently of final PDF container.
- **Shall** contain at least one XAdES signature and one or more signed data objects.
- Signed data objects and XAdES signature(s) **shall** satisfy one of:
  1. All signed data objects are embedded within the signed XML document.
  2. If detached, it must be possible to build a valid `ds:Reference` per [3] for retrieval.

#### 4.2.4 Mandatory Operations
##### 4.2.4.1 Protecting the Signing Certificate
- **Recommends** using `xades:SigningCertificate` property; may use inclusion of signing certificate within `ds:KeyInfo` and sign at least that certificate.

#### 4.2.5 Requirements on XAdES Optional Properties
- **SigningTime** (optional signed property): if present, syntax, semantics, usage per TS 101 903.
- **SignaturePolicyIdentifier** (optional signed property): if present, per TS 101 903.
- **SignatureProductionPlace** (optional signed property): if present, per TS 101 903.
- **SignerRole** (optional signed property): if present, per TS 101 903.
- **SignedDataObjectFormat** (optional signed property): if present, per TS 101 903.
- **CommitmentTypeIndication** (optional signed property): if present, per TS 101 903.
- **AllDataObjectsTimeStamp** (optional signed property): if present, per TS 101 903.
- **IndividualDataObjectsTimeStamp** (optional signed property): if present, per TS 101 903.
- **SignatureTimeStamp** (optional unsigned property): if present, per TS 101 903.

#### 4.2.6 Serial Signatures
##### 4.2.6.1 The CounterSignature Element
- Optional unsigned property; if present, syntax/semantics/usage per TS 101 903.
##### 4.2.6.2 Detached Serial Signature
- Optionally, a XAdES signature may be countersigned by a detached XAdES signature with `ds:Reference` Type per TS 101 903 clause 7.2.4.1.
- If used, the countersignature **should** also be present within the signed XML content embedded in PDF.

#### 4.2.7 Parallel Signatures
- A signed XML document may include several XAdES signatures signing in parallel the same data objects.

### 4.3 Profile for Long-Term XAdES Signatures of Signed XML Documents Embedded in PDF Containers

#### 4.3.1 Features
- Long-term signature production.
- Signature encoded as XAdES-C, XAdES-X, XAdES-XL, XAdES-A (TS 101 903).

#### 4.3.2 Upgrading Mechanism
- Conforming signature handlers **shall** detach the signed XML document from the PDF, add suitable unsigned XAdES properties for upgrading, then embed the upgraded document back into the PDF container.

#### 4.3.3 Optional Properties
- No additional requirements on unsigned properties for upgrading; all optional. If present, syntax/semantics/usage per TS 101 903.

#### 4.3.4 Validation Process
- **Recommended**:
  1. Validate latest `xades:ArchiveTimeStamp` at current time with current validation data.
  2. Validate inner `xades:ArchiveTimeStamp` at time indicated in previous `xades:ArchiveTimeStamp`.
  3. Validate any `xades:SigAndRefsTimeStamp` or `xades:RefsOnlyTimeStamp` signature and signature time-stamp at latest innermost LTV archive timestamp time using DSS-stored and time-stamped validation data.
  4. Validate signature and any `xades:SignatureTimeStamp` at time indicated in `SigAndRefsTimeStamp`/`RefsOnlyTimeStamp` or latest innermost `ArchiveTimeStamp`.

## 5 Profiles for XAdES Signatures on XFA Forms

### 5.1 Overview
Two profiles for signing dynamic XFA forms: (1) signing only XFA data or any XML content allowed by XFA spec [2]; (2) long-term validation using DSS/VRI (TS 102 778-4).
- Basic profile (clause 5.2): XAdES-BES/EPES/T.
- Long-term profile (clause 5.3): XAdES-LTV, functionally equivalent to XAdES-XL and XAdES-A.

### 5.2 Profile for Basic XAdES Signatures on XFA Forms

#### 5.2.1 Features
- Signs XFA data or XML content per XFA specification.
- Provides integrity, authenticity, and optionally non-repudiation (time-stamp).
- Supported forms: XAdES-BES/EPES/T.
- Supports serial and parallel signatures.

#### 5.2.2 General Syntax and Requirements
- **Shall** follow TS 101 903 syntax with restrictions in this profile.
- **Shall** respect XML Sig requirements from XFA spec [2] except where conflicting with XAdES.
- Unsigned properties not in profile may be ignored.
- A time-stamp from a trusted server **should** be applied immediately after signing.
- Conforming signature handlers **shall** sign `xades:SignedProperties` and `ds:SignatureProperties` containing signing time and reasons, without listing them in the signature manifest.

#### 5.2.3 Mandatory Operations
##### 5.2.3.1 Protecting the Signing Certificate
- **Recommends** `xades:SigningCertificate`; may use inclusion within `ds:KeyInfo` and sign at least that certificate.

#### 5.2.4 Requirements on XAdES Optional Properties
- **SigningTime**: **shall not** be used.
- **SignaturePolicyIdentifier**: optional signed property; if present per TS 101 903.
- **SignatureProductionPlace**: optional signed property; if present per TS 101 903.
- **SignerRole**: optional signed property; if present per TS 101 903.
- **SignedDataObjectFormat**: optional signed property; if present per TS 101 903.
- **CommitmentTypeIndication**: 
  - If not XAdES-EPES or built on EPES, **shall not** be present.
  - If XAdES-EPES or built on EPES, may be present per TS 101 903. In that case, the `ds:SignatureProperties` **shall not** include the reason for signing within the Dublin Core `description` element.
- **AllDataObjectsTimeStamp**: optional signed property; if present per TS 101 903.
- **IndividualDataObjectsTimeStamp**: optional signed property; if present per TS 101 903.
- **SignatureTimeStamp**: **should** be present as unsigned property; syntax/semantics/usage per TS 101 903.

#### 5.2.5 Serial Signatures
##### 5.2.5.1 CounterSignature element
- Optional unsigned property; if present per TS 101 903.
##### 5.2.5.2 Detached serial signature
- May be countersigned by detached XAdES signature with `ds:Reference` Type per TS 101 903 clause 7.2.4.1.
- If used, the countersignature **should** also be present within the XFA dynamic form.

#### 5.2.6 Parallel Signatures
- XFA form may include several XAdES signatures signing in parallel the same XML content.

### 5.3 Profile for Long-Term Validation XAdES Signatures on XFA Forms (XAdES-LTV)

#### 5.3.1 Overview
- Upgrades basic XAdES signatures using DSS/VRI as per TS 102 778-4 to support equivalent of XAdES-XL and XAdES-A.
- No support for XAdES-C or XAdES-X equivalents.

#### 5.3.2 Features
- Provides long-term signature validity equivalent to XAdES-XL and XAdES-A.
- Incorporates validation data in DSS and VRI dictionaries, with document time-stamps.

#### 5.3.3 General Requirements
- Conforming signature handlers **shall** support signing/verifying XFA forms with XAdES-LTV.
- **Shall** support PDF documents with:
  - Document security store as per clause A.1 of [i.9].
  - Document time-stamps as per clause A.2 of [i.9].
- Supports validation data carried by value within DSS.
- **Shall** support generation/validation of signatures with one or more DSS entries and document time-stamps.

#### 5.3.4 Validation Process
- **Recommended**:
  1. Validate latest archive time-stamp at current time.
  2. Validate inner archive time-stamps at previous archive timestamp time using validation data in previous DSS.
  3. Validate signature and any `xades:SignatureTimeStamp` at latest innermost LTV archive timestamp time using DSS-stored validation data.
- Validation of documents without archive time-stamps is outside scope.

### 5.4 Extensions Dictionary
- The extensions dictionary **should** include:
  ```pdf
  <</ESIX<</BaseVersion /1.7 /ExtensionLevel 1>>>>
  ```
  to identify PDF extensions per clause 5.

## Informative Annexes (Condensed)

- **Annex A (informative): Matching of Basic PAdES-LTV XAdES-based profiles to XAdES** – Describes how combinations of basic XAdES signatures (profiled in clause 5.2) and LTV dictionaries (DSS, VRI, document time-stamps) achieve functional equivalence to XAdES-XL and XAdES-A. A table shows the progression: e.g., PAdES-LTV row 3 builds on XAdES-T by adding DSS with indirect references to certificates/CRLs/OCSP and a document time-stamp to achieve XAdES-X-L equivalence; row 4 adds another DSS and time-stamp for XAdES-A. XAdES-C and XAdES-X equivalents are not supported due to reference management issues.
