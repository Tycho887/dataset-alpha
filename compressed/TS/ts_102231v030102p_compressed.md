# ETSI TS 102 231: Electronic Signatures and Infrastructures (ESI); Provision of harmonized Trust-service status information
**Source**: ETSI | **Version**: V3.1.2 | **Date**: 2009-12 | **Type**: Normative (Technical Specification)
**Original**: ETSI TS 102 231 V3.1.2 (2009-12)

## Scope (Summary)
Specifies a standard for a Trust-service Status List (TSL) that makes available trust service status information to determine whether a trust service is or was operating under the approval of any recognized scheme. Defines TSL structure, meaning, and mechanisms for locating, accessing and authenticating TSLs. Applicable to assessment scheme operators and relying parties.

## Definitions and Abbreviations
- **Approval**: Assertion that a trust service has been positively endorsed (active) or has no explicit restriction (passive).
- **Assessment scheme**: Organized process of supervision, monitoring, or approval to ensure adherence to specific criteria.
- **(Electronic) Trust Service (TS)**: Service enhancing trust in electronic transactions.
- **Qualified Certificate**: PKC issued per Directive 1999/93/EC.
- **Scheme operator**: Body responsible for an assessment scheme.
- **Trust Service Provider (TSP)**: Body operating one or more trust services.
- **Trust Service Token (TrST)**: Binary or physical object generated by a trust service.
- **Trusted List (TL)**: EU Member State's supervision/accreditation status list of CSPs per Directive 1999/93/EC.

Abbreviations: ASN.1, CMS, CRL, CSP, ETSI, HTTP, LDAP, OID, PKI, QC, URI, UTC, XML, etc.

## Trust-service status information
- Each assessment scheme **MUST** comply with format and semantics in clause 5.
- TSL types: "Generic" (lists services with direct oversight), "Schemes" (lists TSL issuers). "List" type for pointers to other TSLs only.
- TSL **MUST** support human-readable (PDF/A) and machine-processable (ASN.1 DER or XML) formats.
- TSL **MUST** contain current status and history (if scheme retains history).
- Four major components: scheme info, TSP list, service list, history.

## Trust-service Status List structure
### 5.1 Structure
- TSL tag, scheme info, TSP info (optional), service info (required, at least one), history (if history period >0), signature (optional but recommended).
### 5.1.3 Language support
- Multilingual strings: ISO 10646 UTF-8 with RFC 5646 language tag. EU schemes **SHALL** use at least official language(s) of issuing MS and UK English.
### 5.1.4 Date-time
- **SHALL** be ISO 8601, UTC (Zulu), no fractional seconds.
### 5.1.7 TSL Format
- Human-readable: PDF/A [45]; machine-processable: ASN.1 DER (annex A) or XML (annex B).

### 5.2 TSL Tag
- **REQUIRED**: URI `http://uri.etsi.org/02231/TSLTag`.

### 5.3 Scheme information
| Field | Status | Type/Format | Rule |
|---|---|---|---|
| TSL version identifier | REQUIRED | Integer = 3 | Must be "3". |
| TSL sequence number | REQUIRED | Integer ≥1 | Start 1, increment each release. |
| TSL type | REQUIRED | URI (clause D.2) | "Generic" or "Schemes". |
| Scheme operator name | REQUIRED | MultiLangString | Formal name. |
| Scheme operator address | REQUIRED | PhysicalAndElectronicAddress | Postal + electronic. |
| Scheme name | REQUIRED | MultiLangString | Unique per scheme. |
| Scheme information URI | REQUIRED | MultiLangPointer | Points to scheme details. |
| Status determination approach | REQUIRED | URI | Active/Passive/Delinquent/List. |
| Scheme type/community/rules | OPTIONAL | Sequence of URIs | Identifies community rules. |
| Scheme territory | OPTIONAL | CountryCode (2 letters) | ISO 3166-1 with exceptions (UK, EL, EU). |
| TSL policy/legal notice | OPTIONAL | MultiLangPointer or MultiLangString | Policy or legal notice. |
| Historical information period | REQUIRED | Integer (0..MAX) | 0=no history; 1-65534 = days; ≥65535 = indefinite. For List type: 0. |
| Pointers to other TSLs | OPTIONAL | Sequence of (URI, digital identities, qualifiers) | For List type: **MUST** be present. |
| List issue date and time | REQUIRED | GeneralizedTime | UTC issuance time. |
| Next update | REQUIRED | CHOICE: NULL or GeneralizedTime | NULL if closed. **MUST** be re-issued by this time. |
| Distribution points | OPTIONAL | Sequence of URIs | Where latest TSL is published. |
| Scheme extensions | OPTIONAL | Extensions (RFC 5280) | Criticality as X.509. |
| List of Trust Service Providers | OPTIONAL | Sequence of TSP info | Absent if no TSPs. For List type: **SHALL NOT** be present. |

### 5.4 TSP information
- **TSP name** (REQUIRED, MultiLangString), **TSP trade name** (OPTIONAL, MultiLangString), **TSP address** (REQUIRED, PhysicalAndElectronicAddress), **TSP information URI** (REQUIRED, MultiLangPointer), **TSP information extensions** (OPTIONAL, Extensions), **List of services** (REQUIRED, at least one service).

### 5.5 Service information
- **Service type identifier** (REQUIRED, URI from clause D.2 or registered), **Service name** (REQUIRED, MultiLangString), **Service digital identity** (REQUIRED, may be null or multiple), **Service current status** (REQUIRED, URI: inaccord/expired/suspended/revoked/notinaccord), **Current status starting date and time** (REQUIRED, GeneralizedTime), **Scheme service definition URI** (OPTIONAL), **Service supply points** (OPTIONAL), **TSP service definition URI** (OPTIONAL), **Service information extensions** (OPTIONAL), **Service approval history** (OPTIONAL, but **MUST** be present if historical period >0).

### 5.6 Service approval history information
- Repeats service type, name, digital identity, previous status, starting time, extensions.

### 5.7 Signature
- **Signed TSL** (REQUIRED if signed, otherwise authenticity via TLS with out-of-band fingerprint). Signature **SHALL** cover all fields except signature value. Scheme operator **REQUIRED** to use X.509 certificate for signing.
- **Scheme identification** (REQUIRED, included in signature): X.509 certificate, subject key identifier, certificate identifier, or public key.
- **Signature algorithm identifier** (REQUIRED, included).
- **Signature value** (REQUIRED).

### 5.8 TSL extensions
- **expiredCertsRevocationInfo** (Extension, service level, optional, non-critical): Indicates from when CRL/OCSP include revocation info for expired certificates.
- **additionalServiceInformation** (Extension, service level, optional): Provides additional info (e.g., QC with SSCD, qualifiers).

## Operations
### 6.1 TSL publication
- Repository providers **MUST** support at least HTTP transport; applications **MUST** support HTTPS.
- LDAP transport (clause 6.1.1.1), HTTP transport (6.1.1.2), FTP (6.1.1.3), Email (6.1.1.4).
- MIME types: `application/vnd.etsi.tsl.der`, `application/vnd.etsi.tsl+xml`, `application/pdf`.
- File extensions: `.dtsl`/`.dts` (DER), `.xtsl`/`.xts` (XML).

### 6.2 TSL Signer Certificate
- Recommended: Subject DN matches scheme territory and operator name; KeyUsage digitalSignature or nonRepudiation; ExtendedKeyUsage with id-tsl-kp-tslSigning; SubjectKeyIdentifier present; BasicConstraints CA=false.

### 6.3 TSL Distribution Points
- TSPs may include extension in trust service tokens to locate TSL; **MUST NOT** be marked critical.

## Informative Annexes (Condensed)
- **Annex F (TSL Signing considerations)**: Recommends use of CAdES, XAdES, or PAdES signatures with SigningCertificate attribute. Basic CMS/ESS signatures are sufficient; advanced features optional.
- **Annex G (Management and Policy considerations)**: Covers change management, trust-service identification, amendment response times (within 4 working hours), authentication verification, scheme cessation (final TSL with expired status), user reference guidance, hard-copy reliance, TSL size recommendations.
- **Annex H (Locating and Authenticating a TSL)**: Describes bound, linked, and de-coupled location models. Searching strategies: same-scheme, known scheme, blind (HTML page with TSLTag). Authentication via signature verification; trusting requires policy.
- **Annex I (General TSL usage)**: Examples: EU Trusted Lists, legal evidence, anomalous status checking, cross-certification, hierarchical schemes (Schemes TSL), collections of TSLs, communities, industrial consortia.
- **Annex J (TSL manual/auto field usage)**: Table listing each field and whether it should be human-readable and/or machine-processable.
- **Annex M (Changes since last major version)**: Summarizes changes from v2.1.1 to v3.1.1 and v3.1.1 to v3.1.2 (minor corrections, new fields like DistributionPoints, expiredCertsRevocationInfo, additionalServiceInformation, EU Trusted List annex).
