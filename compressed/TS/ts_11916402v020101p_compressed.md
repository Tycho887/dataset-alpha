# ETSI TS 119 164-2: Electronic Signatures and Infrastructures (ESI); Associated Signature Containers (ASiC) - Testing Conformance and Interoperability; Part 2: Test suites for testing interoperability of ASiC baseline containers
**Source**: ETSI | **Version**: V2.1.1 | **Date**: 2016-06 | **Type**: Normative
**Original**: http://www.etsi.org/standards-search

## Scope (Summary)
Defines test suites to assess interoperability between implementations claiming conformance to ASiC baseline containers as specified in ETSI EN 319 162-1 [1]. Covers four levels (B‑B, B‑T, B‑LT, B‑LTA) for both ASiC‑S and ASiC‑E forms, including augmentation and negative test cases. PKI‑agnostic.

## Definitions and Abbreviations
### Definitions
- **negative test case**: test case either for a container including signature(s) that are not CAdES baseline signatures [i.4] or XAdES baseline signatures [i.5], or for a signature whose validation according to ETSI EN 319 102‑1 [i.3] would not result in TOTAL_PASSED.

### Abbreviations
- **BCS**: Baseline Container Structure
- **BSC**: Baseline Syntactical Conformance
- **STV**: Signature Time‑stamp token Value
- **TC**: Test Case

## 4 ASiC baseline container interoperability test specification overview
Test suites are defined for both forms (ASiC‑S, ASiC‑E) and all four levels (B‑B, B‑T, B‑LT, B‑LTA). Signature‑specific test cases refer to the CAdES test suites [4] and XAdES test suites [5]. Includes positive, negative, and augmentation tests.

## 5 Test suite for testing interoperability of ASiC‑S baseline containers

### 5.1 Introduction to testing ASiC‑S baseline containers
Clause 5 specifies all ASiC‑S test cases per baseline level. Clause 5.2 covers common test cases; clauses 5.3 and 5.4 cover level‑specific test cases.

### 5.2 Test cases common to all ASiC‑S forms and levels
**Prerequisites**: none unless noted in the table.

| TC ID | Description | Pass Criteria | Interop. Level |
|---|---|---|---|
| ASiC‑S/BCS/1 | Test if container ZIP format is correct. | The container content **shall** comply with ISO/IEC 21320‑1 [6] and be successfully extracted. | Baseline Container Structure |
| ASiC‑S/BCS/2 | Verify if the container format is identifiable. | The container extension **shall** be ".asics". | Baseline Container Structure |
| ASiC‑S/BCS/3 | mimetype is set appropriately. | mimetype value **shall** comply with ASiC [1], clauses 4.3.3.1, point 2) b) and A.1. | Baseline Container Structure |
| ASiC‑S/BCS/4 | Test that a signature is present. | A META‑INF folder in the root folder **shall** be present and contain one signature.p7s file (ASiC [1], clause 4.3.3.2 item 3b) or signatures.xml file (ASiC [1], clause 4.3.3.2 item 3c). | Baseline Container Structure |
| ASiC‑S/BCS/5 | Presence of the signed file. | A single file, in addition to the optional mimetype, **shall** be present in the root folder. | Baseline Container Structure |

## 6 Test suite for testing interoperability of ASiC‑E baseline containers

### 6.1 Introduction to testing ASiC‑E baseline containers
Clause 6 specifies all ASiC‑E test cases per baseline level. Clause 6.2 covers common test cases; clauses 6.2 and 6.3 cover level‑specific test cases. Note: only a single baseline container type is defined for ASiC‑E in ASiC [1].

### 6.2 Test cases common to all ASiC‑E forms and levels
**Prerequisites**: none unless noted in the table.

| TC ID | Description | Pass Criteria |
|---|---|---|
| ASiC‑E/BCS/1 | Test if container ZIP format is correct. | The container content **shall** comply with ISO/IEC 21320‑1 [6] and be successfully extracted. |
| ASiC‑E/BCS/2 | Verify if the container format is identifiable. | The container extension **shall** be ".asice". |
| ASiC‑E/BCS/3 | mimetype is set appropriately. | mimetype value **shall** comply with ASiC [1], clauses 4.4.3.1, item 2) and A.1. |
| ASiC‑E/BCS/4 | Test if signature metadata is present. | META‑INF folder in the root folder contains one or more *signatures*.xml file. |
| ASiC‑E/BCS/5 | Presence of the signed file. | One or more files, in addition to the optional mimetype, **shall** be present in the container outside of the META‑INF folder. |

### 6.3 Test cases for ASiC‑E baseline containers with XAdES signatures

#### 6.3.1 Positive test cases
**Prerequisites**: The test cases in clause 6.2 are a prerequisite.

**Container syntactical conformance test cases** (Table 7):

| TC ID | Description | Pass Criteria |
|---|---|---|
| ASiC‑E/BSC/X1 | Test if XAdES signature files are conformant. | META‑INF folder in the root folder contains one or more *signatures*.xml whose content **shall** be compliant with ASiC [1], clause 4.4.3.2, item 2 and item 3a. |
| ASiC‑E/BSC/X2 | Test if data objects referenced in XAdES signature files are present. | XAdES signature present in one or more *signatures*.xml files **shall** reference only data objects present in the container. |
| ASiC‑E/BSC/X3 | Test if manifest.xml is conformant. | META‑INF/manifest.xml **shall** be present and compliant with ASiC [1], clause 4.4.3.2, item 5b. |
| ASiC‑E/BSC/X4 | Test if mime types in manifest.xml are coherent with referenced objects. | All media types associated to files referenced by manifest.xml **shall** be coherent with the media type of the referenced objects. |

*These test cases shall be repeated with at least one container with a single signature file and one with more than one.*

**Signature test cases** (Table 8): Containers used shall include:
- one or more XAdES signatures signing one file present in the container, and/or
- one or more XAdES signatures signing two files present in the container.

Different XAdES signatures in the same container shall be of the same baseline level and created according to [5] using signatures as described (single or two data objects) as per the requirements in Table 8.

| TC ID | Description | Pass Criteria |
|---|---|---|
| ASiC‑E/XBB | Tests signatures in one or more ASiC‑E containers with XAdES created according to XAdES/BB/* test cases in [5], clause 5. | All XAdES/BB/* test cases in ASiC‑E/XBB **shall** be passed according to criteria in [5], clause 5. |
| ASiC‑E/XBT | Tests signatures in one or more ASiC‑E containers with XAdES created according to XAdES/BT/* test cases in [5], clause 6. | All XAdES/BT/* test cases in ASiC‑E/XBT **shall** be passed according to criteria in [5], clause 6. |
| ASiC‑E/XBLT | Tests signatures in one or more ASiC‑E containers with XAdES created according to XAdES/BLT/* test cases in [5], clause 7. | All XAdES/BLT/* test cases in ASiC‑E/XBLT **shall** be passed according to criteria in [5], clause 7. |
| ASiC‑E/XBLTA | Tests signatures in one or more ASiC‑E containers with XAdES created according to XAdES/BLTA/* test cases in [5], clause 8. | All XAdES/BLTA/* test cases in ASiC‑E/XBLTA **shall** be passed according to criteria in [5], clause 8. |

For each level, one or more containers shall include all signatures needed for full coverage of all test cases defined for that level.

## 7 Test‑suite for augmentation of ASiC baseline containers
Augmentation test cases shall be performed according to [4] clause 9 (CAdES) or [5] clause 9 (XAdES). ASiC baseline containers conformant to B‑T level (generated for ASiC‑S/CBT, ASiC‑S/XBT, or ASiC‑E/XBT) shall be augmented to B‑LTA level baseline signatures per the test cases specified in [4] clause 9 table 5 for CAdES or [5] clause 9 table 6 for XAdES.
