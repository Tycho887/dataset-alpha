# ETSI TS 119 172-2 V1.1.1: Electronic Signatures and Infrastructures (ESI); Signature Policies; Part 2: XML format for signature policies
**Source**: ETSI | **Version**: V1.1.1 | **Date**: December 2019 | **Type**: Technical Specification (Normative)
**Original**: Available at http://www.etsi.org/standards-search (DTS/ESI-0019172-2)

## Scope (Summary)
This document defines an XML format for machine-readable signature policies based on the building blocks of technical constraints on digital signatures specified in ETSI TS 119 172‑1. It specifies semantics and XML syntax for each element, covering signature creation, validation, and augmentation constraints. Pure signature applicability rules are out of scope. Integrity protection of the policy document is not addressed.

## Definitions and Abbreviations
### Terms
- **EU qualified certificate**: qualified certificate as specified in Directive 1999/93/EC or Regulation (EU) No 910/2014
- **EU qualified trust service provider**: trust service provider meeting the qualified requirements of Regulation (EU) 910/2014
- **signature applicability rules**: rules defining requirements for determination of fitness for a business/legal purpose
- **signature augmentation constraint**: criteria used when augmenting a digital signature
- **signature augmentation policy**: set of signature augmentation constraints
- **signature creation application**: application that creates the AdES digital signature using a signature creation device
- **signature creation constraint**: criteria used when creating a digital signature
- **signature creation policy**: set of signature creation constraints processed by the signature creation application
- **signature validation application**: application that validates a signature against a validation policy and outputs status/report
- **signature validation constraint**: technical criteria for validating a digital signature
- **signature validation policy**: set of signature validation constraints processed by the signature validation application

### Abbreviations
BSP, CA, CRL, EUMS, IETF, LOTL, OCSP, OID, RFC, TL, TSP, URI, URN, XML

## XML Syntax for Machine Processable Signature Policy Document

### 4.1 Introduction
#### 4.1.1 Technical approach
The document specifies an XML format for the building blocks of ETSI TS 119 172‑1 that define technical constraints (not applicability rules). XML elements signal whether constraints apply to generation, validation, augmentation, or any combination.

#### 4.1.2 XML namespaces
URIs used:
- `http://uri.etsi.org/19172/v1.1.1`
- `http://www.w3.org/2000/09/xmldsig#`
- `http://uri.etsi.org/02231/v2#`
- `http://www.w3.org/2001/XMLSchema`

One XML schema file `19172xmlSchema.xsd` (see clause B.1). Table 1 maps prefixes: ds, xs, tsl.

#### 4.1.3 AnyType type
The AnyType type **shall** have a content model allowing a sequence of arbitrary XML elements (mixed with text) of unrestricted length. It **shall** also allow text content only and **shall** allow an unrestricted number of arbitrary attributes.

### 4.2 The SignaturePolicy element
- **Semantics**: This element **shall** contain a digest (clause 4.3) and a PolicyComponents element (clause 4.4).
- **Syntax**: Defined in XSD as `SignaturePolicyType` with sequence of `Digest` and `PolicyComponents`.

### 4.3 The Digest element
- **Semantics**: This element **shall** contain one digest algorithm identifier, one digest value, and (for XML) a canonicalization method.
- **Syntax**: `DigestDetailsType` with `ds:DigestMethod`, `ds:DigestValue`, `ds:CanonicalizationMethod`. The digest value **shall** be computed on the canonicalized `PolicyComponents` element.

### 4.4 The PolicyComponents element
- **Semantics**: This element **shall** contain a `PolicyRules` element (clause 4.14) and **should** contain a `GeneralDetails` element (clause 4.5).
- **Syntax**: `PolicyComponentsType` with `GeneralDetails` and `PolicyRules`.

### 4.5 The GeneralDetails element
- **Semantics**: This element **shall** contain `SigPolicyDetails`. It **may** contain `AuthorityDetails` and `OtherDetails`.
- **Syntax**: `GeneralDetailsType` with `SigPolicyDetails`, optional `AuthorityDetails`, optional `OtherDetails`.

### 4.6 The SigPolicyDetails element
- **Semantics**: This element **shall** contain a `PolicyIdentifier` (URI) and a `PolicyName` (multilingual). It **may** contain `DistributionPoints`.
- **Syntax**: `SigPolicyDetailsType`. If OID, encode as URN per RFC 3061.

### 4.7 The AuthorityDetails element
- **Semantics**: This element **shall** contain at least one of `Name` or `TradeName`, and **shall** contain `PostalAddresses` and `ElectronicAddresses`. It **may** contain `ContactPersons`.
- **Syntax**: `AuthorityDetailsType` with optional `Name`, optional `TradeName`, mandatory `PostalAddresses`, mandatory `ElectronicAddresses`, optional `ContactPersons`.

### 4.8 The Name element
- **Semantics**: **Shall** meet clause 5.3.4 of ETSI TS 119 612.

### 4.9 The TradeName element
- **Semantics**: **Shall** meet clause 5.4.2 of ETSI TS 119 612.

### 4.10 The PostalAddresses element
- **Semantics**: **Shall** meet clause 5.3.5.1 of ETSI TS 119 612.

### 4.11 The ElectronicAddresses element
- **Semantics**: **Shall** meet clause 5.3.5.2 of ETSI TS 119 612.

### 4.12 The ContactPersons element
- **Semantics**: **Shall** include one or more contact persons. For each: one name, one or more electronic addresses, one or more phone numbers (starting with '+' and country prefix).
- **Syntax**: `ContactPersonsListType`, each `ContactPerson` has `Name`, `ElectronicAddresses`, `PhoneNumbers` (list of strings).

### 4.13 The OtherDetails element
- **Semantics**: **Shall** contain date and time of issue. **May** contain signing period (NotBefore / optional NotAfter) and other general details.
- **Syntax**: `OtherDetailsType` with mandatory `DateOfIssue`, optional `SigningPeriod`, optional unbounded `Other` (AnyType).

### 4.14 The PolicyRules element
- **Semantics**: This element **shall** contain all rules. It **shall** contain at least one of: CommitmentRules (cl.4.15), DataToBeSignedRules (4.16), SigToDTBSRelationRules (4.17), AugmentationRules (4.21), TimeEvidencesRules (4.25), SigningCertRules (4.24), SignerAttributesConstraints (4.26), QualifyingPropertiesRules (4.27), SCDLoARules (4.28), CryptoSuitesRules (4.29), or OtherRule. Each child element **may** have a `RecommendedScope` attribute (Generation, Validation, Augmentation). Absence of scope means applicable to any scope.
- **Syntax**: `PolicyRulesListType` with unbounded `PolicyRule`. `PolicyRule` is either `CommitmentRules` or one of the `BasicRule` group elements.

### 4.15 The CommitmentRules element
- **Semantics**: **Shall** contain a sequence of `ComitmentRulesComponent`. Each **shall** contain a non-empty list of `Commitment` identifiers, one or more `BasicRule` instances, and a `MatchingIndicator` (All, None, AtLeastOne).
- **Syntax**: `CommitmentRulesType` with unbounded `ComitmentRulesComponent`.

### 4.16 The DataToBeSignedRules element
- **Semantics**: **Shall** contain a sequence of 1 or 2 components: `AnyOfMimeType` list and/or `NoneOfMimeType` list. Must not be empty.
- **Syntax**: `DataToBeSignedRulesType` with optional `AnyOfMimeType` and optional `NoneOfMimeType` (both `StringListType`).

### 4.17 The SigToDTBSRelationRules element
- **Semantics**: **Shall** contain `DTBSCardinality` (cl.4.18). **May** contain `SigDTBSRelativePosition` (4.19) and `SigFormatsAndLevels` (4.20).
- **Syntax**: `SigToDTBSRelationRulesType` with `DTBSCardinality`, optional `SigDTBSRelativePosition`, optional `SigFormatsAndLevels`.

### 4.18 The DTBSCardinality element
- **Semantics**: **Shall** contain at least one of `MaximumValue` or `MinimumValue`, each with a qualifier attribute specifying comparison (LessThan, LessOrEqualTo, Equal for max; HigherThan, HigherOrEqualTo, Equal for min).
- **Syntax**: `CardinalityType` with optional `MaximumValue` (qualifier required) and optional `MinimumValue` (qualifier required).

### 4.19 The SigDTBSRelativePosition element
- **Semantics**: Indicates allowable relative positions: EnvelopingSig, EnvelopedSig, DetachedSig, or ASiC. If ASiC present, no other values allowed. Absence means any position.
- **Syntax**: `SigDTBSRelativePositionType` is a list of `RelativePositionValuesType` strings.

### 4.20 The SigFormatsAndLevels element
- **Semantics**: **Shall** include `SigFormats` (list of Format URIs) or `SigLevels` (list of Level URIs), or both. Absence of `SigFormats` means any format allowed; absence of `SigLevels` means any level allowed. The element must not be empty.
- **Syntax**: `SigFormatsAndLevelsType` with optional `SigFormats` and optional unbounded `SigLevels`.

### 4.21 The AugmentationRules element
- **Semantics**: **Shall** contain `PreviousValidationRequired` (boolean) and `LevelId` (URI with Qualifier: ThisLevel, MinimumLevel, MaximumLevel).
- **Syntax**: `AugmentationRulesType` with `PreviousValidationRequired` and `LevelId`.

### 4.22 Types for defining constraints on certificates' trust
#### 4.22.2 TrustAnchorsListType
- **Semantics**: **Shall** contain a list of trust anchors. Each anchor may be a self-signed X.509 certificate (with optional `reliableUntil`) or a reference to a Trusted List (TL or LOTL) with optional service type and status filters.
- **Syntax**: `TrustAnchorListType` with unbounded choice of `X509CertificateBased` (Base64BinaryAndTimeType) or `TAsInTrustedList` (TrustAnchorsInTLType).

#### 4.22.3 NameConstraintsType
- **Semantics**: **Shall** specify permitted and excluded subtrees. Each subtree defined by `Base` (GeneralName per RFC 5280, base64-encoded DER), `Minimum` (default 0), optional `Maximum`. Constraints apply only when name form is present.
- **Syntax**: `NameConstraintsType` with optional `PermittedSubtrees` and `ExcludedSubtrees`.

#### 4.22.4 PolicyConstraintsType
- **Semantics**: **Shall** contain at least one of `RequireExplicitPolicy` or `InhibitExplicitPolicy` (integer values indicating number of additional certificates).
- **Syntax**: `PolicyConstraintsType` with optional `RequireExplicitPolicy` and `InhibitExplicitPolicy`.

#### 4.22.5 CertificateTrustTreesType
- **Semantics**: Instances **shall** contain `TrustAnchors` and may contain `PathLenConstraint`, `AcceptablePolicySet`, `NameConstraints`, `PolicyConstraints`, `UseCertPath` (explicit path or `AsInSignature`). If present, UseCertPath takes precedence.
- **Syntax**: `CertificateTrustTreesType` with unbounded `CertificateTrustPoint`. Each `CertificateTrustPoint` has mandatory `TrustAnchors`, optional others.

### 4.23 Types for defining constraints on certificates' revocation status
#### 4.23.2 CertificateRevReqType
- **Semantics**: **Shall** contain `EndRevReq` and `CACerts`, each with revocation requirement: clrcheck, ocspcheck, bothcheck, eithercheck, nocheck, or other.
- **Syntax**: `CertificateRevReqType` with `EndRevReq` and `CACerts` of type `RevocationReqType`.

#### 4.23.3 CertificateRevTrustType
- **Semantics**: **Shall** contain `CertificateRevReq`. **May** contain `Freshness` (MaxDifferenceRevocationAndValidation or TimeAfterSignature) and `SigCertIssuedByCAKeepsExpiredRevokedCertsInfo` (duration).
- **Syntax**: `CertificateRevTrustType` with `CertificateRevReq`, optional `Freshness`, optional `SigCertIssuedByCAKeepsExpiredRevokedCertsInfo`.

### 4.24 The SigningCertRules element
- **Semantics**: **Shall** include `SigningCertTrustConditions`. **May** include `MandatedSigningCertInfo` (signingCertOnly or fullPath).
- **Syntax**: `SigningCertRulesType` with optional `MandatedSigningCertInfo` and mandatory `SigningCertTrustConditions`.

### 4.25 The TimeEvidencesRules element
- **Semantics**: **Shall** contain a sequence of `RulesForSetOfEvidences` tuples. Each tuple **shall** have `EvidenceIdentifiers` (list of URIs), `LevelOfAssurance` (URI). **May** have `TimeStampTrustCondition` with optional `TstCertificateTrustTrees`, `TstRevocationTrust`, `TstNameConstraints`, `SignatureTimeStampDelay` (DeltaTimeType).
- **Syntax**: `TimeEvidencesRulesType` with unbounded `RulesForSetOfEvidences`.

### 4.26 The SignerAttributesConstraints element
- **Semantics**: Choice between `NoSignerAttributesAllowed` (signatures shall not include signer attributes) or a list of `ConstraintsOnOneSetOfAttributes`. Each such constraint **shall** have `HowCertAttribute` (ClaimedAttribute, CertifiedAttribute, SignedAssertion, Any). **May** include `AttrCertTrustTrees`, `AttributeRevocationTrust`, and `AttributeConstraints` (each with mandatory `AttributeIdMustBePresent` and optional `AttributeValueConstraint`). If no `AttributeIdMustBePresent`, constraints apply to any attribute.
- **Syntax**: `SignerAttributesConstraintsType` as choice.

### 4.27 The QualifyingPropertiesRules element
- **Semantics**: **Shall** contain a sequence of `LevelRules`. Each **may** have `LevelIdentifier`. **Shall** have `SignedQualifyingProperties` and/or `UnsignedQualifyingProperties` (each of type `QualifyingPropertiesListType`). Each property or choice has a `presenceLevel` attribute (Mandatory or Optional).
- **Syntax**: `QualifyingPropertiesRulesType` with unbounded `LevelQualifyingPropertiesRulesType`.

### 4.28 The SCDLoARules element
- **Semantics**: **Shall** indicate the Level of Assurance of the signature creation device via a URI.
- **Syntax**: `SCDLoARules` element of type `xs:anyURI`.

### 4.29 The CryptoSuitesRules element
- **Semantics**: **Shall** contain a non-empty sequence of `AlgConstraints`. Each **shall** have `AlgId` (URI), `ExpirationDate`, and optionally `Usages` (list of usage URIs), `MinKeyLength`, `MinHashLength`, `Other`. If `Usages` absent, algorithm allowed for all usages.
- **Syntax**: `CryptoSuiteRulesType` with unbounded `AlgConstraints`.
