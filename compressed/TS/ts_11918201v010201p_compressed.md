# ETSI TS 119 182-1: Electronic Signatures and Trust Infrastructures (ESI); JAdES digital signatures; Part 1: Building blocks and JAdES baseline signatures
**Source**: ETSI TC ESI | **Version**: V1.2.1 | **Date**: 2024-07 | **Type**: Technical Specification (Normative)
**Original**: https://www.etsi.org/deliver/etsi_ts/119100_119199/11918201/01.02.01_60/ts_11918201v010201p.pdf

## Scope (Summary)
This document specifies a JSON format for AdES signatures (JAdES) built on JSON Web Signatures (JWS) per IETF RFC 7515. It extends JWS with new header parameters (signed and unsigned) to provide features equivalent to CAdES and XAdES for long-term validity, and defines four baseline signature levels (B-B, B-T, B-LT, B-LTA) with incremental requirements.

## Definitions and Abbreviations
- **electronic time-stamp**: Data binding other electronic data to a particular time, establishing existence evidence (per RFC 3161/RFC 5816; not bound to format).
- **JAdES signature**: JSON Web Signature meeting requirements of this multi-part document.
- **JWS Signature Value**: Digital signature cryptographic value calculated over JWS Protected Header and data to be signed.
- **JSON Web Signature (JWS)**: JSON data structure for digitally signed message per RFC 7515.
- **JAdES component**: Any constituent of a JAdES signature (header parameter, JSON object member, array element, or other JSON value).
- Abbreviations: ASCII, ASN.1, CA, CRL, FIPS, HTTP, IETF, ISO, ITU-T, JOSE, JSON, JWS, OCSP, OID, PKI, RFC, SAML, SHA, SIM, SPO, URI, URL, URN, UTC.

## General Requirements (Clause 4)
- JAdES components **shall** be carried within the JOSE header per IETF RFC 7515.
- All signed header parameters (Clause 5.2) and profiled ones (Clause 5.1), if required, **shall** be in the JWS Protected Header.
- JAdES signatures **may** use JWS Compact Serialization or JWS JSON Serialization.
- JWS Unprotected Header **shall** contain only the `etsiU` header parameter (JSON array).
- If JWS Unprotected Header is present, JWS JSON Serialization **shall** be used.
- Header parameters from RFC 7515/RFC 7797 not further profiled **may** be added.
- JWS Payload **may** be attached or detached; detached payload **may** be one or concatenated objects (see `sigD`).

## Header Parameters Semantics and Syntax (Clause 5)

### 5.1 Use of Header Parameters from IETF RFC 7515, RFC 7797, RFC 7519
- **alg**: signed header parameter; semantics/syntax per RFC 7515 clause 4.1.1; algorithm **should** be from ETSI TS 119 312; identifier **shall** be from IANA JOSE Algorithms registry.
- **cty**: signed header parameter qualifying JWS Payload; **should not** be present if `sigD` is present or content type implied; **shall not** be present if payload is a (counter-signed) signature.
- **kid**: signed header parameter; semantics per RFC 7515; content **should** be base64 of DER-encoded IssuerSerial (RFC 5035); used as hint.
- **x5u**: signed header parameter; semantics per RFC 7515 clause 4.1.5; used as hint.
- **x5t**: **shall not** be present in JAdES signatures.
- **x5t#S256**: signed header parameter; semantics/syntax per RFC 7515 clause 4.1.8; JAdES signature **shall** have at least one of: `x5t#S256`, `x5c`, `sigX5ts`, or `x5t#o` in JWS Protected Header.
- **x5c**: signed header parameter; semantics/syntax per RFC 7515 clause 4.1.6.
- **crit**: signed header parameter; semantics/syntax per RFC 7515 clause 4.1.11; if `sigD` is present, `crit` **shall** also be present with `"sigD"` as element.
- **b64**: signed header parameter; semantics/syntax per RFC 7797 clause 3; if `sigD` with `"http://uri.etsi.org/19182/HttpHeaders"`, then `b64` **shall** be present and set to `"false"`.
- **iat**: signed header parameter; value **shall** be the claimed signing time; syntax per RFC 7519 clause 4.1.6 (integer, NumericDate, no fractions). Before 2025-07-15T00:00:00Z **should** be used instead of `sigT`; starting at that date **shall** be used.

### 5.2 New Signed Header Parameters
- **sigT** (claimed signing time): signed header parameter; string formatted per RFC 3339, UTC, no fractions. Before 2025-07-15T00:00:00Z **should not** be used; after **shall not** be used. Validation apps **shall** parse and process it.
- **x5t#o** (X509 certificate digest): signed header parameter; object with `digAlg` (string, identifier from IANA Named Information Hash Algorithm Registry) and `digVal` (base64url of DER-encoded certificate digest); `digAlg` **shall not** be SHA-256.
- **sigX5ts** (X509 certificates digests): signed header parameter; array of `x5t#o` objects; minItems 2; first reference **shall** be the signing certificate.
- **srCms** (signer commitments): signed header parameter; array of objects; each contains `commId` (oId) and optionally `commQuals` (array of qualifiers).
- **sigPl** (signature production place): signed header parameter; object with properties from schema.org PostalAddress; `addressCountry` **may** be name or ISO 3166-1 alpha-2 code.
- **srAts** (signer attributes): signed header parameter; object with `certified`, `claimed`, `signedAssertions` arrays; `certifiedAttrs` **shall** be base64 of DER-encoded X.509 attribute cert or other attribute cert; `signedAssertions`/`claimed` are `qArrays` type (JSON array of objects with `mediaType`, `encoding`, `qVals`).
- **adoTst** (signed data time-stamp): signed header parameter; `tstContainer`; message imprint input depends on presence of `sigD` and `b64`; **shall not** contain `canonAlg`.
- **sigPId** (signature policy identifier): signed header parameter; object with `id` (oId), optional `digAlg`, `digVal`, `digPSp`, `sigPQuals`; qualifiers: `spURI`, `spUserNotice`, `spDSpec`.
- **sigD** (detached data signing): signed header parameter; object with `mId` (URI), `pars` (array of strings), optional `hashM`, `hashV`, `ctys`. Mechanisms: `HttpHeaders`, `ObjectIdByURI`, `ObjectIdByURIHash`.

### 5.3 New Unsigned Header Parameter
- **etsiU**: JSON array in JWS Unprotected Header; elements **shall** be either clear (clear JSON incorporation) or base64url-encoded; all same type; **shall** be the only header parameter in JWS Unprotected Header; members: `sigPSt`, `sigTst`, `xVals`, `rVals`, `axVals`, `arVals`, `anyValData`, `tstVD`, `arcTst`, `xRefs`, `rRefs`, `axRefs`, `arRefs`, `sigRTst`, `rfsTst`, `cSig`.
- **cSig** (counter signature): JSON object containing a JWS (may be JAdES) signing the JWS Signature Value of the embedding signature.
- **sigPSt** (signature policy store): JSON object with `sigPolDoc` (base64url) or `sigPolLocalURI` (URI), optional `spDSpec` (oId).
- **sigTst** (signature time-stamp): `tstContainer` time-stamping base64url-encoded JWS Signature Value; **shall not** contain `canonAlg`.
- **xVals**, **rVals**, **axVals**, **arVals**: arrays/objects for certificates and revocation data; see syntax and semantics in original.
- **anyValData**: `validationVals` type; may contain `xVals` and/or `rVals` for any signature present.
- **tstVD** (time-stamp validation data): `validationVals` for electronic time-stamps.
- **arcTst** (archive time-stamp): `tstContainer` time-stamping JWS Payload, JWS Protected Header, JAdES Signature Value, and `etsiU` array at generation time. Generation steps in clause 5.3.6.2.2.

### 5.4 Generally Useful Syntax
- **oId**: object with `id` (URI/URN for OID), optional `desc`, `docRefs`.
- **pkiOb**: object with `val` (base64), optional `encoding` (URI), `specRef`; for PKI objects.
- **tstContainer**: object with `canonAlg` (optional URI), `tstTokens` (non-empty array of `tstToken`); `tstToken` has `val` (base64) and optional `type`, `encoding`, `specRef`.

## JAdES Baseline Signatures (Clause 6)
- Four levels: B-B, B-T, B-LT, B-LTA.
- Algorithm requirements: per ETSI TS 119 312; MD5 **shall not** be used.
- Table 1 defines presence and cardinality of header parameters and services per level.

### Requirements Summary (Table 1 – Key Normative Statements)
- **alg**: shall be present at all levels (card. 1).
- **cty**: conditioned presence (0 or 1).
- **kid**, **x5u**: may be present (0 or 1).
- **x5c**: conditioned presence (0 or 1).
- **crit**: conditioned presence.
- **Service: claimed signing time**: shall be provided; via `iat` (conditioned) or `sigT` (conditioned per dates).
- **Service: signing certificate reference**: conditioned; via `x5t#S256`, `x5t#o`, or `sigX5ts` (0 or 1 each).
- **sigD**: may be present (0 or 1).
- **srAts**, **srCms**, **sigPl**, **sigPId**: may be present (0 or 1).
- **cSig**: may be present (≥0).
- **adoTst**: may be present (0 or 1).
- **sigPSt**: conditioned (0 or 1; only if `sigPId` with `digVal` present).
- **sigTst**: * at B-B; shall be present at B-T, B-LT, B-LTA (≥1 at latter).
- **xVals**, **axVals**: * at B-B/B-T; conditioned at B-LT/B-LTA (0 or 1).
- **anyValData**: * at B-B/B-T; conditioned at B-LT/B-LTA (≥0).
- **xRefs, axRefs, rRefs, arRefs, sigRTst, rfsTst**: * at B-B/B-T; shall not be present at B-LT/B-LTA.
- **rVals, arVals**: * at B-B/B-T; conditioned at B-LT/B-LTA (0 or 1).
- **Service: validation data for time-stamps**: * at B-B/B-T; shall be provided at B-LT/B-LTA via `tstVD`, embedded, or `anyValData`.
- **arcTst**: * at B-B/B-T/B-LT; shall be present at B-LTA (1).

Additional requirements (a–m) in Table 1 notes: e.g., `iat` shall be used after 2025-07-15; `sigPSt` only if `sigPId` with digest; `arcTst` may contain multiple time-stamps; before generating new `arcTst`, include all necessary validation material.

## Informative Annexes (Condensed)
- **Annex C (Correspondence between XAdES and JAdES tags)**: Maps XAdES qualifying property tags (e.g., SigningTime, SigningCertificateV2) to JAdES header parameters/components (e.g., `iat`/`sigT`, `x5t#o`).
- **Annex E (IANA Considerations)**: Registers all JAdES-specific header parameters (e.g., `x5t#o`, `sigX5ts`, `srCms`, `sigPl`, `srAts`, `adoTst`, `sigPId`, `sigD`, `etsiU`, `cSig`, `sigPSt`, `sigTst`, `xVals`, `rVals`, `axVals`, `arVals`, `anyValData`, `tstVD`, `arcTst`, `xRefs`, `rRefs`, `axRefs`, `arRefs`, `sigRTst`, `rfsTst`) in the IANA JSON Web Signature and Encryption Header Parameters registry.
- **Annex F (Change history)**: Documents revisions from v1.1.1 to v1.2.1, including replacement of `sigT` with `iat`, additions of `anyValData`, refinements of `arcTst` generation, and editorial fixes.

## Normative Annexes
- **Annex A (Additional components Specification)**: Defines `xRefs`, `rRefs`, `axRefs`, `arRefs`, `sigRTst`, `rfsTst` with semantics and syntax for references to validation data.
- **Annex B (JSON Schema files)**: Points to JSON Schema files location for JAdES components and related RFC schemas.
- **Annex D (Alternative mechanisms for long term availability and integrity of validation data)**: Requires that if unsigned components are used for alternative mechanisms, their semantics, syntax, and protection strategy must be specified (example: RFC 4998).
