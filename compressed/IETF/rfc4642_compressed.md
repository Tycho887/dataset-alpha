# RFC 4642: Using Transport Layer Security (TLS) with Network News Transfer Protocol (NNTP)
**Source**: IETF | **Version**: Standards Track | **Date**: October 2006 | **Type**: Normative
**Original**: https://tools.ietf.org/html/rfc4642

## Scope (Summary)
This document specifies the STARTTLS extension for NNTP that enables TLS negotiation to provide encryption, data integrity, optional certificate-based peer authentication, and optional data compression for single-link confidentiality. It formalizes the STARTTLS command and deprecates the use of a separate TLS port (563) in favor of this command.

## The STARTTLS Extension
### Advertising the STARTTLS Extension
- **Capability Label**: "STARTTLS" MUST be advertised in CAPABILITIES response, but MUST NOT be advertised once TLS layer is active or after successful authentication.
- Cached CAPABILITIES results from a previous session MUST NOT be relied on.
- MAY be advertised before and after MODE READER command.

### STARTTLS Command
#### Usage
- **Syntax**: `STARTTLS`
- **Responses**:
  - `382 Continue with TLS negotiation`
  - `502 Command unavailable` (if TLS already active or authentication occurred)
  - `580 Can not initiate TLS negotiation`
- **NOTES**:
  - This command MUST NOT be pipelined.
  - Server MUST NOT return 480 or 483 in response to STARTTLS.

#### Description
- **Client MAY** issue STARTTLS at any time after receiving a 483 response, or unsolicited if server advertises STARTTLS.
- **Server SHOULD** accept unsolicited TLS negotiation requests.
- **Server MUST** reject with 580 if unable to initiate TLS negotiation.
- Upon 382, client MUST start TLS negotiation immediately; failure to do so SHOULD result in immediate connection close.
- Servers MUST understand backwards-compatible TLS Client Hello messages (client_version >= TLS 1.0); clients MAY use backwards-compatible Client Hello.
- TLS Server Name Indication (`server_name`) SHOULD be implemented by clients and by servers known by multiple hostnames.
- If TLS negotiation fails, both client and server SHOULD close the connection immediately.
- On successful TLS handshake, NNTP protocol resets to state after initial greeting response, except MODE READER effects are not reversed. Server MUST discard pre-TLS knowledge; client SHOULD discard and MUST NOT rely on pre-TLS knowledge.
- Server remains in non-authenticated state; AUTHINFO SASL EXTERNAL MAY be used for authentication, but is not required.
- **MUST NOT** start TLS if already active; server MUST return 502. Client MUST NOT issue MODE READER while TLS active; server MUST NOT advertise MODE-READER capability.
- CAPABILITIES list after TLS MAY differ from before.

- (Preserved in original, but in compression can be summarized if needed. However, examples are informative and may be condensed. Since they are important for understanding, keep them as brief examples with key points.)
  - Example 1: Client prompted to use encryption, successful negotiation, then STARTTLS rejected because TLS already active.

  - Example 3: TLS negotiation fails; server responds with 483 or 400 for restricted commands.

## Security Considerations (Condensed)
- TLS protocol security considerations and NNTP Section 12 apply.
- Before TLS handshake, interactions are in cleartext; sensitive knowledge obtained prior MUST be discarded upon establishing security layer.
- CAPABILITIES SHOULD be re-issued after TLS.
- NNTP is not end-to-end; TLS only secures one link.
- Man-in-the-middle attacks can delete STARTTLS capability; clients should record TLS offering and alarm if missing.
- If TLS negotiation fails, client may continue, retry later, or abandon.

## IANA Considerations (Condensed)
- Extension name: STARTTLS
- Capability label: "STARTTLS" (no arguments)
- Defines command STARTTLS (Section 2.2)
- Affects overall behavior: after successful STARTTLS, all communication uses TLS.
- STARTTLS cannot be pipelined.
- Once TLS active, STARTTLS and MODE-READER capabilities not advertised.
- No new responses for pre-existing commands.
