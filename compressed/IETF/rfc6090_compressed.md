# RFC 6090: Fundamental Elliptic Curve Cryptography Algorithms
**Source**: Internet Engineering Task Force (IETF) | **Version**: Informational | **Date**: February 2011 | **Type**: Informational (contains normative text)
**Original**: http://www.rfc-editor.org/info/rfc6090

## Scope (Summary)
This note describes the fundamental algorithms of Elliptic Curve Cryptography (ECC) as defined in seminal references from 1994 and earlier, including Elliptic Curve Diffie-Hellman (ECDH) and elliptic curve versions of the ElGamal signature scheme (KT-IV and KT-I). It only covers elliptic curves over fields of characteristic greater than three, as used in Suite B.

## Definitions and Abbreviations
- **Finite field Fp**: A prime field where p is a prime number; the set Zp = {0,1,...,p-1} with modular arithmetic. Every nonzero element has an inverse. Also denoted GF(p).
- **Elliptic curve group**: The set of points (x,y) in Fp satisfying y^2 = x^3 + a*x + b, plus the point at infinity. The group operation (addition) is defined as in Section 3.
- **Point at infinity (@,@)**: The identity element of the elliptic curve group. Its inverse is itself.
- **Affine coordinates**: Representation of an elliptic curve point as (x,y) in Fp.
- **Homogeneous coordinates**: Representation of a point (x,y) as (X,Y,Z) with x=X/Z mod p, y=Y/Z mod p, and Z ≠ 0. The point at infinity is (0,1,0) in homogeneous coordinates.
- **Order n**: The number of points in the cyclic subgroup generated by a generator g.
- **Cofactor c**: The quotient of the total number of points on the curve divided by the subgroup order n.
- **Compact representation**: Only the x-coordinate of an elliptic curve point is transmitted or used.
- **Compact output**: Only the x-coordinate of the shared secret is used in ECDH.

## 1. Introduction (Informative)
ECC is a public-key technology offering performance advantages at higher security levels. This note aims to provide a normative reference for fundamental algorithms predating specialized optimizations.

## 2. Mathematical Background
### 2.1 Modular Arithmetic
- **Congruence modulo n**: x ≡ y (mod n) if x-y is a multiple of n.
- **Set Zq**: {0,1,...,q-1} closed under modular addition, subtraction, multiplication, and inverse (if coprime).
- **Modular addition**: a+b if a+b<q, else a+b-q.
- **Modular subtraction**: a-b if a-b≥0, else a-b+q.
- **Modular multiplication**: remainder of a*b divided by q.
- **Modular inverse**: For x coprime with q, 1/x mod q computed using extended Euclidean algorithm.

### 2.2 Group Operations
- **Group**: Set G with associative operation * having identity e and inverses.
- **Exponentiation a^N**: Repeated application of * (N-1 times) to a.
- **Additive notation**: * becomes +, a^N becomes N*a.
- **Identity**: e such that e*a = a*e = a.
- **Inverse a^-1**: a*a^-1 = e.
- **Order of element**: smallest positive X such that a^X = e.
- **Cyclic subgroup**: set {a, a^2, ..., a^R=e} generated by a.
- **Generator**: element whose powers generate the whole subgroup.

### 2.3 The Finite Field Fp
- Fp is a finite field of characteristic p. Equations implicitly mod p.

## 3. Elliptic Curve Groups
### 3.1 Affine Coordinates
- **Curve equation**: y^2 = x^3 + a*x + b, with a,b in Fp, discriminant nonzero.
- **Point equality**: (x1,y1) = (x2,y2) iff x1=x2 and y1=y2; point at infinity is equal only to itself.
- **Inverse of (x1,y1)**: (x1,-y1).
- **Group operation**: defined by cases (see Appendix F.1). For distinct points with different x, compute slope; for doubling, use tangent.

### 3.2 Homogeneous Coordinates
- Representation: (X,Y,Z) with x=X/Z, y=Y/Z, Z≠0. Point at infinity is (0,Y,0) for any nonzero Y.
- **Group operation formulas** for addition and doubling (see Appendix F.2). Avoids modular inverse except when converting to affine.

### 3.3 ECC Parameters
- **Parameter set**: prime p, curve coefficients a,b, generator g, order n of subgroup generated by g.
- **Discriminant must be nonzero**: 4*a^3 + 27*b^2 ≠ 0 mod p. (Section 3.3.1)
- **Security**: The order n MUST be divisible by a large prime. (Section 3.3.2)
- **Inferior parameter sets**: b=0 and p≡3 mod 4; a=0 and p≡2 mod 3. These SHOULD NOT be used.

### 3.4 Other Coordinates (Informative)
- Jacobi coordinates and others exist; see [CC1986].

## 4. Elliptic Curve Diffie-Hellman (ECDH)
- **DH protocol**: Parties A and B choose random private keys j,k in [1,n-1]; compute public keys g^j, g^k; shared secret g^(j*k).
- **Data types**: Private key is integer in Zt (t = subgroup order). Public keys and shared secret are group elements.
- **Compact representation** (Section 4.2): Only x-coordinate of public key is transmitted. Compact output uses only x-coordinate of shared secret.
- **Normative rules**:
  - Both parties in a run MUST use the same method regarding compact output.
  - If compact representation is used, then compact output MUST be used.

## 5. Elliptic Curve ElGamal Signatures
### 5.1 Background (Informative)
ElGamal signatures based on discrete log; generalized to elliptic curves.

### 5.2 Hash Functions
- Use collision-resistant hash function h() mapping arbitrary bit strings to non-negative integers.
- Procedure: evaluate hash H(m) to fixed-length bit string, then convert to integer by treating leftmost bit as most significant.

### 5.3 KT-IV Signatures (Koyama-Tsuruoka Type IV)
- **Keypair generation** (5.3.1): Private key z uniformly random in [1,q-1]; public key Y=alpha^z.
- **Signature creation** (5.3.2):
  1. Choose random k in [1,q-1].
  2. Compute R=(r_x,r_y)=alpha^k.
  3. s1 = r_x mod q.
  4. If h(m)+z*s1 ≡ 0 mod q, MUST generate new k and recalculate. Optionally check s1=0; if so, SHOULD recalculate.
  5. s2 = k/(h(m)+z*s1) mod q.
- **Signature verification** (5.3.3):
  1. Verify 0 < s1 < q and 0 < s2 < q; if either fails, SHALL reject signature.
  2. Compute u1 = h(m)*s2 mod q, u2 = s1*s2 mod q.
  3. Compute R' = alpha^u1 * Y^u2.
  4. Accept if x-coordinate of R' mod q equals s1; else reject.

### 5.4 KT-I Signatures (Type I, equivalent to ECDSA)
- **Keypair generation**: Same as KT-IV.
- **Signature creation** (5.4.2):
  1. Choose random k in [1,q-1].
  2. Compute R=(r_x,r_y)=alpha^k.
  3. s1 = r_x mod q.
  4. s2 = (h(m)+z*s1)/k mod q.
  5. Optionally check s1=0 or s2=0; if either zero, SHOULD recalculate.
- **Signature verification** (5.4.3):
  1. Verify 0 < s1 < q and 0 < s2 < q; SHALL reject otherwise.
  2. Compute s2_inv = 1/s2 mod q.
  3. Compute u1 = h(m)*s2_inv mod q, u2 = s1*s2_inv mod q.
  4. Compute R' = alpha^u1 * Y^u2.
  5. Accept if x-coordinate of R' mod q equals s1; else reject.

### 5.5 Converting KT-IV to KT-I
- (s1, s2) as KT-IV signature for message m → (s1, 1/s2 mod q) is KT-I signature for same message. Conversion uses only public information; implementation MAY use this method.

### 5.6 Rationale (Informative)
Background on generalized ElGamal equations, strong equivalence, and security checks.

## 6. Converting between Integers and Octet Strings
### 6.1 Octet-String-to-Integer Conversion
- Let S = S1...Sk be octets (first to last). Integer x = Σ 2^(8(k-i)) Si. (Most significant octet first.)

### 6.2 Integer-to-Octet-String Conversion
- Given integer x, produce octet string S of length k such that the same sum equals x. This method SHOULD be used for representing elliptic curve points/coordinates.

## 7. Interoperability (Informative)
- ECDH can interoperate with IKE (RFC 2409/5996) using ECP groups; with IEEE P1363 and ANSI X9.62 under specific options.
- KT-I is functionally equivalent to ECDSA; signatures are interchangeable with IEEE P1363 and ANSI X9.62 ECDSA.

## 8. Validating an Implementation (Informative)
- Use Known Answer Tests (KATs) from RFC 5903 or RFC 5114 for ECDH; from RFC 4754 for KT-I. Consistency testing recommended.

## 9. Intellectual Property (Informative)
- All normative references for ECDH and KT-I predate 1994. Factual notice; not legal advice.

## 10. Security Considerations (Informative)
- General attacks: Pohlig-Hellman, Shanks, Pollard rho, lambda. No subexponential attacks on general elliptic curves.
- Subgroups: Cofactor should be 1; small subgroup attacks exist if cofactor>1.
- ECDH: not secure against active attacks without authentication of public keys.
- Group representation: Malicious points on different curve can leak key if implementation does not verify curve equation.
- Signatures: Must use collision-resistant hash to prevent existential forgery. Suitable hashes: SHA-256, SHA-384, SHA-512.
