# RFC 2898: PKCS #5: Password-Based Cryptography Specification Version 2.0
**Source**: RSA Laboratories (IETF Informational) | **Version**: 2.0 | **Date**: September 2000 | **Type**: Informational
**Original**: https://tools.ietf.org/html/rfc2898

## Scope (Summary)
This document specifies password-based cryptography techniques: key derivation functions (PBKDF1, PBKDF2), encryption schemes (PBES1, PBES2), and a message authentication scheme (PBMAC1), including ASN.1 syntax for identifying the algorithms. It is intended for protection of sensitive information such as private keys (as in PKCS #8). Password-based key agreement and password selection guidelines are out of scope.

## 3. Overview (Condensed)
Password-based cryptography combines a password with a salt and an iteration count to derive a key. The salt (at least 64 bits) expands the key space; the iteration count (minimum 1000) increases the cost of exhaustive search. Password is an octet string of arbitrary length; it is recommended to follow ASCII or UTF-8 encoding. Key derivation functions (PBKDF1, PBKDF2) are used to produce encryption and authentication keys.

## 4. Salt and Iteration Count

### 4.1 Salt
- **Recommendation (shall)**: If no concern about multiple uses of same key, salt **should** be at least eight octets (64 bits) long and randomly generated.
- **Recommendation (shall)**: Otherwise, salt **shall** contain distinguishing data in addition to a random part of at least eight octets, and this data **shall** be checked or regenerated by the party receiving the salt.
- **Note**: Deterministic alternative: S = KDF(P, M) is not recommended if message M is from a small space.

### 4.2 Iteration Count
- **Recommendation (shall)**: A minimum of 1000 iterations **is recommended**.

## 5. Key Derivation Functions

### 5.1 PBKDF1
- **Input**: P (password), S (salt, eight-octet string), c (iteration count, positive integer), dkLen (intended length, at most 16 for MD2/MD5, 20 for SHA-1).
- **Output**: DK (derived key, dkLen-octet string).
- **Requirement (shall)**: Hash function **shall** be MD2, MD5, or SHA-1.
- **Requirement (shall)**: Step 1: If dkLen > bound, output "derived key too long" and stop.
- **Requirement (shall)**: Step 2: Apply hash for c iterations: T_1 = Hash(P || S); T_i = Hash(T_{i-1}); DK = T_c<0..dkLen-1>.
- **Recommendation**: PBKDF1 is only for compatibility; not recommended for new applications.

### 5.2 PBKDF2
- **Input**: P, S (salt, octet string), c (iteration count), dkLen (positive integer, at most (2^32 - 1) * hLen).
- **Output**: DK (derived key, dkLen-octet string).
- **Requirement (shall)**: If dkLen > (2^32 - 1) * hLen, output "derived key too long" and stop.
- **Requirement (shall)**: Compute l = CEIL(dkLen / hLen), r = dkLen - (l-1)*hLen.
- **Requirement (shall)**: For each block i (1..l): T_i = F(P, S, c, i) where F = U_1 \xor U_2 \xor ... \xor U_c, and U_1 = PRF(P, S || INT(i)), U_j = PRF(P, U_{j-1}).
- **Requirement (shall)**: Concatenate blocks and extract first dkLen octets: DK = T_1 || ... || T_l<0..r-1>.
- **Note**: INT(i) is four-octet big-endian encoding.
- **Recommendation**: PBKDF2 is recommended for new applications.

## 6. Encryption Schemes

### 6.1 PBES1
#### 6.1.1 Encryption Operation
- **Input**: password P, message M.
- **Output**: ciphertext C.
- **Requirement (shall)**: Select eight-octet salt S and iteration count c.
- **Requirement (shall)**: Apply PBKDF1 to P, S, c with dkLen=16 to get DK.
- **Requirement (shall)**: Split DK: K = DK<0..7>, IV = DK<8..15>.
- **Requirement (shall)**: Encode message as EM = M || PS, where PS consists of (8 - (||M|| mod 8)) octets each with that value (RFC 1423 padding).
- **Requirement (shall)**: Encrypt EM with underlying block cipher (DES or RC2) in CBC mode using K and IV. For DES, key considered as 64-bit encoding of 56-bit key with parity ignored. For RC2, effective key bits shall be 64.

#### 6.1.2 Decryption Operation
- **Requirement (shall)**: Obtain S and c.
- **Requirement (shall)**: Apply PBKDF1 to P, S, c, dkLen=16 to get DK.
- **Requirement (shall)**: Split DK into K = DK<0..7>, IV = DK<8..15>.
- **Requirement (shall)**: Decrypt ciphertext with underlying cipher in CBC mode; if length not multiple of 8, output "decryption error" and stop.
- **Requirement (shall)**: Parse EM = M || PS where PS consists of psLen octets each with value psLen (1..8); if not possible, output "decryption error" and stop.
- **Recommendation**: PBES1 only for compatibility; not recommended for new applications.

### 6.2 PBES2
#### 6.2.1 Encryption Operation
- **Requirement (shall)**: Select salt S and iteration count c.
- **Requirement (shall)**: Select dkLen for derived key for underlying encryption scheme.
- **Requirement (shall)**: Apply selected KDF (shall be PBKDF2) to P, S, c, dkLen to produce DK.
- **Requirement (shall)**: Encrypt message M with underlying encryption scheme under DK to produce ciphertext C.
- **Recommendation**: PBES2 is recommended for new applications.

#### 6.2.2 Decryption Operation
- **Requirement (shall)**: Obtain S, c, dkLen.
- **Requirement (shall)**: Apply selected KDF to P, S, c, dkLen to get DK.
- **Requirement (shall)**: Decrypt ciphertext under DK; if error, output "decryption error" and stop.

## 7. Message Authentication Schemes

### 7.1 PBMAC1
#### 7.1.1 MAC Generation
- **Requirement (shall)**: Select salt S and iteration count c.
- **Requirement (shall)**: Select dkLen for underlying MAC.
- **Requirement (shall)**: Apply selected KDF (shall be PBKDF2) to P, S, c, dkLen to get DK.
- **Requirement (shall)**: Process message M with underlying MAC scheme under DK to generate T.

#### 7.1.2 MAC Verification
- **Requirement (shall)**: Obtain S, c, dkLen.
- **Requirement (shall)**: Apply selected KDF to P, S, c, dkLen to get DK.
- **Requirement (shall)**: Process M with underlying MAC scheme under DK to verify T.
- **Requirement (shall)**: If verification successful, output "correct"; else "incorrect".

## 8. Security Considerations (Condensed)
Password-based cryptography is limited against off-line dictionary attacks. Use of salt and iteration count increase attack complexity. Passwords should be selected well (see [17]) and protected if stored. Different keys should be derived for different uses; random salt is sufficient for single algorithm, but structured salt may be necessary to prevent interactions (see Section 4 recommendations).

## Informative Annexes (Condensed)

- **Annex A: ASN.1 Syntax** – Defines ASN.1 structures for key derivation functions (PBKDF2-params), encryption schemes (PBEParameter for PBES1, PBES2-params), and message authentication (PBMAC1-params). Object identifiers are given: id-PBKDF2 (pkcs-5.12), PBES1 OIDs depending on hash and cipher (e.g., pbeWithMD5AndDES-CBC = pkcs-5.3), id-PBES2 (pkcs-5.13), id-PBMAC1 (pkcs-5.14). Default PRF is HMAC-SHA-1.

- **Annex B: Supporting Techniques** – Provides examples of underlying algorithms:
  - **B.1**: Pseudorandom function: HMAC-SHA-1 (OID id-hmacWithSHA1). Note: effective search space for derived key limited to 160 bits due to HMAC structure.
  - **B.2**: Encryption schemes for PBES2: DES-CBC-Pad (OID desCBC), DES-EDE3-CBC-Pad (OID des-EDE3-CBC), RC2-CBC-Pad (OID rc2CBC with RC2-CBC-Parameter), RC5-CBC-Pad (OID rc5-CBC-PAD). Each specifies key size, IV, and padding.
  - **B.3**: Message authentication scheme: HMAC-SHA-1 (same OID as pseudorandom function, distinguished by context).

- **Annex C: ASN.1 Module** – Contains the full ASN.1 module (PKCS5v2-0) with type definitions for all algorithms, supporting extensibility via ALGORITHM-IDENTIFIER class. Includes OID definitions and structure for PBKDF2, PBES1, PBES2, PBMAC1, and supporting techniques (DES, RC2, RC5, HMAC-SHA-1).
