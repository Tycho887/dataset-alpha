# RFC 4853: Cryptographic Message Syntax (CMS) Multiple Signer Clarification
**Source**: IETF | **Version**: Standards Track | **Date**: April 2007 | **Type**: Normative (Update to RFC 3852)
**Original**: https://datatracker.ietf.org/doc/rfc4853/

## Scope (Summary)
This document updates RFC 3852 (CMS) by clarifying the handling of multiple digital signatures in the SignedData content type. It specifies that when multiple signerInfos are present, the successful validation of one signature from a given signer is normally treated as a valid signature by that signer, but alternative rules may be defined by application-specific specifications.

## Definitions and Abbreviations
- **CMS**: Cryptographic Message Syntax
- **SignedData**: CMS protected content type supporting multiple digital signatures
- **SignerInfo**: Per-signer information structure in SignedData

## 1. Introduction (Summarized)
This document clarifies that in CMS SignedData, the protected content is validly signed by a given signer if any of the digital signatures from that signer are valid. This is crucial when recipients support different signature algorithms (e.g., RSA and ECDSA) or during hash-function transitions (e.g., SHA-1 to SHA-256).

## 2. Terminology
- **MUST, MUST NOT, REQUIRED, SHALL, SHALL NOT, SHOULD, SHOULD NOT, RECOMMENDED, MAY, OPTIONAL**: Interpreted as defined in RFC 2119 [STDWORDS].

## 3. Update to RFC 3852, Section 5: Signed-data Content Type
The next-to-last paragraph of RFC 3852 Section 5 is replaced by:

- **R1 (Replacement text)**: A recipient independently computes the message digest. This message digest and the signer's public key are used to verify the signature value. The signer's public key is referenced either by an issuer distinguished name along with an issuer-specific serial number or by a subject key identifier that uniquely identifies the certificate containing the public key. The signer's certificate can be included in the SignedData certificates field.

- **R2 (New text appended after R1)**: When more than one signature is present, the successful validation of one signature associated with a given signer is usually treated as a successful signature by that signer. However, there are some application environments where other rules are needed. An application that employs a rule other than one valid signature for each signer **MUST** specify those rules. Also, where simple matching of the signer identifier is not sufficient to determine whether the signatures were generated by the same signer, the application specification **MUST** describe how to determine which signatures were generated by the same signer. Support of different communities of recipients is the primary reason that signers choose to include more than one signature. For example, the signed-data content type might include signatures generated with the RSA signature algorithm and with the ECDSA signature algorithm. This allows recipients to verify the signature associated with one algorithm or the other.

## 4. Update to RFC 3852, Section 5.1: SignedData Type
The next-to-last paragraph of RFC 3852 Section 5.1 is replaced by:

- **R3 (Replacement text)**: signerInfos is a collection of per-signer information. There MAY be any number of elements in the collection, including zero. When the collection represents more than one signature, the successful validation of one of signature from a given signer ought to be treated as a successful signature by that signer. However, there are some application environments where other rules are needed. The details of the SignerInfo type are discussed in section 5.3. Since each signer can employ a different digital signature technique, and future specifications could update the syntax, all implementations **MUST** gracefully handle unimplemented versions of SignerInfo. Further, since all implementations will not support every possible signature algorithm, all implementations **MUST** gracefully handle unimplemented signature algorithms when they are encountered.

## 5. Security Considerations
The replacement text reduces the likelihood of interoperability errors during transition from MD5 and SHA-1 to stronger one-way hash functions, or to better signature algorithms.
