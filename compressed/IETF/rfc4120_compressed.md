# RFC 4120: The Kerberos Network Authentication Service (V5)
**Source**: IETF Network Working Group | **Version**: 5 | **Date**: July 2005 | **Type**: Normative (Standards Track)
**Original**: https://www.rfc-editor.org/rfc/rfc4120

## Scope (Summary)
This document specifies Version 5 of the Kerberos protocol, a trusted third‑party authentication service for open networks. It obsoletes RFC 1510, clarifying protocol details, message formats, and intended use. The protocol uses symmetric cryptography to verify principal identities without relying on host addresses or physical security.

## Definitions and Abbreviations
- **Authentication**: Verifying the claimed identity of a principal.
- **Authenticator**: Record containing information recently generated using the session key.
- **KDC**: Key Distribution Center – service that supplies tickets and session keys (AS + TGS).
- **Ticket**: Record containing client identity, session key, timestamp, encrypted with server’s key.
- **Session key**: Temporary encryption key generated by the KDC for a client-server interaction.
- **Sub‑session key (subkey)**: Temporary key chosen and exchanged by principals using the session key.
- **Principal**: A named client or server entity.
- **Realm**: Organizational boundary for a KDC domain.
- **Pre‑authentication**: Additional data (e.g., encrypted timestamp) used before ticket issuance.

## 1. Introduction
The Kerberos model is based on Needham‑Schroeder with Denning‑Sacco modifications. The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHALL”, “SHALL NOT”, “SHOULD”, “SHOULD NOT”, “RECOMMENDED”, “MAY”, and “OPTIONAL” are as defined in RFC 2119.

### 1.1 The Kerberos Protocol
- **Process**: Client → AS request → AS responds with ticket + session key encrypted in client’s key. Client sends ticket (encrypted in server’s key) + authenticator to server.
- **Exchanges**: AS exchange (initial TGT), TGS exchange (service ticket), CS exchange (application authentication).
- **Security assumptions**: Denial of service not solved; principals must keep keys secret; passwords are vulnerable to guessing; clocks must be loosely synchronized; principal identifiers not recycled short‑term.

### 1.2 Cross‑Realm Operation
- **Inter‑realm keys** allow authentication across realms. `TRANSITED-POLICY-CHECKED` flag indicates KDC validated transited field. Application server **SHALL** check transited field or reject tickets without `TRANSITED-POLICY-CHECKED`.

### 1.3 Choosing a Principal with Which to Communicate
- **Implementations **MUST NOT** use insecure DNS queries to canonicalize hostnames for service principal names. Application authors **MAY** append a statically configured domain name. Clients **SHOULD** fold user‑entered hostname to lowercase without other modifications.

### 1.4 Authorization
- **Kerberos does not provide authorization**. Possession of a ticket provides only authentication. Applications **MUST** make separate authorization decisions.

### 1.5 Extending Kerberos without Breaking Interoperability
- Extensions use typed holes (PA‑DATA, AuthorizationData). Unknown extensions **SHOULD** be ignored but **MUST NOT** cause request rejection. Unknown authorization data in AP‑REQ or ticket **MUST** cause authentication failure unless encapsulated in `AD-IF-RELEVANT`.

### 1.6 Environmental Assumptions
- Denial of service not solved.
- Principals **MUST** keep secret keys secret.
- Password guessing not solved.
- Host clocks **MUST** be loosely synchronized (5 minutes typical).
- Principal identifiers **MUST NOT** be recycled short‑term.

## 2. Ticket Flag Uses and Requests
- Clients **MUST** ignore unrecognized ticket flags. KDCs **MUST** ignore unrecognized KDC options. Clients **MUST** confirm that returned ticket meets needs.

### 2.1 Initial, Pre‑authenticated, and Hardware‑Authenticated Tickets
- `INITIAL`: ticket issued via AS protocol.
- `PRE-AUTHENT`, `HW-AUTHENT`: indicate pre‑authentication method used.

### 2.2 Invalid Tickets
- `INVALID` flag: application servers **MUST** reject such tickets. Validated by TGS with `VALIDATE` option after starttime.

### 2.3 Renewable Tickets
- `RENEWABLE`: ticket can be renewed before expiration; KDC **MAY** check hot‑list. Renew‑till defines absolute expiration.

### 2.4 Postdated Tickets
- `MAY-POSTDATE` flag in TGT allows postdated tickets. `POSTDATED` flag set; `INVALID` flag also set. **MUST** be validated before use.

### 2.5 Proxiable and Proxy Tickets
- `PROXIABLE` allows proxy tickets with different network addresses. `PROXY` flag set in proxy tickets.

### 2.6 Forwardable Tickets
- `FORWARDABLE` allows TGT forwarding with different addresses. `FORWARDED` flag set.

### 2.7 Transited Policy Checking
- Application server **SHALL** check transited field. KDC may set `TRANSITED-POLICY-CHECKED`. Clients **MAY** request `DISABLE-TRANSITED-CHECK`. Application servers **MUST** either do checks or reject cross‑realm tickets without flag.

### 2.8 OK as Delegate
- `OK-AS-DELEGATE` flag indicates server is suitable recipient of delegation.

### 2.9 Other KDC Options
- `RENEWABLE-OK`: client accepts renewable if requested life cannot be provided.
- `ENC-TKT-IN-SKEY`: for user‑to‑user authentication; ticket encrypted in second TGT’s session key.
- `OPT-HARDWARE-AUTH`: request hardware authentication.

## 3. Message Exchanges

### 3.1 Authentication Service Exchange (AS)
- **KRB_AS_REQ** (client → KDC): contains client identity, server identity, nonce, etype list.
- **KRB_AS_REP** (KDC → client): contains ticket + session key encrypted in client’s key. Nonce **MUST** match.
- **Pre‑authentication**: **SHOULD** be used. If required, KDC returns `KDC_ERR_PREAUTH_REQUIRED` with METHOD‑DATA in e‑data.
- The KDC **MUST** respond with identical KRB_AS_REP to duplicate requests (replay protection).
- Encryption of EncTicketPart uses server’s key (key usage 2). Encryption of enc‑part uses client’s key (key usage 3).

### 3.2 Client/Server Authentication Exchange (CS)
- **KRB_AP_REQ** (client → server): ticket + encrypted authenticator (key usage 11 or 7 for TGS). `MUTUAL-REQUIRED` option requests mutual authentication.
- Server verifies decrypts, checks timestamp, clock skew (<5 min), replay cache (**MUST** use unless application provides own replay protection). Address check if ticket contains addresses.
- On success, server trusts client identity.
- **KRB_AP_REP** (server → client, if mutual): encrypted with session key (key usage 12). Client verifies timestamp matches authenticator.

### 3.3 Ticket‑Granting Service (TGS) Exchange
- **KRB_TGS_REQ** (client → KDC): uses PA‑TGS‑REQ (AP‑REQ) for authentication. Checksum over KDC‑REQ‑BODY **MUST** be collision‑proof.
- KDC decrypts TGT, verifies checksum, checks hot‑list. Returns **KRB_TGS_REP** encrypted with session key or subkey (key usage 8/9).
- Transited field encoding: DOMAIN‑X500‑COMPRESS **MUST** be supported.

### 3.4 KRB_SAFE Exchange
- Message integrity using keyed checksum (key usage 15). Timestamp or sequence number **MUST** be used.
- Checksum computed over KRB‑SAFE encoding (double‑encoding method).

### 3.5 KRB_PRIV Exchange
- Encryption of user data using key chosen by application (key usage 13). Timestamp or sequence number **MUST** be used.

### 3.6 KRB_CRED Exchange
- Sends tickets and session keys using encryption (key usage 14). Optional addresses, nonce.

### 3.7 User‑to‑User Authentication
- Client obtains server’s TGT (external exchange), sends it as additional‑tickets in TGS‑REQ with `ENC-TKT-IN-SKEY` option. Server decrypts ticket using its TGT session key (`USE-SESSION-KEY` flag in AP‑REQ).

## 4. Encryption and Checksum Specifications
- Encryption and checksum mechanisms defined in [RFC3961] and [RFC3962].
- **String‑to‑key default salt**: realm + name components concatenated.
- **Key usage numbers** (unsigned 32‑bit, zero not permitted):
  - 1: AS‑REQ PA‑ENC‑TIMESTAMP
  - 2: Ticket encrypted part
  - 3: AS‑REP encrypted part
  - 4: TGS‑REQ AuthorizationData (TGS session key)
  - 5: TGS‑REQ AuthorizationData (subkey)
  - 6: TGS‑REQ PA‑TGS‑REQ Authenticator checksum
  - 7: TGS‑REQ PA‑TGS‑REQ Authenticator
  - 8: TGS‑REP encrypted part (TGS session key)
  - 9: TGS‑REP encrypted part (subkey)
  - 10: AP‑REQ Authenticator checksum
  - 11: AP‑REQ Authenticator
  - 12: AP‑REP encrypted part
  - 13: KRB‑PRIV encrypted part
  - 14: KRB‑CRED encrypted part
  - 15: KRB‑SAFE checksum
  - 19: AD‑KDC‑ISSUED checksum
  - 1024–2047: reserved for application use.

## 5. Message Specifications (ASN.1)
*(Full ASN.1 module in Appendix A)*
- **Encoding**: **MUST** use DER; implementations **MAY** accept BER for backward compatibility.
- **KerberosString**: GeneralString constrained to IA5String (US‑ASCII). Control characters **SHOULD NOT** be used.
- **KerberosTime**: GeneralizedTime without fractional seconds, UTC, no separators.
- **Int32** - signed 32‑bit, **UInt32** unsigned.
- **HostAddress**: addr‑type + address. Address types: IPv4 (2), IPv6 (24), DECnet (12), Netbios (20), Directional (3).
- **AuthorizationData**: sequence of ad‑type/ad‑data. AD‑IF‑RELEVANT (1), AD‑KDC‑ISSUED (4), AD‑AND‑OR (5), AD‑MANDATORY‑FOR‑KDC (8).
- **PA‑DATA**: used for pre‑authentication and extensions. PA‑TGS‑REQ (1), PA‑ENC‑TIMESTAMP (2), PA‑PW‑SALT (3), PA‑ETYPE‑INFO (11), PA‑ETYPE‑INFO2 (19).
- **Tickets**: Ticket (app 1), EncTicketPart (app 3), flags: forwardable, forwarded, proxiable, proxy, may‑postdate, postdated, invalid, renewable, initial, pre‑authent, hw‑authent, transited‑policy‑checked, ok‑as‑delegate.
- **AS/TGS messages**: KDC‑REQ (app 10/12), KDC‑REP (app 11/13). Options: forwardable, forwarded, proxiable, proxy, allow‑postdate, postdated, renewable, disable‑transited‑check, renewable‑ok, enc‑tkt‑in‑skey, renew, validate.
- **CS messages**: AP‑REQ (app 14), AP‑REP (app 15).
- **Safe/Priv/Cred**: KRB‑SAFE (20), KRB‑PRIV (21), KRB‑CRED (22).
- **Error**: KRB‑ERROR (30). e‑data for `KDC_ERR_PREAUTH_REQUIRED` contains METHOD‑DATA (SEQUENCE OF PA‑DATA).

## 6. Naming Constraints

### 6.1 Realm Names
- **Case sensitive**. Three styles: domain (e.g., ATHENA.MIT.EDU), X.500 (e.g., C=US/O=OSF), other (prefix: rest). Domain names **MUST** look like domain names. New domain‑based realms **MUST** use recognized Internet domain or authority from parent realm.

### 6.2 Principal Names
- **Name‑type** is a hint; two names with same components but different type are equivalent. Name types: NT‑UNKNOWN (0), NT‑PRINCIPAL (1), NT‑SRV‑INST (2), NT‑SRV‑HST (3), NT‑SRV‑XHST (4), NT‑UID (5), NT‑X500‑PRINCIPAL (6), NT‑SMTP‑NAME (7), NT‑ENTERPRISE (10).
- **Reserved**: initial component `krbtgt` for TGS.

### 6.2.1 Name of Server Principals
- **Two‑component NT‑SRV‑HST** for Internet domain hosts. **Host name MUST be lowercase** if case‑insensitive.

## 7. Constants and Other Defined Values

### 7.1 Host Address Types
- IPv4: 2, IPv6: 24 (must not include Unspecified, Loopback, Link‑Local), DECnet: 12, Netbios: 20, Directional: 3.

### 7.2 KDC Messaging: IP Transports
- **UDP/IP**: port 88. Response **MUST** use UDP. If response too large, KDC returns `KRB_ERR_RESPONSE_TOO_BIG`.
- **TCP/IP**: port 88. Length‑prefixed (4 bytes, network order). High bit reserved (zero). KDC **MUST** return `KRB_ERR_FIELD_TOOLONG` if high bit set.
- **KDC Discovery**: **MUST** provide means to locate KDCs. DNS SRV records `_kerberos._udp.<realm>` and `_kerberos._tcp.<realm>`. Realm **MUST** be domain‑style and case sensitive.

### 7.3 Name of the TGS
- Principal: `krbtgt/<realm>` in realm of issuing KDC.

### 7.4 OID Arc for KerberosV5
- `id-krb5`: 1.3.6.1.5.2

### 7.5 Protocol Constants and Associated Values
- **Key Usage Numbers**: (see Section 4)
- **PreAuthentication Data Types**: PA‑TGS‑REQ (1), PA‑ENC‑TIMESTAMP (2), PA‑PW‑SALT (3), PA‑ETYPE‑INFO (11), PA‑ETYPE‑INFO2 (19), etc.
- **Address Types**: IPv4 (2), Directional (3), IPv6 (24), etc.
- **Authorization Data Types**: AD‑IF‑RELEVANT (1), AD‑KDC‑ISSUED (4), AD‑AND‑OR (5), AD‑MANDATORY‑FOR‑KDC (8).
- **Transited Encoding Types**: DOMAIN‑X500‑COMPRESS (1).
- **Protocol Version**: 5.
- **Kerberos Message Types**: AS‑REQ (10), AS‑REP (11), TGS‑REQ (12), TGS‑REP (13), AP‑REQ (14), AP‑REP (15), KRB‑SAFE (20), KRB‑PRIV (21), KRB‑CRED (22), KRB‑ERROR (30).
- **Error Codes**: (selected) KDC_ERR_NONE (0), KDC_ERR_C_PRINCIPAL_UNKNOWN (6), KDC_ERR_PREAUTH_FAILED (24), KDC_ERR_PREAUTH_REQUIRED (25), KRB_AP_ERR_BAD_INTEGRITY (31), KRB_AP_ERR_TKT_EXPIRED (32), KRB_AP_ERR_REPEAT (34), KRB_AP_ERR_SKEW (37), KRB_ERR_RESPONSE_TOO_BIG (52), KRB_ERR_FIELD_TOOLONG (61).

## 8. Interoperability Requirements

### 8.1 Specification 2 (5.2)
- **Transport**: TCP/IP and UDP/IP **MUST** be supported.
- **Encryption**: AES256‑CTS‑HMAC‑SHA1‑96 **MUST** be supported. AES128‑CTS‑HMAC‑SHA1‑96, DES‑CBC‑MD5, DES3‑CBC‑SHA1‑KD **SHOULD** be supported.
- **Checksums**: HMAC‑SHA1‑96‑AES256 **MUST** be supported. HMAC‑SHA1‑DES3‑KD, DES‑MD5, HMAC‑SHA1‑96‑AES128 **SHOULD** be supported.
- **Realm Names**: Understand hierarchical domain and X.500 styles.
- **Transited Field Encoding**: DOMAIN‑X500‑COMPRESS **MUST** be supported.
- **Pre‑authentication**: TGS‑REQ **MUST** be supported. PA‑ENC‑TIMESTAMP **MUST** be supported by clients. ETYPE‑INFO2 **MUST** be supported; ETYPE‑INFO **SHOULD** be supported.
- **Mutual Authentication**: **MUST** be supported.
- **Ticket Addresses and Flags**: KDCs **MUST** pass through addressless tickets. Implementations **SHOULD** default to addressless tickets. Directional address type **SHOULD** be accepted. Proxies and forwarded tickets **MUST** be supported. Renewable and postdated tickets **MUST** be recognized.
- **User‑to‑User Authentication**: **MUST** be provided (may be disabled by policy).
- **Authorization Data**: **MUST** pass all subfields from TGT to derivative tickets. **MUST** make contents available to server.

### 8.2 Recommended KDC Values
- Minimum lifetime: 5 minutes
- Maximum renewable lifetime: 1 week
- Maximum ticket lifetime: 1 day
- Acceptable clock skew: 5 minutes
- Empty addresses: allowed
- Proxiable, etc.: allowed

## 9. IANA Considerations
- Realm name types, host address types, key usage numbers, preauthentication data types, authorization data types, transited encoding types, message types, name types, error codes are assigned via IETF Kerberos working group review or expert review. Private use allowed for negative values in host addresses, realm types beginning with “X‑”.

## 10. Security Considerations
- **Authorization**: Applications **MUST NOT** accept ticket issuance as authorization.
- **Denial of Service**: Not solved; KDC must protect against overload.
- **Principal key secrecy**: Mandatory.
- **Password guessing**: Mitigating by requiring strong passwords and pre‑authentication.
- **Pre‑authentication**: Strongly encouraged to prevent known‑plaintext attacks.
- **Server key strength**: Keys **SHOULD** be randomly generated for server principals.
- **DES‑CBC‑MD5**: Weak; stronger algorithms **MUST** be used when possible.
- **Clock synchronization**: **MUST** be secure; skew limits per‑server.
- **Principal identifier reuse**: Must not be short‑term.
- **Replay cache**: **MUST** be used if application does not provide its own replay protection.
- **DNS security**: **MUST NOT** use insecure DNS for canonicalization or realm discovery.
- **Perfect forward secrecy**: Not provided; application may use other key exchange.
- **Authorization data**: Many RFC 1510 implementations ignore unknown authorization data.
