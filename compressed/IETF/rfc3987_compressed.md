# RFC 3987: Internationalized Resource Identifiers (IRIs)
**Source**: IETF | **Version**: Standards Track | **Date**: January 2005 | **Type**: Normative
**Original**: https://datatracker.ietf.org/doc/rfc3987/

## Scope (Summary)
Defines the Internationalized Resource Identifier (IRI) as a complement to URIs, using characters from the Universal Character Set (Unicode/ISO 10646). Provides a deterministic mapping from IRIs to URIs to enable use in place of URIs where appropriate, without modifying URI definitions.

## Definitions and Abbreviations
- **character**: A member of a set of elements used for the organization, control, or representation of data.
- **octet**: An ordered sequence of eight bits considered as a unit.
- **character repertoire**: A set of characters (in the mathematical sense).
- **sequence of characters**: A sequence of characters (one after another).
- **sequence of octets**: A sequence of octets (one after another).
- **character encoding**: A method of representing a sequence of characters as a sequence of octets (maybe with variants). Also, a method of (unambiguously) converting a sequence of octets into a sequence of characters.
- **charset**: The name of a parameter or attribute used to identify a character encoding.
- **UCS**: Universal Character Set. The coded character set defined by ISO/IEC 10646 and the Unicode Standard.
- **IRI reference**: Denotes the common usage of an Internationalized Resource Identifier. An IRI reference may be absolute or relative. However, the "IRI" that results from such a reference only includes absolute IRIs; any relative IRI references are resolved to their absolute form.
- **running text**: Human text (paragraphs, sentences, phrases) with syntax according to orthographic conventions of a natural language, as opposed to syntax defined for ease of processing by machines.
- **protocol element**: Any portion of a message that affects processing of that message by the protocol in question.
- **presentation element**: A presentation form corresponding to a protocol element; for example, using a wider range of characters.
- **create (a URI or IRI)**: With respect to URIs and IRIs, the term is used for the initial creation.
- **generate (a URI or IRI)**: With respect to URIs and IRIs, the term is used when the IRI is generated by derivation from other information.

## 1. Introduction
### 1.1 Overview and Motivation
IRIs extend URI syntax to a wider repertoire of characters to allow use of natural language scripts beyond US-ASCII.

### 1.2 Applicability
IRIs are designed to be compatible with new URI schemes [RFC2718]. Conditions for use:
a. A protocol or format element MUST be explicitly designated to carry IRIs.
b. The protocol or format MUST have a mechanism to represent the wide range of characters used in IRIs.
c. The corresponding URI MUST encode original characters into octets using UTF-8.

### 1.3 Definitions
(As defined in Definitions and Abbreviations section above.)

### 1.4 Notation
Uses U+, XML Notation (&#x;), Bidi Notation, and octet notation (<xx>). Key words are interpreted as described in [RFC2119].

## 2. IRI Syntax
### 2.1 Summary of IRI Syntax
IRIs use the same structure as URIs, but the class of unreserved characters is extended to include UCS characters beyond U+007F, subject to limitations. Characters outside US-ASCII MUST NOT be used for syntactical purposes (e.g., as delimiters).

### 2.2 ABNF for IRI References and IRIs
Normative ABNF rules are provided. Key differences from [RFC3986]: expanded `iunreserved` includes `ucschar`; `iprivate` allowed only in query. The grammar uses "first-match-wins" algorithm.

#### IRI Grammar (partial)
```
IRI            = scheme ":" ihier-part [ "?" iquery ] [ "#" ifragment ]
IRI-reference  = IRI / irelative-ref
absolute-IRI   = scheme ":" ihier-part [ "?" iquery ]
iunreserved    = ALPHA / DIGIT / "-" / "." / "_" / "~" / ucschar
ucschar        = %xA0-D7FF / ... (full ranges defined)
iprivate       = %xE000-F8FF / %xF0000-FFFFD / %x100000-10FFFD
```
All other rules (scheme, port, IPv6address, etc.) are identical to [RFC3986].

## 3. Relationship between IRIs and URIs
### 3.1 Mapping of IRIs to URIs
Applications MUST map IRIs to URIs using two steps:
**Step 1**: Generate UCS character sequence from original IRI format:
- a. If IRI is from non-digital form, normalize to NFC.
- b. If from non-Unicode digital encoding, convert to UCS and normalize to NFC.
- c. If from Unicode encoding, do not normalize.

**Step 2**: For each character in `ucschar` or `iprivate`:
- 2.1 Convert to UTF-8 octets.
- 2.2 Percent-encode each octet as %HH (SHOULD use uppercase hex).
- 2.3 Replace original character with percent-encoded sequence.

Systems MAY convert ireg-name using ToASCII (IDNA) for schemes known to use domain names, if percent-encoding is not allowed.

The mapping is identity for URIs and idempotent. Every URI is an IRI.

### 3.2 Converting URIs to IRIs
Procedure (or equivalent):
1. Represent URI as sequence of US-ASCII octets.
2. Convert all percent-encodings to corresponding octets, except those for "%", reserved characters, and US-ASCII chars not allowed in URIs.
3. Re-percent-encode any octet not part of a strictly legal UTF-8 sequence.
4. Re-percent-encode octets that in UTF-8 represent characters not allowed per sections 2.2, 4.1, and 6.1.
5. Interpret resulting octet sequence as UTF-8.

MUST NOT use any character encoding other than UTF-8 in steps 3 and 4.

## 4. Bidirectional IRIs for Right-to-Left Languages
### 4.1 Logical Storage and Visual Presentation
- Bidirectional IRIs MUST be stored in full logical order and conform to IRI syntax.
- MUST be rendered using the Unicode Bidirectional Algorithm as if in left-to-right embedding (U+202A ... U+202C).
- Bidirectional formatting characters (LRM, RLM, LRE, RLE, LRO, RLO, PDF) MUST NOT be part of the IRI.

### 4.2 Bidi IRI Structure
For each component (iuserinfo, ireg-name, isegment, iquery, ifragment):
1. A component SHOULD NOT use both right-to-left and left-to-right characters.
2. A component using right-to-left characters SHOULD start and end with right-to-left characters.

If restrictions cannot be satisfied, the component can be mapped to URI notation.

### 4.3 Input of Bidi IRIs
Bidi input methods MUST generate IRIs in logical order while rendering according to section 4.1. Rendering SHOULD be updated after every new character.

## 5. Normalization and Comparison
### 5.1 Equivalence
Equivalence is based on string comparison, possibly augmented by scheme-specific rules. Applications using IRIs as identity tokens with no protocol relationship MUST use Simple String Comparison (5.3.1). All other applications MUST select one of the comparison practices from the Comparison Ladder (5.3) or after IRI-to-URI conversion, from [RFC3986] section 6.2.

### 5.2 Preparation for Comparison
IRI comparison REQUIRES that all escapings or encodings in the protocol/format are resolved.

### 5.3 Comparison Ladder
#### 5.3.1 Simple String Comparison
Character-by-character comparison after conversion to common encoding form. MUST NOT map IRIs to URIs. IRI SHOULD NOT be modified during transport if used as identifier.

#### 5.3.2 Syntax-Based Normalization
Techniques include case normalization, character normalization, percent-encoding normalization, and dot-segment removal.
- **Case Normalization**: Percent-encoding hex digits SHOULD be normalized to uppercase. Scheme and US-ASCII host SHOULD be normalized to lowercase.
- **Character Normalization**: Equivalence MUST rely on IRIs being appropriately pre-normalized. Conversion from non-digital or non-UCS forms MUST use NFC. IRIs SHOULD be created using NFC.
- **Percent-Encoding Normalization**: Decode any percent-encoding that corresponds to unreserved characters.
- **Path Segment Normalization**: Remove "." and ".." segments.

#### 5.3.3 Scheme-Based Normalization
Scheme-specific rules (e.g., default port, empty path => "/"). For Internationalized Domain Names (IDN) in IRIs, SHOULD validate using ToASCII with flags UseSTD3ASCIIRules and AllowUnassigned. Validated IDN components SHOULD be character normalized using Nameprep but SHOULD NOT be converted to ACE.

#### 5.3.4 Protocol-Based Normalization
Aggressive techniques like observing redirects (informative).

## 6. Use of IRIs
### 6.1 Limitations on UCS Characters Allowed in IRIs
- Repertoire limited by component definitions.
- Avoid strong visual look-alikes; use NFKC where possible.

### 6.2 Software Interfaces and Protocols
- Intermediate software MUST map IRIs to URIs per section 3.1 when transferring to URI-only components. This mapping SHOULD be applied as late as possible. MUST NOT be applied between components known to handle IRIs.

### 6.3 Format of URIs and IRIs in Documents and Protocols
- Where document has native character encoding, IRIs MUST be encoded in that encoding and converted accordingly. IRI characters not expressible SHOULD be escaped using document escaping conventions (e.g., numeric character references). They MAY be percent-encoded per section 3.1.
- If document encoding is not UTF-8, IRIs MUST NOT be placed in the document in UTF-8.

### 6.4 Use of UTF-8 for Encoding Original Characters
The corresponding URI MUST encode original characters into octets using UTF-8. This applies to all parts of a URI (except possibly ireg-name via IDNA). The requirement for UTF-8 applies to the whole URI or only to some parts, as per scheme or component definition.

### 6.5 Relative IRI References
Processing follows algorithms of [RFC3986], treating IRI characters as unreserved.

## 7. URI/IRI Processing Guidelines (Informative)
- **7.1** APIs/protocols need interfaces for IRIs; UTF-8 recommended.
- **7.2** Entry software must update to allow non-ASCII input.
- **7.3** Transfer between applications (e.g., clipboard, URI detection) should use IRI syntax.
- **7.4** Generation should expose UTF-8 based URIs.
- **7.5** Selection should avoid confusable characters; prefer NFKC.
- **7.6** Display should percent-encode if rendering not possible.
- **7.7** Interpretation should accept multiple forms (UTF-8, legacy encodings) but be cautious of collisions.
- **7.8** Upgrading strategy: prioritize interpreting software, then generation, then display.

## 8. Security Considerations
- Incorrect encoding/decoding (e.g., overlong UTF-8 sequences) can lead to security problems.
- Spoofing may occur due to normalization mismatches, similar characters (homoglyphs), bidirectional ambiguity, and legacy encodings. Proper Unicode bidi implementation is critical.
